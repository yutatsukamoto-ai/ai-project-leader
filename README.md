# AIプロジェクトリーダー

曖昧な期待しか持たないクライアントの生成AI/DXプロジェクトを、PMBOK第8版の王道に準拠して完走させるためのSkill群とナレッジ基盤。
「1成果物＝1子Skill＋薄い親オーケストレーター」「おまかせ不採用（AIが提案・理由を見せ、本人が判断）」が設計の芯。最終ゴールは配布パッケージ化。

このREADMEは**入口の地図**。詳細の正典は各ファイルにある（下記）。

---

## 5分クイックスタート

新しい案件を試すときは、まず次のどれかをAIに渡す。

- 初回商談メモ
- 議事録
- 提案前の相談内容
- スライド/PDFの要約

最初の依頼文:

```text
前段チェーンを回して。以下の案件説明から、案件理解サマリー、課題仮説、必要ならリサーチ、計画提案書まで順に進めたいです。
```

進み方:

1. `zendan-chain` が `00_前段/01_案件理解サマリー` を作る。
2. 「この理解で合っていますか？」で必ず止まる。
3. 確認後、`02_課題仮説シート` → `03_リサーチメモ` → `04_計画提案書` へ進む。
4. `04_計画提案書` の承認後、`00_前段/引き継ぎ書_前段→立ち上げ.md` を作る。
5. 次は `tachiage-chain`。入力は引き継ぎ書と `project-context.md`。

## フォルダ構造

AIPMのFlow/Stock構造を踏襲。番号は依存・参照の順。

```
AIプロジェクトリーダー/
├── README.md              ← 入口の地図（このファイル）
├── プロジェクト骨子.md     ← 憲法。最初に読む
├── 00_プロジェクト管理/    ← この案件の建設ログ（決定・懸念・引き継ぎ）
├── 10_参考資料/           ← 書籍・お手本（★著作物。配布厳禁）
├── 20_Skills/             ← 作成したSkill（配布物の本体）
├── 30_Flow/               ← 作業中ドラフト・試運転案件（YYYY-MM-DD/案件名。詳細は30_Flow/README.md）
└── 40_Stock/              ← 完成した知識・横断ガイドライン（再利用の基盤）
```

Flow＝作業中、Stock＝完成品。この分離が「何が終わって何が途中か」の検知装置になる。

## 知っておくと迷わない場所

| 用途 | 場所 |
|---|---|
| 文章の書き方（全成果物共通の正典） | `40_Stock/横断ガイドライン/`（README＋`文章作法.md`） |
| PMBOK基準資料の正典 | `10_参考資料/PMBOK第8版_40プロセス一覧.md`・`6原則一覧.md` |
| 共通項の同期・ズレ検査・パッケージ化・基盤点検 | `_tools/build.sh`（`--sync`／`--check`／`<skill>`／`--all`／`--verify`）。手作業zip禁止。`sync-manifest.tsv`がマニフェスト |
| 配布・雛形化の層と除外リスト | `00_プロジェクト管理/配布と雛形化_層の定義と棚卸し.md` |
| Skillの設計原則・チェーン構成 | `プロジェクト骨子.md` §4／`20_Skills/`（フェーズ別） |
| 案件の共有コンテキスト（SSOT） | `{案件ルート}/project-context.md`。新規チェーン開始時に作り、確定事実・決定・未確認をここへ書き戻す |

## 触るときの約束（要点）

- 多段階タスクは各ステップで確認。勝手に進めない。提案には理由を添える。
- 作業ドラフトは `30_Flow`、完成品の整理は `40_Stock`。
- 成果物は文章作法（`40_Stock/横断ガイドライン/文章作法.md`）に従う。
- 共通項（文章作法・PMBOK基準資料）は**正典を1か所で編集し`bash _tools/build.sh --sync`で配る**。コピーを直接編集しない。ズレ確認は`bash _tools/build.sh --check`。
- **Skillのパッケージ化は必ず `bash _tools/build.sh <skillディレクトリ>` 経由**（手作業でzipしない）。0バイト・構造不正・ゴミ残りを構造的に防ぐ。
- 基盤層（裏側）の健全性は `bash _tools/build.sh --verify` で点検。毎朝の定期タスクでも自動チェックが回る。
- 配布・雛形化の前に「配布と雛形化_層の定義と棚卸し」を必ず読む（著作物・実データの除外）。

## やらないこと

- 承認なしに次フェーズへ進まない。
- `outputs/` を正典成果物の置き場にしない。
- `.skill` を手作業でzipしない。
- 共通referencesのコピーを直接編集しない。
- 実案件データを配布物や `40_Stock/` にそのまま入れない。
- 客先提出版に内部用の検査レシートや管理メタをそのまま混ぜない。
