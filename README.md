「1成果物＝1子Skill＋薄い親オーケストレーター」「おまかせ不採用（AIが提案・理由を見せ、本人が判断）」が設計の芯。最終ゴールは配布パッケージ化。

このREADMEは**入口の地図**。詳細の正典は各ファイルにある（下記）。

## フォルダ構造

AIPMのFlow/Stock構造を踏襲。番号は依存・参照の順。

```
AIプロジェクトリーダー/
├── README.md              ← 入口の地図（このファイル）
├── プロジェクト骨子.md     ← 憲法。最初に読む
├── 00_プロジェクト管理/    ← この案件の建設ログ（決定・懸念・引き継ぎ）
├── 10_参考資料/           ← 書籍・お手本（★著作物。配布厳禁）
├── 20_Skills/             ← 作成したSkill（配布物の本体）
├── 30_Flow/               ← 作業中ドラフト・試運転案件（日付で整理）
└── 40_Stock/              ← 完成した知識・横断ガイドライン（再利用の基盤）
```

Flow＝作業中、Stock＝完成品。この分離が「何が終わって何が途中か」の検知装置になる。

## 触るときの約束（要点）

- 多段階タスクは各ステップで確認。勝手に進めない。提案には理由を添える。
- 作業ドラフトは `30_Flow`、完成品の整理は `40_Stock`。
- 成果物は文章作法（`40_Stock/横断ガイドライン/文章作法.md`）に従う。
- 共通項（文章作法・PMBOK基準資料）は**正典を1か所で編集し`bash _tools/sync.sh sync`で配る**。コピーを直接編集しない。ズレ確認は`bash _tools/sync.sh check`。
- **Skillのパッケージ化は必ず `bash _tools/build.sh <skillディレクトリ>` 経由**（手作業でzipしない）。0バイト・構造不正・ゴミ残りを構造的に防ぐ。
- 基盤層（裏側）の健全性は `bash _tools/build.sh --verify` で点検。毎朝の定期タスクでも自動チェックが回る。
- 配布・雛形化の前に「配布と雛形化_層の定義と棚卸し」を必ず読む（著作物・実データの除外）。
