# AIプロジェクトリーダー プロジェクト骨子

最終更新: 2026-06-16（§6ロードマップ・§7未決事項を現状に同期）
このファイルは本プロジェクトの「憲法」。セッション開始時にClaudeが必ず読み込み、すべての判断・提案はここに照らして行う。

---

## 1. Why（固定。逸れる提案はしない）

会社員コンサルタントとして、社外クライアントの生成AI/DX支援を行う。クライアントは「AIやITに期待はあるが、何をどうすればいいか分からない」状態。その曖昧な期待を要件として定義するところから始め、**PMBOK第8版の王道に準拠して、生成AIと協働しながらプロジェクトを終結まで完走させる**。そのためのSkill・ナレッジ基盤を構築するのが本プロジェクト。

### Whyの2本柱

1. **漏れなく完了させる** — フェーズ×ドメインの構造自体がチェックリストとして機能し、「絶対押さえんとあかんこと」の抜け漏れを防ぐガードレールにする
2. **自分自身の成長** — 使うたびに「今PMBOK第8版のこれを実践している」と実体験できる学習装置を兼ねる。最終的にはAIがなくてもプロジェクトを推進できる自分になる

## 2. 準拠フレーム: PMBOK第8版（2025年11月リリース）

- **6つの原理原則**: 包括的視点 / 価値への集中 / 品質の組み込み / 説明責任あるリーダーシップ / サステナビリティ統合 / 文化のエンパワーメント
- **7つのパフォーマンス・ドメイン**: ガバナンス / スコープ / スケジュール / ファイナンス / ステークホルダー / リソース / リスク
- **5つのフォーカスエリア**: 立ち上げ / 計画 / 実行 / 監視・コントロール / 終結
- **40の非規範的プロセス**: 「押さえるべきこと」チェックリストの原型。ただし非規範的なので、案件規模に応じたテーラリング判断基準もSkillに持たせる

**基準資料（2026-06-12作成。Skill設計・PM成果物作成時は必ず参照する）**:

- `10_参考資料/PMBOK第8版_40プロセス一覧.md` — 7ドメイン×5フォーカスエリアの40プロセス全表。プロセス番号#1〜40でSkillと紐づける。「何をやるか」の軸
- `10_参考資料/PMBOK第8版_6原則一覧.md` — 6原則＋マインドセット3志向＋第7版12原則からの統合マッピング。「どう判断するか」の軸

参考資料の『外資系コンサルが教えるプロジェクトリーダーの仕事』は**PMBOK第8版を正面から扱う書籍**であり、成果物マップ（フェーズ×ドメイン）＝チェーン設計の設計図、各章まとめ＝各フェーズの本質と品質基準の源泉として使う（再構成版: 40_Stock/ナレッジ/プロジェクトリーダーの仕事_章まとめ要点.md）。第8版の6原則・価値起点の思想は本書経由で根拠を持つ（原典入手までの橋渡し）。補助線として『外資系コンサルが教える難題を解決する12ステップ プロジェクトリーダーの教科書』の12ステップ（再構成版: 40_Stock/ナレッジ/12ステップ要点.md）を使う。使う場面は3つ: ①難航時に引く ②要チェック時に照合 ③**各フェーズの停止ポイント・最終チェックで該当フェーズの4項目をクリアしているか照合**（立ち上げ→定義4項目、計画→デザイン4項目、実行・監視→推進4項目）。主従は PMBOK第8版＝網羅性の背骨、12ステップ＝質のチェック。

## 3. 協働の思想（最重要の設計判断）

**おまかせ思想は採用しない。「AIが提案し、理由を見せ、人間が判断して進む」設計に倒す。**

- AIPMシステムから踏襲するのは「構造」: フェーズ別コマンド、成果物の段階的蓄積、次の一手の提示
- 採用しないのは「おまかせ進行」: 裏で勝手に処理せず、今どのフェーズ・どのドメインで・なぜこの成果物を作るのかを毎回見せながら進む
- 壁打ちを重視する。AIからの能動的な提案・アドバイスは歓迎。ただし判断は必ず本人が行う
- 依頼・指示を受けたら、自分の言葉で言語化し直して返し、認識確認（合っていますか？）を取ってから動く。明確に見える依頼でも省かない——明確に見えるときほど暗黙の前提のずれが潜む
- **ユーザーの指示・判断に抜け漏れや骨子との矛盾を見つけたら、実行前に必ず指摘する。従うのはその後**（2026-06-11追記）。指摘は理由付きで行い、指摘後の判断は本人に委ねる

## 4. 設計原則

- **アーキテクチャ**: 1成果物=1子Skill + 薄い親オーケストレーター（順番・受け渡し・停止ポイントのみ、10〜20行）。知識は子Skillに持たせ、親には書かない
- **フェーズごとに親Skillを必ず置く**（2026-06-12採用）: 成果物が少ないフェーズ（立ち上げ2つ・終結1つ）でも例外を作らず一律に置く。理由: ①入口の一貫性（どのフェーズも親を呼べば順番・停止ポイントを管理してくれる）②停止ポイント管理・12ステップ照合・テーラリング判断はどの子Skillにも属さないフェーズレベルの知識であり、親が置き場所になる ③zendan-chainの型があるため横展開コストが低い
- **派生資料は独立Skillにしない**（2026-06-12採用）: キックオフ資料のように、PM成果物（チャーター等）の内容を別形式に変換しただけの派生物は「1成果物=1Skill」に数えない。親Skillが承認後に「作りますか？」と聞く任意ステップとし、生成は既存の汎用Skill（tsutawaru-slide等）に渡す
- **現在地の可視化**: フェーズ×ドメインで「今どこにいるか」を常に明示
- **上流前段**: 成果物マップ（チャーター起点）の手前に「ヒアリング・議事録 → 課題と期待の構造化 → チャーター素材」の前段チェーンを置く。曖昧な期待の要件化が勝負どころ
- **一括/個別生成の判断**: 前工程の誤りを後で直すコストが高い成果物の手前には必ず人間確認ポイント（停止）を置く
- **品質基準**: 自分が提示する参考例（『プロジェクトリーダーの仕事』の成果物マップ、業務設計の教科書、マジカ等）をお手本にSkillを作り込む。検証は想定モデル案件で行う
- **子Skillの作り方は「伴走→写し取り」方式**: 机上でSKILL.mdを設計しない。まず一緒に成果物を1回実際に作ってみて、会話で言語化された手順・判断基準・人が確認すべき分かれ目をSkillに写し取る（出典: Coworkオンボーディング資料）。一緒にやる過程自体がPMBOK実践の体験＝成長のWhyにも直結する
- **ナレッジは再構成方式**: PMBOK・書籍の原文転載はせず、エッセンスを業務向けに自分の言葉で再構成して焼き込む
- **ポータビリティ**: Cowork / Claude Code主体。Copilot転用を見据え、知識はプレーンMarkdownに寄せ、Claude固有機構は薄い層に隔離
- **子Skillの標準形はreferences分離**（2026-06-11採用）: SKILL.mdは薄い本体（起動条件・手順・停止ポイント）のみ。品質基準・PMBOK照合チェック等の知識はreferences/配下のMarkdownに分離し、該当ステップでのみ読み込む（Progressive Disclosure。出典: 『完全解説！Claude Codeテクニック集』第3章）
- **二層構え**（2026-06-11採用）: Cowork単独で完結する設計を基本層とし、Claude Codeで動かす場合はHooks（停止ポイントの機械的強制）・サブエージェント（レビューの独立コンテキスト化）を上乗せ層として追加する。レビューはCowork版では「レビュー専用子Skill」として実装し、将来そのままサブエージェント化できる形にする
- **進行ルール**: 多段階タスクは勝手に進めない。各ステップで確認を取る。Skill完成時は.skillファイルでパッケージして渡す

## 5. フォルダ構成（AIPMのFlow/Stock構造を踏襲）

```
AIプロジェクトリーダー/
├── プロジェクト骨子.md   ← 憲法。セッション開始時に必ず読む
├── 00_プロジェクト管理/   ← 議論ログ・決定事項・引き継ぎ文書
├── 10_参考資料/           ← 書籍・AIPMシステム・マジカ等のお手本
├── 20_Skills/             ← 作成した子Skill・親Skill
├── 30_Flow/               ← 作業中ドラフト（日付フォルダで整理）
└── 40_Stock/              ← 完成した成果物（フェーズ×ドメインで整理）
```

Flow＝作業中、Stock＝完成品。この分離が「何が終わって何が途中か」の検知装置になる。

### MVP定義: 「縦に薄く一周」（2026-06-13決定 / 懸念M-03対応）

完走（Why①）と成長（Why②）には、幅より先に「全フェーズを薄くても一度通り抜ける」体験が効く。最初の到達目標＝**縦に薄い一周**。物量が完走を殺すのを防ぐガードレール。

- **定義**: 各フェーズ1成果物の最小骨で、立ち上げ→計画→実行→監視→終結を端から端まで通す。
- **MVPの一周（各フェーズ1成果物）**:
  - 立ち上げ: プロジェクトチャーター1枚（#1）
  - 計画: 統合計画書1枚（簡易・要求/スコープ/リスクを束ねる）
  - 実行: デリバラブル1つ ＋ 課題管理表
  - 監視: 差異レビュー1枚
  - 終結: 完了報告1枚
- **作る順番（依存順）**: 前段（済）→ 立ち上げ親 tachiage-chain（型済）→ **立ち上げのチャーター子Skill（次の一手）** → 以降フェーズ順
- **横の充実**（各フェーズの成果物を◎以外に増やす）はMVP通過後。型は流用できる（[[ai-project-leader-design-principles]]）ので後から足せる。
- テーラリングで薄く保つ。各フェーズの親が「今回はこの1個」を明示する。
