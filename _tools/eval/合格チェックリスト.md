# 回帰eval 目視チェックリスト（機械では見ない品質バー）

位置づけ: M-04 回帰evalの**目視層**。機械層（`_tools/eval.sh`）が「節の核が在るか」を見るのに対し、ここは**読まないと分からない品質**を見る。
使い方: Skillを改訂したとき／モデルを交代したとき／配布前に、対象Skillをゴールデン入力で走らせ直し、その出力を下の問いで点検する（人またはLLMが判定）。✅/❌で記録し、❌は原因をSkillのreferences品質バーに照らす。
出典: 各Skillの `references/型と品質基準.md`・`references/表と数値の見せ方.md` の「やってはいけない」。

---

## 共通チェック（全Skill）

文章・記号・数値の作法は横断ガイドライン由来。全成果物に効く。

- [ ] **状態は言葉**: 状態・達成度・優先度を、確度記号（●◐○）や採否記号（◎○△）で代用していない（言葉で書く）。
- [ ] **数字をぼかさない**: 件数・期日・金額・目標値が具体。概算は「概算」と明記しレンジで。
- [ ] **一文一改行**: 句点ごとに改行されている。だらだら続く長文パラグラフが無い。
- [ ] **ラベル型見出し**: 見出しを流し読みすると目次になる（何の話か一目で分かる）。
- [ ] **やらないこと・停止ポイントが明確**: 範囲外の明言・✋確認ポイントが落ちていない。
- [ ] **入力との整合**: 前工程の成果物・project-contextの確定事実と矛盾していない。

## Skill固有チェック

機械層では拾えない、その成果物の“正体”に関わる問い。各1〜3点に絞る。

**anken-rikai-summary（案件理解サマリー）**
- [ ] 期待が複数混在のまま放置されず、分解されている（手段と目的を区別）。
- [ ] 事実・推測・不明点が区別され、不明点が優先度順の質問になっている。
- [ ] クライアントの最重要発言が特定されている。

**kadai-kasetsu-sheet（課題仮説シート）**
- [ ] 仮説と断定が区別されている（確度付き）。
- [ ] 優先順位が決め打ちでなく「何が分かれば変わるか」（逆転条件）が書かれている。

**research-memo（リサーチメモ）**
- [ ] 観点が「業種×課題仮説」で組まれている（汎用リサーチになっていない）。
- [ ] 価値が確度昇格＋説得材料（同業の数字）の獲得になっている。出典が付いている。

**keikaku-teiansho（計画提案書）**
- [ ] 読み手＝意思決定者ひとりに絞られ、開始の意思決定から逆算されている。
- [ ] 「途中でやめる自由」（ゲート）が渡されている。
- [ ] 社内用付録（品質チェック）がクライアント版に混ざっていない。

**project-charter（チャーター）**
- [ ] 「承認書類」でなく「迷ったとき立ち戻る一枚」になっている（固定/学習の仕分けがある）。
- [ ] 成功基準がクライアントの言葉＋測り方で書かれている。

**integrated-plan（統合計画書）**
- [ ] ローリングウェーブ（直近詳細＋以降は条件付き骨格）になっている。
- [ ] 不可逆リスク（情報漏洩等）が点検されている。
- [ ] 計画系プロセスのテーラリング記録（採用/省略＋理由）がある。

**wbs-builder（WBS／wbs.json）**
- [ ] 事実だけ（予定/実績の日付＋qty/hours）を持ち、工数・進捗・イナズマ線を数値で持っていない（自動計算に委ねている）。
- [ ] 最大3階層・ローリングウェーブ（直近段階のみ仮日付＋詳細／先はnull＋骨格）。
- [ ] ★律速（クリティカルパス）と不可逆の関所がnoteに示されている。
- [ ] 統合計画書§2と二重化でなく深掘り・接続になっている。

**issue-log（課題管理表）**
- [ ] 全課題に「次の一手・担当・期限」がある。
- [ ] リスク登録簿と混ざっていない（ID=R-xxのリンクで持つ・二重管理しない）。
- [ ] 区分（作業課題/判断待ち/リスク対応）がどれか1つに付いている。

**status-report（週次状況報告）**
- [ ] 台帳（課題管理表）の全行貼りでなく、「動き」と「要対応」に絞った巻き上げになっている。
- [ ] ダッシュボード1行が3指標（スケジュール状態／高優先度オープン件数／次の意思決定）。
- [ ] §リスクの動きは“動きがある週だけ”載っている。

**project-closeout（終結報告書）**
- [ ] 閉じる4要素（達成確認・引き継ぎ・教訓・終結宣言）が揃っている。
- [ ] 達成度が3値（達成/一部/未達）で正直（○×やぼかしになっていない）。
- [ ] 残課題・不可逆リスクが引き継がれている。教訓が次フェーズで使える具体になっている。

## 判定の残し方

- 走らせた日・モデル・対象Skill・PASS/FAILを1行記録する（軽量。run-log化は運用が回ってから）。
- ❌が出たら、原因が「Skillの劣化」か「ゴールデンの陳腐化（仕様が正しく変わった）」かを切り分ける。後者ならゴールデンとgoldens.tsvを更新する（M-09 学習還流の入口）。
