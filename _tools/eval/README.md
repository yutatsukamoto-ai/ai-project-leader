# 回帰eval（M-04）— 入口の地図

Skillやモデルを変えたとき、「前は通っていたことが黙って壊れていないか」を検知するための軽量eval。
LLM出力は値の厳密一致ができないので、**ゴールデン案件＋型チェック**で回帰を捉える。原型（まず1回動かして固める）であり、作り込みは運用が回ってから。

## 2層構成

| 層 | 何を見る | 実行 | 判定 |
|---|---|---|---|
| 機械層 | 各ゴールデンに「節の核トークン」が在るか・ファイルが在るか | `bash _tools/eval.sh`（全件）／`bash _tools/eval.sh <skill>`（個別） | 自動（0=全PASS） |
| 目視層 | 読まないと分からない品質バー（状態は言葉/数字ぼかさない/下流と矛盾しない 等） | `bash _tools/eval-judge.sh`（プロンプト自動生成）。基準=`eval/合格チェックリスト.md`、プロンプト雛形=`eval/judge-prompt.md`、初回基準値=`eval/baseline_2026-06-14.md` | LLM-judge（自動生成プロンプト→Claude判定） |

- 機械層の入力＝`eval/goldens.tsv`（skill・case・パス・必須トークン）。トークンは見出しの厳密一致でなく”言い回しが変わっても残るべき核”。
- 現在のゴールデン＝山藤12件＋テクノブリッジ10件＋スマートファクトリー2件＋丸山22件＋slide-craft構成案1件＋追加3件の計50件（2026-06-19時点）。カバーSkill: 25種。

## いつ流すか

1. **Skill改訂時**: そのSkillを `bash _tools/eval.sh <skill>` で点検＋目視チェックリストの該当節。
2. **モデル交代時**: 全件 `bash _tools/eval.sh`＋主要Skillの目視。挙動が変わりやすいので特に重要。
3. **配布前**: 全件＋目視一式。配布パッケージ化（最終ゴール）の品質ゲート。

## build.sh --verify との違い（混同しない）

- `build.sh --verify`＝**裏側（L2基盤）の構造健全性**。.skillが壊れていないか・ゴミ・共通項のドリフト。毎朝の `aipm-backend-verify` で自動。
- `eval.sh`＝**表（L1成果物）の型と質**。Skillが作る出力が崩れていないか。改訂・モデル交代・配布の節目で人が流す。
- 役割が違うので別物として持つ。evalは毎朝には載せない（ゴールデン点検は節目で十分・目視層は自動化できない）。

## ゴールデンが動いたら（D-01）

案件が `30_Flow/` から `40_Stock/` へ移動したら、`goldens.tsv` の**パス列だけ**直す（単一の真実）。トークンは原則そのまま。

## 仕様変更とゴールデン更新（M-09 学習還流）

❌が出たら原因を切り分ける——「Skillの劣化（直すべき）」か「仕様が正しく変わった（ゴールデンを更新すべき）」か。
後者なら、新しい正解を反映してゴールデン本体と `goldens.tsv` を更新する。これが案件横断の学習を仕組みに戻す入口（M-09）。
