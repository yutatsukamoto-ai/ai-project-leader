# PoC評価報告書 作成（子Skill）

終結フェーズの基幹成果物（#9 プロジェクトまたはフェーズの終結・PoC特化／#7 パフォーマンスの監視・コントロール）。
PoC提案書で設定した評価軸に実績を対照し、**Go/NoGoの判断材料を意思決定者に返す**。
中身の品質基準・節構成は `references/型と品質基準.md` を見る。

## 位置づけ

PoC提案書（計画フェーズ）が「何を・どの評価軸で・どこまで検証するか」を提案し、本Skillはその結果を評価して「Go/NoGoの判断材料」を返す。
計画提案書 → PoC提案書 → PoC実行 → **PoC評価報告書**（本Skill）の流れ。
PoC評価報告書は生きた台帳でなく、**PoCフェーズを閉じるためのワンショット評価文書**。

## 手順

1. **入力を読む**: `{案件名}/02_計画/NN_PoC提案書`・PoC実行結果（データ・ログ・観察記録）・`project-context.md`・`references/型と品質基準.md`。
   - PoC提案書の✋承認が済か確認する。未なら止める。
   - PoC実行結果が十分に揃っているか確認する。不足があればユーザーに申告を求める。

2. **提案書との対照表を組む**: PoC提案書の評価軸4軸（ビジネス・業務／技術／運用／コスト）ごとに、評価項目・成功基準・実績・判定を並べる。
   - 提案書に無い評価軸を勝手に足さない。提案書で約束した基準で判定する。
   - 【差し込みカード参照枠】ドメインカードがあれば引く（無ければ素通り）。

3. **ノックアウトファクター判定**: 提案書で特定したノックアウトファクターの結果を先に確定する。ここが不合格ならGo/NoGo推奨は自動的にNoGo寄りになる。

4. **推測には根拠を明示**: 定性的評価は観察記録を出典として併記する。未確定は🟡で残して判断材料としての限界を示す。
   - 記号は `references/表と数値の見せ方.md` の正典どおり（確度`●◐○`／採否`◎○△`／状態は言葉）。

5. **アンチパターンチェック**: PoC提案書の `references/PoCアンチパターン.md` 7パターンのうち、評価段階で特に注意すべきものを確認する。
   - AP-04（失敗が許されないPoC）: ネガティブな結果を正直に報告できているか。
   - AP-06（やりっぱなしPoC）: 結果整理→報告→次アクション決定の流れが切れていないか。
   - **ゴール後付け禁止**: PoC実行後に成功基準を変えていないか。提案書の基準が正。

6. **検査ゲート（出力前・必須停止）**: 共通7項目＋PoC評価固有3項目を回す。
   - ①入力✋確認済か ②必須項目・品質バー（未確定は🟡） ③記号は正典どおりか ④正典を二重化していないか ⑤出力先パス・版番号 ⑥未確認の明示（確認しなかった範囲＋理由を一覧） ⑦サボり回収ループ（スキップ理由を再検証、サボりはやり直し）
   - ⑧提案書の評価軸4軸すべてに実績と判定があるか ⑨ノックアウトファクターの判定が明示されているか ⑩Go/NoGo推奨に根拠と条件があるか
   - 結果を ✅／🟡／❌ で出し、最終判断は本人に返す。✋✋承認を得る（Go/NoGoはプロジェクトの方向を決める不可逆判断）。
   - **結果はレシート書式で必ず提示する**（チャットに出す＋`{案件名}/_検査ログ.md`へ1行追記）。

7. **承認後**: 確定事実を `project-context.md` へ書き戻す。
   - 出力先 `{案件名}/05_終結/NN_PoC評価報告書_vX.md`。
   - 希望があればWord（.docx）化。

## 停止ポイント

- 手順6の検査ゲートは必須。Go/NoGo判定はプロジェクトの方向を決める不可逆判断＝✋✋（二重停止）。
- ノックアウトファクターが不合格の場合、NoGo推奨を明示したうえで「それでもGoにする条件」をユーザーに問う。

## 出力

- `{案件名}/05_終結/NN_PoC評価報告書_vX.md` →（希望時）`.docx`

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う。
- 記号は `references/表と数値の見せ方.md` の正典（他軸へ流用しない）。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#9／#7）。
- 多段階で勝手に進めない。✋✋承認なしにGo/NoGo判定を確定しない。
- 各ステップ開始時に現在地を明示する（例:「終結 PoC評価報告書」）。
- 入力はPoC提案書・PoC実行結果・project-contextのみ。提案書に無い評価基準を勝手に足さない。
- **ゴール後付け禁止**: PoC実行後に成功基準を変えない。提案書の基準が正。結果に合わせて基準を緩めるのはアンチパターン。
- PoCの「失敗」も成果。期待と異なる結果を正直に報告し、その学びを§4課題と対策に残す。
