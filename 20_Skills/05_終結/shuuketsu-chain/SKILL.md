# 終結フェーズ（親オーケストレーター）

監視で巻き上げた事実 → 達成を確認し、残りを引き継ぎ、教訓を残して正式に閉じるところまで。中身の知識は持たない。順番と停止だけを守る。
終結の本質は「終わらせる」だけでなく「**正しく閉じて次へ渡す**」——達成確認・残課題の引き継ぎ・教訓・終結宣言の4つを欠かさない。これは最後のフェーズなので、次工程は別案件ではなく運用・次プロジェクトへの引き渡し。

> **★ 現状: 型（骨）のみ。子Skillは project-closeout（終結報告書）のみ実装済み ★**
> 契約・調達のクローズ／実績の最終集計／顧客満足の確認等の子Skillはまだ無い。本書は終結フェーズを構造として閉じるための型。
> 中身は実案件で「伴走→写し取り」方式（[[ai-project-leader-design-principles]]）で詰める。接続規約は `00_プロジェクト管理/接続規約_前段→本体_v0.1.md` の様式に準じる。

## 実行手順

1. **入力を読む**: 案件のここまでの正本＝`01_立ち上げ/…チャーター`・`02_計画/01_統合計画書_vX.md`（§2受入基準・§4ゲート・§3リスク登録簿）・`03_実行/01_課題管理表_vX`・`04_監視コントロール/01_週次状況報告_vX`・`project-context.md`。生データは `project-context.md` をポインタ経由で参照する。
2. **現在地と種別を明示**: 「フェーズ＝終結／本質は"正しく閉じて次へ渡す"」を提示し、**種別（フェーズ終結／プロジェクト終結）を確定**する。閉じる対象と巻き上げ範囲を先に決める（おまかせにしない）。
3. **01 終結報告書（#9 プロジェクトまたはフェーズの終結）** を project-closeout で実行 → ✋提出版（docx）確定前に「閉じる4要素（達成確認・引き継ぎ・教訓・終結宣言）が揃っているか」「残課題・不可逆リスクが引き継がれているか」を確認（**必須停止**）。
4. **（将来の子Skill・未実装）** 調達・契約のクローズ／実績の最終集計（スコープ・スケジュール・コスト・品質）／顧客満足の確認。テーラリングで要るものだけを足す。
5. **完了後（閉じる）**: 確定事実を `project-context.md` へ書き戻し、終結報告書を `05_終結/` にアーカイブ。フェーズ終結なら次フェーズへ、プロジェクト終結なら運用・次プロジェクトへ引き継ぎ、本フェーズの終結を宣言する。次のフェーズ境界の引き継ぎ書（あれば）を起案する。
   - あわせて **テーラリング判断ログを終結報告書§5教訓へ回収する**（省いて正解だったか／痛かったかを残す。規約: `40_Stock/横断ガイドライン/テーラリング判断ログ.md`）。

## ルール

- 各ステップの入力は **これまでの正本（チャーター／統合計画書／課題管理表／リスク登録簿／状況報告）／project-context.md のみ**。
- 出力先と命名: `{案件名}/05_終結/01_終結報告書_vX.docx` 形式（フェーズ別サブフォルダ＋フェーズ内ローカル採番。番号空間はフェーズごとに独立）。
- 停止ポイントを勝手にスキップしない。承認・確認なしに提出版を確定しない。
- 各ステップ開始時に現在地と種別を明示する（例:「終結 01／終結報告書・フェーズ終結（段階1）」）。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#9。教訓は#5の到達点）。
