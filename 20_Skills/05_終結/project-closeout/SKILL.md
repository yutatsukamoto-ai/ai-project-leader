# 終結報告書 作成（子Skill）

終結フェーズの基幹成果物（#9 プロジェクトまたはフェーズの終結）。
約束した到達点に「届いたか」を確認し、残課題・残リスクを次へ正しく渡し、教訓を残して正式に閉じる。
中身の品質基準・節構成・巻き上げの引きどころは `references/型と品質基準.md` を見る（Progressive Disclosure。本体は薄く保つ）。

## 種別（入口で必ず決める）

- **フェーズ終結**: 段階・局面の節目で閉じる。巻き上げ範囲＝そのフェーズ。残りは次フェーズへ引き継ぐ。
- **プロジェクト終結**: 案件全体を閉じる。巻き上げ範囲＝全期間・全スコープ。残りは運用・次プロジェクトへ引き継ぐ。
- 型（節構成）は同じ。種別で§2達成度・§3スコープ・§5教訓の巻き上げ範囲だけ広げる。

## 手順

1. **種別を確定**: フェーズ終結／プロジェクト終結。閉じる対象と巻き上げ範囲を先に決める。
   - 種別確定と同時に、`project-context.md` の「現在のフェーズ」を終結に更新する（サブエージェントが読む前に最新状態にしておく）。

2. **入力を並列読み取り（サブエージェント）**: 5+ソースを**サブエージェント（Agentツール）で並列に読み取り・要約**する（詳細は下記§サブエージェント設計）。

3. **巻き上げる（停止・壁打ち）**: サブエージェントの中間ファイルを読み、達成と残りを統合する。`references/型と品質基準.md` の引きどころに沿う。
   - 成功基準（チャーター§4）→ §2を**計画 vs 実績の3値**（達成／一部／未達）で。状態は言葉。
   - 受入基準・ゲート通過条件（統合計画書§2・§4）→ §3完了確認。
   - 課題管理表のオープン／リスク登録簿の状態 → §4で「閉じた／引き継ぐ」に仕分け。**ここは2つのサブエージェント出力を突合し、引き継ぎ漏れがないか確認する。**
   - フェーズ全体を振り返り → §5教訓（続ける／変える／申し送り）。ソース4（project-context）だけでなく、ソース2（課題の閉じ方）・ソース3（状況報告の推移）の中間ファイルも材料にして教訓の具体性を補強する。

4. **一枚に組む**: 物語の順（閉じる宣言→達成→詳細→引き継ぎ→教訓→締め）で7節に収める。
   - サマリー（終結の宣言＋ダッシュボード1行）／達成度（計画vs実績）／スコープ完了確認／残課題・残リスクの引き継ぎ／教訓登録簿／成果物の引き渡しと保管／終結の宣言と次の意思決定。
   - 教訓は独立Skillにせず**§5として内包**する。
   - 統合後、中間ファイルは削除する。

5. **出力（停止・承認後）**: 最終形は **Word（.docx）**＝正式な終結・引き継ぎ文書。
   - 体裁: A4・游ゴシック・見出しに色＋下線・表ヘッダーに薄い背景・状態セルを色分け・一文一改行（状況報告と同じ家の流儀）。
   - 運用: マスターはマークダウン、提出用docx変換は人の確認後。
   - 出力先 `{案件名}/05_終結/01_終結報告書_vX.docx`。

## サブエージェント設計（手順2の詳細）

設計基準: `40_Stock/横断ガイドライン/サブエージェント設計基準.md` に準拠。

### なぜ並列化するか

終結報告書は5+ソースをプロジェクト全期間にわたって読む。直列に読むとコンテキストが圧迫され、後半のソース（特にリスク登録簿・状況報告）の読み取り品質が落ちる（原則A-2）。
さらに、各ソースを初見のエージェントが読むことで、メインが見落としがちな「残課題と残リスクの不整合」を構造的に拾いやすくなる。

### 起動方法

手順2で、Agentツールを**1つのメッセージ内に4つ同時に**呼び出す（並列実行）。
各エージェントは担当ソースのみ読む（原則A-1）。

### 各エージェントの仕様

**共通ルール（4エージェント共通でpromptに含める）:**
- 種別（フェーズ終結／プロジェクト終結）と巻き上げ範囲を渡す
- 結果はファイルに書き出す（原則A-3）。パスは `outputs/closeout_source_{番号}.md`
- メインへの戻り値は「○件抽出。詳細はファイル参照」程度の要約のみ
- 「8ステップ以内で完了せよ」と明記（原則A-4: 暴走防止）
- 成果物ファイルは読み取りのみ。編集しない

| エージェント | description | 担当ソース | 抽出する内容 | 出力ファイル |
|---|---|---|---|---|
| ソース1: チャーター＋統合計画書 | 「チャーターと統合計画書から成功基準・受入基準・ゲート条件を抽出」 | チャーター§4、統合計画書§2・§4 | 成功基準一覧、受入基準、ゲート通過条件 | `outputs/closeout_source_1.md` |
| ソース2: 課題管理表 | 「課題管理表から全課題のステータスを抽出し、閉じた/オープンに仕分け」 | 課題管理表 | オープン件数、閉じた件数、引き継ぎ候補の一覧 | `outputs/closeout_source_2.md` |
| ソース3: リスク登録簿＋状況報告 | 「リスク登録簿の状態と状況報告の推移から残リスク・経緯を抽出」 | リスク登録簿、状況報告 | アクティブリスク一覧、状態推移の要約、不可逆リスクのフラグ | `outputs/closeout_source_3.md` |

### 統合時の突合チェック（手順3の注意点）

メインが4本の中間ファイルを統合する際、以下を必ず突合する:
- ソース2（課題管理表のオープン課題）とソース3（リスク登録簿のアクティブリスク）で、**同じ事象が課題とリスクの両方に存在しないか**、または**片方にしかなく引き継ぎ漏れになっていないか**を確認
- ソース1（受入基準）とソース2・3（実績）を照合し、§3スコープ完了の判定が下流と矛盾しないことを確認

突合で不整合を発見した場合、§4に「不整合として検出→どちらを採用したか」を明記する。

## ルール

- **生きた台帳でなく、フェーズ/プロジェクトを閉じるコミュニケーション文書**。課題管理表・リスク登録簿（台帳）とは役割が違う。終結報告書は達成と引き継ぎを巻き上げる物語型。
- **状態は言葉**（達成／一部／未達・順調／注意／要対応）。確度（●◐○）・採否（◎○△）を状態に流用しない（`references/表と数値の見せ方.md`）。
- **数字をぼかさない**。受入基準・残課題件数・期日・ゲート条件は具体で。引き継ぐ項目は [引継ぎ]、意思決定が要る項目は [要対応] を頭に付ける。
- **閉じるとは4つ**: 達成を確認し・残りを引き継ぎ・教訓を残し・正式に終わりを宣言する。どれが欠けても終結報告にならない。
- 文体は `references/文章作法.md`（6ルール）。一文一改行・ラベル見出し。Word内では絵文字を文字に置換（[要対応]／[引継ぎ]等）。
- プロセス参照は `references/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#9。教訓は#5プロジェクト知識のマネジメントの到達点）。
- 多段階で勝手に進めない。✋承認なしに提出版（docx）を確定しない。各ステップ開始時に現在地と種別を明示する（例:「終結 01／終結報告書・フェーズ終結（段階1）」）。
