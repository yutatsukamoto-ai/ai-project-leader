### 未整備のうち Skill化 優先候補（たたき台・決めるのは本人）

実案件での出現頻度×「認識を揃える」効果の目安。テーラリング前提（毎回作る意味ではない）。

1. **スコープ定義書／WBS** … スコープの認識合わせの核。いまは統合計画書に簡易内包＝独立で欲しくなりやすい。
   - 逆転条件（着手トリガー）: スコープの認識ズレが課題化／統合計画§2の簡易内包では粒度が足りなくなったら独立化（WBSは `wbs-builder` でSkill済）。
2. **ステークホルダー＆コミュニケーション計画** … 人と期待値の整理。実行の土台で抜けると後で効く。
   - 逆転条件: 関係者が複数部門に増える／報告経路が「月1報告会」等で足りなくなったらコミュニケーション計画を独立化（ステークホルダー登録簿は `stakeholder-register` でSkill済）。
3. **スケジュール（ベースライン）** … 律速と期日の見える化。クリティカルパス止まりを前に進める。
   - 逆転条件: クリティカルパス止まりで対外的な期日コミットが要る／全体ベースラインで進捗管理する要求（D5スピード重）が出たら #16-17 を厚くする。
4. **リスク管理計画／定量分析** … 簡易内包の一歩先。不可逆リスクのある案件で要る。
   - 逆転条件: 不可逆・重大リスク（漏洩・契約・安全）が複数化／定量評価を求められたら統合§3の簡易内包の一歩先へ（#35-40）。
5. **コスト見積もり／コストベースライン** … 予算が論点の案件では必須。
   - 逆転条件: 予算が承認・説明の論点になる（D2説明責任重）／PoC後に本予算を引く段になったらコスト見積・コストBL（#19-22）を作成。

## バトンの流れ（一本道）

案件理解(0-1) → 課題仮説(0-2) → リサーチ(0-3) → 計画提案(0-4)
　→ ［引き継ぎ書］ → チャーター(01) → 統合計画(02) → 課題管理(03)
　→ 監視(04：状況報告／必要時 変更管理) → 終結報告(05)

監視は節目で回し、見つけた矛盾を課題管理（手を打つ）か変更管理（基準を変える）へ戻す。
