# PoC提案書 作成（子Skill）

計画フェーズの基幹成果物（#2 プロジェクト計画の統合と整合・PoC特化）。
承認済み計画提案書（前段0-4）を入力に、**ステークホルダーがPoCの実施可否を判断できる提案書**を作る。
中身の品質基準・節構成は `references/型と品質基準.md` を見る。

## 位置づけ

計画提案書（0-4）が「PoCで検証してから判断する」という進め方を提案した場合に起動する。
計画提案書 → **PoC提案書**（本Skill）→ PoC実行 → PoC評価報告 の流れ。
統合計画書の代わりではなく、**PoC型案件における統合計画書の役割**を担う。

## 手順

1. **入力を読む**: `{案件名}/00_前段/04_計画提案書`・`project-context.md`・`references/型と品質基準.md`。
   - 計画提案書の✋承認が済か確認する。未なら止める。
   - `project-context.md` が無ければ止めず、承認後の書き戻し（手順6）で新規作成する。

2. **フィージビリティスタディの結果を整理**: 前段チェーンで出た課題仮説・リサーチ結果から、ノックアウトファクター（これが駄目なら先に進めない要素）を特定する。

3. **中身を埋める**: `references/型と品質基準.md` の11項目に沿って埋める。
   - 評価軸は4軸（ビジネス・業務／技術／運用／コスト）を必ず設定する。
   - 【差し込みカード参照枠】ドメインカードがあれば引く（無ければ素通り）。
   - PoCキャンバス（`references/PoCキャンバス.md`）形式での視覚化を検討する。

4. **推測には根拠を明示**: 推測は出典を併記し、未確定は🟡で残して次工程への申し送りにする。
   - 記号は `references/表と数値の見せ方.md` の正典どおり（確度`●◐○`／採否`◎○△`／状態は言葉）。

5. **アンチパターンチェック**: `references/PoCアンチパターン.md` の7パターンに照らして、計画段階で陥っていないか確認する。
   - 特に「とりあえずPoC」「一本鎗PoC」「際限なきPoC」が計画段階で発生しやすい。

6. **検査ゲート（出力前・必須停止）**: 共通7項目＋PoC固有2項目を回す。
   - ①入力✋確認済か ②必須項目・品質バー（未確定は🟡） ③記号は正典どおりか ④正典を二重化していないか ⑤出力先パス・版番号 ⑥未確認の明示（確認しなかった範囲＋理由を一覧） ⑦サボり回収ループ（スキップ理由を再検証、サボりはやり直し）
   - ⑧評価軸4軸すべてに評価項目があるか ⑨ノックアウトファクターが特定されているか
   - 結果を ✅／🟡／❌ で出し、最終判断は本人に返す。✋承認を得る。
   - **結果はレシート書式で必ず提示する**（チャットに出す＋`{案件名}/_検査ログ.md`へ1行追記）。

7. **承認後**: 確定事実を `project-context.md` へ書き戻す。
   - 出力先 `{案件名}/02_計画/NN_PoC提案書_vX.md`。
   - 希望があればWord（.docx）化。

## 停止ポイント

- 手順6の検査ゲートは必須。承認なしにPoC実行へ進まない。
- PoC期間・予算に制約がある場合、評価項目の優先順位付けでユーザーに判断を返す。

## 出力

- `{案件名}/02_計画/NN_PoC提案書_vX.md` →（希望時）`.docx`

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う。
- 記号は `references/表と数値の見せ方.md` の正典（他軸へ流用しない）。
- プロセス参照は `references/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#2）。
- 多段階で勝手に進めない。✋承認なしに次工程／提出版を確定しない。
- 各ステップ開始時に現在地を明示する（例:「計画 PoC提案書」）。
- 入力は計画提案書（0-4）とproject-contextのみ。新しい事実を勝手に足さない。
- 「やらないこと・やれないこと」を必ず明文化し、いつやるかも記載する。
