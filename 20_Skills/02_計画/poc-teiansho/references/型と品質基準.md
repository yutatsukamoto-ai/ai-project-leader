# PoC提案書 型と品質基準

出典: 『IoTプロジェクトで学ぶ実践PoC』Ch4（計画-準備）＋Ch10（実践編）より再構成。
アイデアの再構成であり、表現・図版の転載はしない。

## 文書の型（11項目・この順で埋める）

1. **背景と目的** — なぜこのPoCが必要になったか（背景）、PoCを通して何を実現したいか（目的）をステークホルダー間で共有する。目的が不明瞭だと評価すべき内容が定まらず、収拾がつかなくなる。
2. **制約事項** — 予算上限、期間制約、利用可能なリソースなど既知の前提。制約を先に確認しないと評価項目の優先度が決まらない。
3. **スコープ** — PoCとして評価する領域・対象を定める。全てを検証するのではなく、机上で評価しきれない動作・機能を中心に。
4. **やらないこと・やれないこと** — スコープ外を明文化。暗黙の了解を作らない。さらに「いつやるか」も決める（次フェーズ／別PoC／机上評価で済ます／やらない）。
5. **評価軸および評価項目** — 4軸で網羅的に洗い出す（下記参照）。ノックアウトファクターを特定し最優先にする。評価項目は定量的（できた/できない）と定性的（特性や癖の把握）の2種を意識する。
6. **システム構成・開発対象** — 評価に必要なプロトタイプの構成。本番同等は不要だが、評価のエッセンスを抽出した構成をデザインする。
7. **評価の実施環境** — どこでPoCを実施するか。実際の現場が理想。代替環境の場合はデータ取得条件（時間的・人的・環境的要因）が本番と乖離しないか検証する。
8. **遵守すべきレギュレーション** — 法的要件、セキュリティ要件、社内規定など。PoC段階でも最低限必要な対応を整理する。
9. **実施スケジュール** — PoC期間、マイルストン（デモ／中間報告／最終報告）。疎通確認期間を初期に設ける。
10. **体制** — PM、エンジニア、ビジネスドメイン担当、レポートライン。現場担当者の協力も明記。
11. **タスクと担当者** — 準備作業・TODOを担当者に割り当て。現場側への依頼事項も整理。

## 評価の4軸（必須）

| 軸 | 観点 |
|---|---|
| ビジネス・業務 | 施策が期待する収益改善・業務効率化の効果を得られそうか |
| 技術 | 採用する技術・システムが期待通りに動き、利用できるか |
| 運用 | 仕組みを無理なく運用できるか。教育コスト、メンテナンス性 |
| コスト | 導入・運用に必要なコストは適切か。ROI算出の基礎情報 |

4軸すべてに1つ以上の評価項目を必ず設定する。
技術面だけに偏らない（DXの文脈ではPoC＝技術検証と誤解されやすい）。

## ノックアウトファクターの扱い

ノックアウトファクター＝「これが駄目なら先に進めない」要素。
評価項目の中から特定し、最優先で検証する。
フィージビリティスタディ（前段チェーン）で候補は見えているはず。

## 品質バー（検査ゲートで使う）

- [ ] 11項目すべてに記載があるか（「該当なし」も明記なら可）
- [ ] 評価4軸すべてに評価項目があるか
- [ ] ノックアウトファクターが特定されているか
- [ ] 「やらないこと」に「いつやるか」が併記されているか
- [ ] スケジュールにマイルストン（最低: 中間報告・最終報告）があるか
- [ ] 体制にビジネスドメイン担当が含まれているか（技術だけで閉じていないか）
- [ ] 評価項目に定量的・定性的の両方が含まれているか

## PoCの3つの役割（忘れないこと）

1. **実現性の評価** — 4軸で総合的に判断する
2. **学びと発見** — 机上では気づけないポイントの発見。定性的評価項目がこれを担う
3. **予算確保** — 本番導入の費用見積りの基礎情報を得る。コスト軸の評価項目がこれを担う

提案書の「目的」セクションで、このPoCがどの役割を主に担うかを明示する。
