# 計画フェーズ（親オーケストレーター）

立ち上げ→計画の引き継ぎ書 → 統合計画書（またはPoC提案書）の承認まで。中身の知識は持たない。順番と停止だけを守る。
計画の本質は「全部を厚く描く」ことではなく「立ち戻る一枚（チャーター）を、実行できる計画に翻訳する」——要求・スコープ・リスクを束ね、未確定はローリングウェーブで操舵する。

> **★ 現状: 子Skillは integrated-plan（統合計画書）・wbs-builder（WBS）・poc-teiansho（PoC提案書）が実装済み ★**
> 詳細スケジュール・コスト見積・調達計画・コミュニケーション計画等の子Skillはまだ無い。本書は計画フェーズを構造として閉じるための型。
> 中身は実案件で「伴走→写し取り」方式（[[ai-project-leader-design-principles]]）で詰める。接続規約は `00_プロジェクト管理/接続規約_前段→本体_v0.2.md` の様式に準じる。

## 実行手順

1. **入力を読む**: `{案件名}/01_立ち上げ/引き継ぎ書_立ち上げ→計画.md`（接続規約の様式に準じる凍結スナップショット）。冒頭メタの「✋人による確認: ☐済」が立っているか確認。**未確認なら止める**。引き継ぎ書が無ければ `01_立ち上げ/01_プロジェクトチャーター_vX.md`（✋承認済）と `project-context.md` を直接入力にしてよい。生データは案件ルート直下の `project-context.md` をポインタ経由で参照する。
2. **現在地を明示**: 「フェーズ＝計画／本質は"チャーターを実行できる計画に翻訳する"」を提示。チャーターのテーラリング（採用/省略の#プロセス）に従い、計画で作る成果物を確定する（おまかせにしない）。
3. **01 統合計画書（#2 計画の統合と整合）** を integrated-plan で実行 → ✋承認前に「ローリングウェーブの段階分け」「要求の3層化」「不可逆リスクの点検」「計画系プロセスのテーラリング記録」を確認（**必須停止**）。
4. **（PoC型の場合）** 計画提案書が「PoCで検証してから判断する」進め方を示していれば、統合計画書の代わりに `poc-teiansho`（PoC提案書）を起動する。PoC提案書は統合計画書のPoC版であり、PoCで何を・どの評価軸で・どこまで検証するかをステークホルダーに提案し Go 判断をもらう一枚。
5. **（将来の子Skill・未実装）** スコープ定義（#10-13）／スケジュール（#16-17）／コスト（#19-21）／品質計画（#33）／調達戦略（#3）／コミュニケーション計画（#25）。テーラリングで要るものだけを順に足す。
5. **完了後**: 確定事実を `project-context.md` へ書き戻し、次のフェーズ境界の引き継ぎ書（計画→実行）を起案 → 次は実行であることを案内。統合計画書（§3リスク登録簿を含む）はそのまま実行フェーズの入力になる。
   - 計画系プロセスの採用／省略／保留を **テーラリング判断ログへ追記・更新する**（規約: `40_Stock/横断ガイドライン/テーラリング判断ログ.md`）。

## ルール

- 各ステップの入力は **引き継ぎ書／チャーター／前ステップの成果物／project-context.md のみ**。
- 出力先と命名: `{案件名}/02_計画/01_統合計画書_vX.md` 形式（フェーズ別サブフォルダ＋フェーズ内ローカル採番。番号空間はフェーズごとに独立）。
- 停止ポイントを勝手にスキップしない。承認・確認なしに次へ進まない。
- 各ステップ開始時に現在地を明示する（例:「計画 01／統合計画書」）。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う。
