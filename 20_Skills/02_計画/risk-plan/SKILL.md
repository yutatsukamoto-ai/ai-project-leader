# リスク管理計画 作成（子Skill）

計画フェーズの基幹成果物（#35 リスクマネジメントの計画 / #36 リスクの特定 / #37 リスク分析 / #38 リスク対応の計画）。
統合計画書§3が簡易内包しているリスクを**独立した管理計画＋登録簿**に展開する。
中身の品質基準・節構成は `references/型と品質基準.md` を見る（Progressive Disclosure。本体は薄く保つ）。

## 位置づけ

統合計画書§3はリスクを「対応計画＋早期警戒の兆候」レベルで簡易内包している。
本Skillは案件の不確実性が高い場合に起動し、**リスクカテゴリ定義・確率×影響度の評価基準・対応戦略・リスク登録簿**を体系化する。
統合計画書§3の内容を引き継ぎ、独立した管理体制に格上げする位置づけ。

## 手順

1. **入力を読む**: `{案件名}/02_計画/01_統合計画書_vX.md`・`project-context.md`。
   - 統合計画書の✋承認が済かを確認する。未なら止める。
   - §3リスクの既存記述（対応計画・早期警戒の兆候・前提条件）を基盤にする。
   - `project-context.md` が無ければ止めず、承認後の書き戻し（手順7）で新規作成する。

2. **リスクカテゴリと評価基準を定義する**: `references/型と品質基準.md` の型に沿う。
   - リスクカテゴリ（技術・外部・組織・PM等）を案件に合わせて定義する。
   - 確率×影響度の評価基準（定量 or 定性）を明示する。
   - **確率×影響度マトリクス表を必ず出力する**（§2として。スコアから優先度が一意に決まるようにする。「確率×影響度で決定」の一文だけでは不可——マトリクス表がないと中×高と高×中の優劣が曖昧になる）。
   - 【差し込みカード参照枠】ドメインカードがあれば引く（無ければ素通り）。

3. **リスク登録簿を作成する**: 統合計画書§3のリスクを移行し、新規リスクを特定・追加する。
   - 各リスクに ID／カテゴリ／確率／影響度／リスクスコア／対応戦略／担当／トリガー／状態 を付与する。
   - **不可逆リスク（情報漏洩・契約違反・安全・法規制等）は✋✋で関所を設ける**。

4. **推測には根拠を明示**: 確率・影響度の評価根拠を添える。未確定（情報不足で評価できない等）は🟡で残す。
   - 記号は `references/表と数値の見せ方.md` の正典どおり。

5. **停止・壁打ち**: リスク対応戦略の妥当性（回避/軽減/転嫁/受容の選択根拠）と不可逆リスクの関所設計の素材を出す。
   - 曖昧・不可逆な分岐が出たら `grill-me`（`20_Skills/90_横断/`）で詰める。

6. **検査ゲート（出力前・必須停止）**: 共通7項目＋本Skill固有を回す（共通ひな形§検査ゲート）。
   - 共通: ①入力✋確認済か ②必須項目・品質バー（未確定は🟡） ③記号は正典どおりか ④正典を二重化していないか ⑤出力先パス・版番号 ⑥未確認の明示（確認しなかった範囲＋理由を一覧） ⑦サボり回収ループ（スキップ理由を再検証、サボりはやり直し）。
   - 固有: ⑧不可逆リスクに関所（✋✋）があるか ⑨全リスクに対応戦略と担当があるか ⑩確率×影響度の評価基準が明示されているか。
   - 結果を ✅／🟡／❌ で出し、最終判断は本人に返す。✋承認を得る。
   - **結果はレシート書式で必ず提示する**（チャットに出す＋`{案件名}/_検査ログ.md`へ1行追記）。

7. **承認後**: 確定事実を `project-context.md` へ書き戻す。
   - 出力先 `{案件名}/02_計画/NN_リスク管理計画_vX.md`。

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う。
- 記号は `references/表と数値の見せ方.md` の正典（他軸へ流用しない）。
- プロセス参照は `references/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#35/#36/#37/#38）。
- 統合計画書§3との**二重化をしない**＝§3は簡易版の台帳、本計画は独立した管理体制。§3のリスクは本登録簿に移行し、§3からは参照リンクに切り替える。
- 多段階で勝手に進めない。✋承認なしに次工程／提出版を確定しない。
- 各ステップ開始時に現在地を明示する（例:「計画 リスク管理計画」）。
