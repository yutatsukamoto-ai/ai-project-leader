# スコープ管理計画 作成（子Skill）

計画フェーズの基幹成果物（#10 スコープマネジメントの計画 / #11 要求事項の引き出しと分析 / #12 スコープの定義）。
統合計画書§2スコープの簡易記述を、**要求事項の粒度で精緻化**し、スコープクリープ防止の基盤にする。
中身の品質基準・節構成は `references/型と品質基準.md` を見る（Progressive Disclosure。本体は薄く保つ）。

## 位置づけ

統合計画書§2スコープは「何をやるか/やらないか」の骨格。
本Skillはその骨格を**要求事項一覧・スコープ定義書・WBS辞書**の粒度に展開する。
wbs-builder（#13）が出力するWBS構造（wbs.json）に、各ワークパッケージの詳細定義（WBS辞書）を付与する補完関係。
両者は補完であり重複ではない＝WBS構造はwbs-builder、スコープ定義とWBS辞書は本Skill。

## 手順

1. **入力を読む**: `{案件名}/02_計画/01_統合計画書_vX.md`・`project-context.md`。前段の計画提案書（`{案件名}/00_前段/04_計画提案書_vX.md`）があればそれも参照する。
   - 統合計画書の✋承認が済か確認する。未なら止める。
   - `project-context.md` が無ければ止めず、承認後の書き戻し（手順7）で新規作成する。

2. **スコープ管理方針を定める**: `references/型と品質基準.md` の型に沿う。
   - 変更管理プロセス（スコープ変更の申請→評価→承認フロー）を定義する。
   - 受入基準の定義方法（誰が・いつ・どの基準で検証するか）を明記する。
   - 【差し込みカード参照枠】ドメインカードがあれば引く（無ければ素通り）。

3. **要求事項一覧を組む**: 統合計画書§1要求と計画提案書の課題仮説を展開する。
   - ビジネス要求／ステークホルダー要求／ソリューション要求／移行要求の4区分で整理する。
   - 各要求にID・優先度・出典・受入基準・確度（●◐○）を付与する。
   - スコープ定義書を埋める: 含むもの／含まないもの／前提条件／制約条件を明示する。

4. **WBS辞書を組む**: wbs-builderのWBS構造（wbs.json）がある場合はそれと整合させる。
   - 各ワークパッケージにID・名称・説明・受入基準・担当・見積工数を定義する。
   - WBSが未作成の場合は統合計画書§2の簡易WBSを基にWBS辞書の骨格を先行作成し、wbs-builder実行後に整合させる旨を🟡で残す。

5. **推測には根拠を明示**: 要求の確度・優先度の判断根拠を添える。未確定（出典未確認・受入基準未定等）は🟡で残す。
   - 記号は `references/表と数値の見せ方.md` の正典どおり。

6. **停止・壁打ち**: 含むもの／含まないものの線引き、要求の優先順位づけの素材を出す。
   - 曖昧・不可逆な分岐が出たら `grill-me`（`20_Skills/90_横断/`）で詰める。

7. **検査ゲート（出力前・必須停止）**: 共通7項目＋本Skill固有を回す（共通ひな形§検査ゲート）。
   - 共通: ①入力✋確認済か ②必須項目・品質バー（未確定は🟡） ③記号は正典どおりか ④正典を二重化していないか ⑤出力先パス・版番号 ⑥未確認の明示（確認しなかった範囲＋理由を一覧） ⑦サボり回収ループ（スキップ理由を再検証、サボりはやり直し）。
   - 固有: ⑧全要求事項にID・優先度・受入基準があるか ⑨含むもの/含まないものが明示されているか ⑩WBS辞書の粒度がWBSと整合しているか。
   - 結果を ✅／🟡／❌ で出し、最終判断は本人に返す。✋承認を得る。
   - **結果はレシート書式で必ず提示する**（チャットに出す＋`{案件名}/_検査ログ.md`へ1行追記）。

8. **承認後**: 確定事実を `project-context.md` へ書き戻す。
   - 出力先 `{案件名}/02_計画/NN_スコープ管理計画_vX.md`。

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う。
- 記号は `references/表と数値の見せ方.md` の正典（他軸へ流用しない）。
- プロセス参照は `references/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#10/#11/#12）。
- 統合計画書§2との**二重化をしない**＝§2は簡易の骨格、本計画は精緻化した台帳。役割を分けリンクで繋ぐ。
- wbs-builderとの**二重化をしない**＝WBS構造（wbs.json）はwbs-builder、WBS辞書（各パッケージの定義）は本Skill。
- 多段階で勝手に進めない。✋承認なしに次工程／提出版を確定しない。
- 各ステップ開始時に現在地を明示する（例:「計画 スコープ管理計画」）。

## chain-trace.json

- 案件ルートに `chain-trace.json` がある場合は、検査ゲート通過後・✋承認後に本Skillのエントリを追記する（規約: `40_Stock/横断ガイドライン/チェーントレース規約.md`）。無い場合（前段のみ等）はスキップしてよい。
