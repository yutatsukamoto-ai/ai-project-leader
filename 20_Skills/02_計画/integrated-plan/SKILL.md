# 統合計画書作成（子Skill）

計画フェーズの基幹成果物（#2 計画の統合と整合）。チャーターを実行可能な計画に翻訳した一枚を作る。
MVP=簡易版＝要求・スコープ・リスクを束ねる。日程・費用は厚くしない（テーラリング○）。
中身の品質基準は `references/型と品質基準.md` を見る（Progressive Disclosure。本体は薄く保つ）。

## 手順

1. **入力を読む**: `{案件名}/01_立ち上げ/01_プロジェクトチャーター_vX.md` と `project-context.md`（案件ルート直下）。
   - チャーターが✋承認済かを確認。未承認なら止める（未確定版を計画に流さない）。
   - チャーターの「固定する事項／学習に委ねる事項」とテーラリング（採否の#プロセス）を、計画書の濃淡に引き継ぐ。

2. **粒度方針を決める（停止・壁打ち）**: 下流（要求・対象業務）が未確定なら**ローリングウェーブ**を採る。
   - 直近フェーズ（段階1）は実行できる粒度まで詳細化。以降は条件付きの骨格に留め、ゲート通過後に詳細化。
   - 案件が段階型でなければ全体を均等粒度にする選択もある。案件の段階性を見て選び、理由を添えて確認する。

3. **4本立てで束ねる（節ごとに判断を見せて確認）**: `references/型と品質基準.md` の型に沿って埋める。
   - **§1 要求（#11）**: アウトカム＋機能の方向性（仮説）＋制約の3層で書く。確定すべき要求は🟡に切り分ける。チャーターの「定着の約束」等は**制約要求に格上げ**する。
   - **§2 スコープ（#10/12/13）**: やらないこと先置き／簡易WBS（直近段階のみ詳細）／受入基準（各段階1行）／**クリティカルパス（律速工程）を明示**。
   - **§3 リスク（#35-38）**: チャーターのリスクを「対応計画（先手）＋早期警戒の兆候」に展開。計画段階で新規に特定したリスクを足す。前提条件を集約。**不可逆リスク（情報漏洩・契約・安全等）を必ず点検**し、あれば起票する。
   - **§4 🟡の確定計画**: いつ・どの段階・どう確定するかの表＋ゲート通過条件。ローリングウェーブを「先送り」でなく「計画」にする固有節。空にしない。

4. **推奨項目の抜けチェック**: 計画系スコープ標準（**受入基準・前提条件**）と照らし、欠落を埋める。
   - 過剰なもの（WBS辞書・詳細スコープ記述書・#14妥当性確認手順）は簡易版に入れない。

5. **✋承認前の照合（必須停止）**: `references/型と品質基準.md` の照合節と `references/12ステップ要点.md` のデザインフェーズ4項目（5資源見積／6体制構築／7作業設計／8規範設計）を照合する。
   - あわせて**計画系プロセスのテーラリング記録**（載せた#／意識的に省いた#）を提示する。
   - 照合・メタ情報は本文に載せない（文章作法D項）。承認材料として別に示す。

6. **承認後**: 確定事実を `project-context.md` へ書き戻す。出力先 `{案件名}/02_計画/01_統合計画書_vX.md`。

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う。一文一改行・ラベル見出し・🟡で未確定を残す。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う。
- チャーター・0-4計画提案書との内容重複は許容する（統合計画書は別目的＝実行可能な計画。立ち戻る一枚＝チャーター、承認を得る提案＝0-4）。
- 多段階で勝手に進めない。✋承認なしに次工程へ渡さない。
- 各ステップ開始時に現在地を明示する（例:「計画 02／統合計画書」）。
