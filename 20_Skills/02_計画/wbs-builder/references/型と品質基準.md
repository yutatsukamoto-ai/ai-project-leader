# WBS 型と品質基準

出典: 統合計画書§2スコープ／PMBOK第8版 #13 スコープ構造の構築／WBS Viewer（single-file-wbs, MIT）の `CLAUDE.md`＝データ形式の上流仕様／伴走(山藤商事 2026-06-15)。
位置づけ: 統合計画書§2の簡易WBSを深掘りした「事実だけのWBS（wbs.json）」。工数・進捗・遅延は描画側で自動計算＝手でメンテする数字を持たない。

## 品質バー（このSkillの合格条件）

1. 統合計画書§2の段階・主な作業・受入基準・**律速**が、漏れなくWBSに写っている。
2. **事実（予定/実績の日付＋工数の素）だけ**を持つ（工数・進捗・イナズマ線を数値で持たない）。
3. **ローリングウェーブ**＝直近段階は詳細、先は骨格＋日付null。
4. §2と**二重化でなく深掘り・接続**になっている。

## データ形式（wbs.json）

```json
{ "projects": [ {
  "name": "プロジェクト名",
  "milestones": [ { "date":"YYYY-MM-DD", "label":"…", "color":"#ef4444" } ],
  "tasks": [
    { "id":"1", "name":"段階/工程", "children":[
      { "id":"1.1", "name":"作業(全角14字目安)", "qty":1, "hours":16, "assignee":"担当",
        "plan":{ "start":"YYYY-MM-DD", "end":"YYYY-MM-DD" },
        "actual":{ "start":null, "end":null }, "note":"" }
    ] }
  ] } ] }
```

- タスクは**最大3階層**。`children`あり＝集計ノード、なし＝リーフ（工数を持つ）。
- リーフ：`id, name, qty(数量), hours(1単位の時間), assignee, plan{start,end}, actual{start,end}, note`。`_`始まりは自由なカスタムキー（ビューア無視・保持）。
- **工数(人日)＝`qty×hours÷8`**、進捗・親集計・イナズマ線は描画側で自動計算（データに持たせない）。
- 日付は `YYYY-MM-DD`（1900〜2099）。未定は `null`。
- 計画段階では `actual` は全て `null`（実績は実行フェーズで入る）。

## 判断の分かれ目（人が確認すべき点。Skillはここで止まる/問う）

- **二重化回避**: 統合計画書§2は簡易の土台として残す。WBSは深掘りの別物。両方を正典化しない（リンクで繋ぐ）。
- **日付の置き方（最重要）**: ローリングウェーブなら **直近段階のみ仮の絶対日付・先の段階は `null`** が既定（バーは直近だけ出る＝正直）。全期間に仮日付を引くのは「先まで見せたい」時のみ・全部"仮"と明記。
- **粒度**: 直近段階のみリーフまで割り、先は集計1〜2行の骨格。3階層を超えない。
- **律速（クリティカルパス）**: 該当リーフの `note` に「★律速」。ここが遅れると全体が遅れる工程を明示。
- **不可逆の関所**: 漏洩・契約・安全等の不可逆リスクに関わる作業は `note` に関所（✋✋）を明記。
- **🟡の扱い**: 未確定の日付・担当は `null`、確定タイミング（どのゲートで決めるか）をnoteか申し送りに残す。

## 出力セット（承認後）

- `wbs.json`（本体）／`_閲覧版.html`（埋め込み単一HTML・ダブルクリック共有用）／WBS Viewer同梱（MIT帰属・編集とイナズマ線用）／更新依頼テンプレの案内。
