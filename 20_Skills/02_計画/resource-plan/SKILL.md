# 資源管理計画 作成（子Skill）

計画フェーズの基幹成果物（#30 資源マネジメントの計画 / #31 資源の見積り）。
統合計画書の体制・役割記述とWBSのワークパッケージを入力に、**資源見積もり＋体制・責任＋チームチャーター＋カレンダー**を一体化した計画にする。
中身の品質基準・節構成は `references/型と品質基準.md` を見る（Progressive Disclosure。本体は薄く保つ）。

## 位置づけ

統合計画書の体制記述を**具体的な配置・ルール・カレンダー**に展開する。
外部調達が絡む場合はソーシング戦略（#3 調達戦略の策定）との接続が発生する。
※正典の表記は「資源」（リソースではない）。

## 手順

1. **入力を読む**: `{案件名}/02_計画/01_統合計画書_vX.md`・`{案件名}/02_計画/NN_wbs.json`・`project-context.md`。
   - 統合計画書の✋承認が済か確認する。未なら止める。
   - WBSが無い場合は統合計画書のスコープ記述から概算で進めてよいが、タスク割り当ての精度が落ちる旨を🟡で明示する。
   - `project-context.md` が無ければ止めず、承認後の書き戻し（手順6）で新規作成する。

2. **資源見積もりと体制を組む**: `references/型と品質基準.md` の型に沿う。
   - 人的資源（ロール・スキル・人数）と物的資源（ツール・環境・設備）を区別して見積もる。
   - 【差し込みカード参照枠】ドメインカードがあれば引く（無ければ素通り）。

3. **RACIマトリクスを埋める**: WBSのワークパッケージ（またはデリバラブル）ごとに、Responsible / Accountable / Consulted / Informed を割り当てる。
   - 全タスクにAccountableが1人だけいることを確認する（RACI原則）。
   - 体制図（ロール・責任・権限の関係）を併記する。

4. **推測には根拠を明示**: 稼働率・兼務比率の前提を添える。未確定（メンバー未アサイン・外注先未選定等）は🟡で残す。
   - 記号は `references/表と数値の見せ方.md` の正典どおり。

5. **停止・壁打ち**: チームチャーター（グラウンドルール・意思決定プロセス・コンフリクト解決）の素案を出す。
   - 曖昧・不可逆な分岐が出たら `grill-me`（`20_Skills/90_横断/`）で詰める。

6. **検査ゲート（出力前・必須停止）**: 共通7項目＋本Skill固有を回す（共通ひな形§検査ゲート）。
   - 共通: ①入力✋確認済か ②必須項目・品質バー（未確定は🟡） ③記号は正典どおりか ④正典を二重化していないか ⑤出力先パス・版番号 ⑥未確認の明示（確認しなかった範囲＋理由を一覧） ⑦サボり回収ループ（スキップ理由を再検証、サボりはやり直し）。
   - 固有: ⑧全ロールに責任・権限の定義があるか ⑨RACIの全タスクに1人のAccountableがいるか ⑩チームチャーターにコンフリクト解決手順があるか。
   - 結果を ✅／🟡／❌ で出し、最終判断は本人に返す。✋承認を得る。
   - **結果はレシート書式で必ず提示する**（チャットに出す＋`{案件名}/_検査ログ.md`へ1行追記）。

7. **承認後**: 確定事実を `project-context.md` へ書き戻す。
   - 出力先 `{案件名}/02_計画/NN_資源管理計画_vX.md`。

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う。
- 記号は `references/表と数値の見せ方.md` の正典（他軸へ流用しない）。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#30/#31）。
- 統合計画書・WBSとの**二重化をしない**＝統合計画書は体制の概要と前提、WBSはスコープ構造、本計画は資源見積もり・体制詳細・チームチャーター・カレンダー。役割を分けリンクで繋ぐ。
- 多段階で勝手に進めない。✋承認なしに次工程／提出版を確定しない。
- 各ステップ開始時に現在地を明示する（例:「計画 資源管理計画」）。
