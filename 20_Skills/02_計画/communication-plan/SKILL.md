# コミュニケーション計画 作成（子Skill）

計画フェーズの基幹成果物（#24 ステークホルダーエンゲージメントの計画 / #25 コミュニケーションマネジメントの計画）。
ステークホルダー登録簿の「誰を・どこへ動かす」を、具体的な**コミュニケーション手段・頻度・責任者**に落とし込む計画にする。
中身の品質基準・節構成は `references/型と品質基準.md` を見る（Progressive Disclosure。本体は薄く保つ）。

## 位置づけ

ステークホルダー登録簿（#23）が「誰がいて、どう動かすか」の台帳。
本Skillはその戦略を**実行可能な会議体・報告・情報共有の設計**に翻訳する。
登録簿の「巻き込み戦略」列と本計画が1対1で対応する。

## 手順

1. **入力を読む**: `{案件名}/01_立ち上げ/02_ステークホルダー登録簿_vX`・`{案件名}/02_計画/01_統合計画書_vX.md`・`project-context.md`。
   - ステークホルダー登録簿の✋承認が済か確認する。未なら止める。
   - 統合計画書が無い場合は、チャーターから直接作成してよい（登録簿が最低入力）。
   - `project-context.md` が無ければ止めず、承認後の書き戻し（手順6）で新規作成する。

2. **エンゲージメント計画を組む**: `references/型と品質基準.md` の型に沿う。
   - 登録簿の重点管理対象を中心に、**関与度の現在→目標のギャップ**から具体策を引く。
   - 【差し込みカード参照枠】ドメインカードがあれば引く（無ければ素通り）。

3. **コミュニケーションマトリクスを埋める**: 誰に／何を／いつ／どの手段で／誰が責任者。
   - 会議体（キックオフ・定例・ステコミ等）と非同期報告（週報・チャット等）を区別する。
   - 情報の過不足を防ぐ: 重点管理には手厚く、様子見には最小限。

4. **推測には根拠を明示**: 頻度・手段の選択根拠を添える。未確定（参加者未定・ツール未選定等）は🟡で残す。
   - 記号は `references/表と数値の見せ方.md` の正典どおり。

5. **停止・壁打ち**: エンゲージメント戦略の巻き上げ（登録簿の戦略列と矛盾がないか照合）の素材を出す。
   - 曖昧・不可逆な分岐が出たら `grill-me`（`20_Skills/90_横断/`）で詰める。

6. **検査ゲート（出力前・必須停止）**: 共通5項目＋本Skill固有を回す（共通ひな形§検査ゲート）。
   - 共通: ①入力✋確認済か ②必須項目・品質バー（未確定は🟡） ③記号は正典どおりか ④正典を二重化していないか ⑤出力先パス・版番号。
   - 固有: ⑥重点管理の全員にコミュニケーション手段があるか ⑦登録簿の巻き込み戦略と整合しているか ⑧エスカレーションパスが定義されているか。
   - 結果を ✅／🟡／❌ で出し、最終判断は本人に返す。✋承認を得る。
   - **結果はレシート書式で必ず提示する**（チャットに出す＋`{案件名}/_検査ログ.md`へ1行追記）。

7. **承認後**: 確定事実を `project-context.md` へ書き戻す。
   - 出力先 `{案件名}/02_計画/NN_コミュニケーション計画_vX.md`。

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う。
- 記号は `references/表と数値の見せ方.md` の正典（他軸へ流用しない）。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#24/#25）。
- ステークホルダー登録簿との**二重化をしない**＝登録簿は「誰が・どう動かす」の台帳、本計画は「どう伝える」の設計図。役割を分けリンクで繋ぐ。
- 多段階で勝手に進めない。✋承認なしに次工程／提出版を確定しない。
- 各ステップ開始時に現在地を明示する（例:「計画 コミュニケーション計画」）。
