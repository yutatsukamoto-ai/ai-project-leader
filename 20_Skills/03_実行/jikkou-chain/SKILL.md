# 実行フェーズ（親オーケストレーター）

計画→実行の引き継ぎ書 → 計画を実際に動かし、出たことを落とさず捌くところまで。中身の知識は持たない。順番と停止だけを守る。
実行の本質は「作る」だけでなく「ズレを捌く」——計画どおりに進まない事柄を課題に変え、担当・期限つきで閉じにいく。

> **★ 子Skill一覧（3 Skill）★**
> issue-log（課題管理表）・communication-management（コミュニケーション管理）・resource-management（資源管理）。
> テーラリングで要るものだけを選択して回す。接続規約は `00_プロジェクト管理/接続規約_前段→本体_v0.2.md` の様式に準じる。

## 実行手順

1. **入力を読む**: `{案件名}/02_計画/引き継ぎ書_計画→実行.md`（接続規約の様式に準じる凍結スナップショット）。冒頭メタの「✋人による確認: ☐済」が立っているか確認。**未確認なら止める**。引き継ぎ書が無ければ `02_計画/01_統合計画書_vX.md`（✋承認済）と `project-context.md` を直接入力にしてよい。生データは案件ルート直下の `project-context.md` をポインタ経由で参照する。
2. **現在地を明示**: 「フェーズ＝実行／本質は"計画を動かしズレを捌く"」を提示。統合計画書のテーラリング（採用/省略の#プロセス）とローリングウェーブの段階に従い、実行で動かす範囲を確定する（おまかせにしない）。
3. **01 課題管理表（#4 実行のマネジメント／#39 リスク対応策の実行）** を issue-log で実行 → 統合計画書§3リスク表にID（R-xx）が無ければ先に整える。✋承認前に「全課題に次の一手・担当・期限があるか」「リスクと混ざっていないか」を確認（**必須停止**）。
4. **コミュニケーション管理（#26/#27）**: テーラリングで採用していれば `communication-management` を起動。計画で定めたコミュニケーション計画を実行し、生きた台帳として管理する。
5. **資源管理（#32/#33）**: テーラリングで採用していれば `resource-management` を起動。計画で定めた資源管理計画を実行し、調達・配置・育成を台帳管理する。
6. **（未実装）** デリバラブルの作成（#4）／知識のマネジメント（#5）／品質保証（#6）。テーラリングで要るときに足す。
7. **完了後**: 確定事実を `project-context.md` へ書き戻し、次のフェーズ境界の引き継ぎ書（実行→監視・コントロール）を起案 → 次は監視・コントロールであることを案内。課題管理表とリスク登録簿はそのまま監視フェーズの入力になる。

## ルール

- 各ステップの入力は **引き継ぎ書／統合計画書／前ステップの成果物／project-context.md のみ**。
- 出力先と命名: `{案件名}/03_実行/01_課題管理表_vX.xlsx` 形式（フェーズ別サブフォルダ＋フェーズ内ローカル採番。番号空間はフェーズごとに独立）。
- 停止ポイントを勝手にスキップしない。承認・確認なしに次へ進まない。
- **リスクを課題化したら、計画§3リスク登録簿の状態を次revで「対応へ移管（移管先I-xx）」に更新する**。連動はID（R-xx↔I-xx）のリンクで持ち二重管理しない（実証メモ§2-B）。
- **完了後の `project-context.md` への書き戻しを必須ステップとして飛ばさない**（確定事実・🟡の解消をその都度反映。書き戻し先は案件ルート直下の1箇所）（実証メモ§2-C）。
- 各ステップ開始時に現在地を明示する（例:「実行 01／課題管理表」）。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う。
