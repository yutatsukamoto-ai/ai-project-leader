# 課題管理表 作成・運用（子Skill）

実行フェーズの基幹成果物（#4 実行のマネジメント／#39 リスク対応策の実行）。
計画どおりに進まない事柄を落とさず「次の一手」に変える、生きた台帳を作る。
中身の品質基準・列設計・連動ルールは `references/型と品質基準.md` を見る（Progressive Disclosure。本体は薄く保つ）。

## 手順

1. **入力を読む＋連動の前提を整える**: `{案件名}/02_計画/01_統合計画書_vX.md` と `project-context.md`。
   - 統合計画書§3のリスク表に **ID（R-01〜）と状態列** があるか確認する。
   - 無ければ連動が成立しない。リスク表にID列＋状態列を足し、計画書の版を上げる（承認済み版は据え置き、変更点を冒頭に明記）。← 連動はリンクで持つための前提。

2. **初期課題をseedする（停止・壁打ち）**: 計画書から「いま手を打つべきこと」を引く。
   - §4の🟡（確定すべき事項）→ 多くは「判断待ち」課題。
   - §3のリスクのうち、もう"かもしれない"でなく今動くもの → 「リスク対応」課題（登録簿の状態を「対応へ移管」に更新）。
   - ゲートの通過条件 → 段階出口までに閉じる課題。
   - 対象業務が未確定なら**仮置きで通し**、その特定自体を1課題（多くは律速）に立てる。

3. **各課題を1行に整える**: `references/型と品質基準.md` の列設計に沿う。
   - 必須: ID／区分（作業課題・判断待ち・リスク対応の3種）／課題／影響／優先度／次の一手／担当／期限／状態／関連。
   - **必ず「次の一手・担当・期限」を持たせる**。持てない課題は課題でなく、まだリスクか願望。
   - 関連列は登録簿のID（R-xx）や計画書の節を指す。

4. **連動を保つ（運用）**: リスク→課題へ移したら、中身は課題表にだけ書く（二重管理しない）。
   - 登録簿側は状態列で「対応へ移管（I-xx）」または「顕在化済」を示し、双方向に辿れる状態を保つ。
   - 課題が片付いたら状態を「完了」にする。**行は消さない**（記録）。

5. **出力（停止・承認後）**: 最終形は **Excel（.xlsx）**＝生きた台帳（並べ替え・絞り込み・行追加に向く）。
   - 2シート構成: 「課題管理表」（優先度で色分け・オートフィルタ・ヘッダー固定）＋「凡例」（区分/優先度/状態/連動ルール/出典）。
   - 出力先 `{案件名}/03_実行/01_課題管理表_vX.xlsx`。中身を固めるまでの壁打ちはマークダウンで回し、固まったらxlsx化する。

## ルール

- **課題とリスクを混ぜない**。リスク登録簿（計画§3）＝未来・不確実、課題管理表＝いま手を打つべきこと。連動はコピーでなくIDのリンク。
- **記号を流用しない**（`references/表と数値の見せ方.md`）。優先度＝高/中/低、状態＝オープン/対応中/完了。確度（●◐○）・採否（◎○△）はこの表で使わない。
- 担当は実プロジェクトでは人名にする。仮置き時は役割（PL等）で置き、確定時に人名へ。
- 文体は `references/文章作法.md`（6ルール）。一文一改行・ラベル見出し・🟡で未確定を残す。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#4・#39）。
- 多段階で勝手に進めない。✋承認なしに出力（xlsx）を確定しない。各ステップ開始時に現在地を明示する（例:「実行 03／課題管理表」）。

## 確定した運用ルール（2026-06-14）

- **「今週の焦点」は課題管理表に持たせない**。台帳は全件を持つのが役割で、「今週動かす数件に絞る」のは監視側の仕事＝状況報告（status-report）§要対応事項が担う。Excelの並べ替え・絞り込みでも代用できる。
- **「変更要求の芽」は当面 区分=判断待ち に寄せる**。4区分目は立てない。正式な変更管理は #8 変更の評価と実装の子Skill（未実装）で捌く。そこを作る段で「判断待ちに混ぜたもの」を変更管理表へ移す設計を検討する。
- 担当は実プロジェクトでは人名。仮置き時は役割（PL等）で置き、確定時に人名へ（上のルール再掲）。
