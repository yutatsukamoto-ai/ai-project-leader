# Design System Rubric

`design_system.md` を作るときのチェックリスト。

---

## 必須セクション

- 目的: 何を一貫させるためのデザインシステムか
- ブランド原則: 再利用できる原則を3〜5個
- ロゴ使用ルール、またはロゴなしルール
- 色の役割: primary / accent / text / line / background / surface
- 色の階層: base_background / main_dark / main_mid / main_light / accent_color / neutral
- タイポグラフィ: フォント方向、階層、ウェイト
- 余白・配置: マージン、整列、密度
- 情報設計: 1メッセージ、グルーピング、表、数値、タイムライン
- 見出し設計: navigation_title / message_headline / supporting_message の役割分担
- レイアウト戦略: slide_role / layout_archetype / emphasis_pattern の使い分け
- 論理構造: logic_pattern / logic_purpose / primary_read / structure_notes の使い分け
- 表・チャート方針
- 画像・アイコン・図版方針
- 避ける表現

---

## デフォルトスタイル

ユーザーが別の方向性を明示しない限り、`reference-informed consulting proposal style` にする。会社Webサイトやロゴをそのまま似せるのではなく、資料テーマに合う世界観へ翻訳する。外資系コンサル系資料からは形とロジック、Retty型資料からは色階層とパーツ設計を抽象化して使う。

- 広い余白
- 低装飾
- 生成り/オフホワイト背景（目安: `#F8F6EF`〜`#FBFAF5`）
- 資料テーマから翻訳した主色。深緑、青、オレンジ、墨色などから1色を選ぶ
- 主色の薄い面、線、チップ、強調棒を定義する
- アクセントは主色で表しきれない推奨、注意、結論、差分だけに使う
- 主色の濃淡で、見出し、図解の主部品、薄い背景面を分ける
- 比較対象、過去値、非推奨案はグレーや低彩度色へ退かせる
- アクセント色は推奨、注意、結論、差分だけに使う
- 細い罫線
- 画面上部または中央に大きな主張文
- 固定ヘッダーは短いカテゴリ名、本文見出しはページの結論や判断を示す
- 2〜3個までの構造化されたロジックブロック
- 課題整理、比較、工程、計画、判断基準では、issue tree、matrix、swimlane、decision tableなどの論理骨格を使う
- 課題 -> 詳細 -> 解決策の対応方向性 -> 今後の対応を示す構造を候補に入れる
- 入力、プロセス、アウトプット、判断軸つきのフロー型を候補に入れる
- Retty型の番号チップ、薄色セル、強調棒、グレー退避、薄い囲みをパーツ候補に入れる
- thin line icons、線画、意味のある抽象図、ラベル付き構造図形
- 下部の横長メッセージボックスや判断軸
- ロゴ由来またはStyle Markdown由来の色は、必要に応じて暗く/鈍く/少量に翻訳する
- スライド役割ごとに構図を選び、同じレイアウトを量産しない
- Noto Sans JP風の細い日本語サンセリフ
- 見出しはRegular、本文はLight、強調だけMedium
- タイトルスライドのロゴは小さく控えめ

避けるもの:

- 汎用カードUIの乱用
- 3カラムカードの乱用
- 左アクセントバーだらけの構成
- 角丸 + 薄影のAI資料っぽさ
- 明るすぎるブランドカラーの全面使用
- どの案件でも深緑/セージへ寄せる固定化
- どの案件でもRetty風オレンジへ寄せる固定化
- 参照資料の青い外資系テンプレートをそのまま真似ること
- 公式サイトやWebデザインの模倣
- ロゴ、キャラクター、サイトスクリーンショットの主役化
- 忙しいモックアップ画面
- 密度の高すぎるダッシュボード
- 装飾的な背景
- 意味のない大きな円、泡、抽象図形
- 重いグラデーション
- 意味のない写真素材
- 多色すぎるカード
- 要素ごとに別々の色を割り当てる多色パレット
- アクセント色の多用
- 真っ白背景のデフォルト使用
- 太字だらけの文字

## 抽象度チェック

NG:

- 現在のデッキのページ別計画になっている
- スライドタイトルだけを並べている
- 無関係なテンプレートをコピーしている
- 全スライドが同じ大見出し + 中央図解 + 下部帯になっている
- `layout_archetype` の使い分け理由がない
- DX/AIコンサル資料なのに、論理構造がなくポスターやLPのようになっている
- 課題と解決策、工程と成果物、判断軸と推奨案の関係が見えない
- 見出しが「について」「重要なポイント」「今後について」のように、何を言うページか分からない
- 項目数、章番号、補助ラベルを根拠なく巨大化している
- すべての表/カード/数値を固定部品として扱っている

OK:

- 同じブランドアンカーで別デッキにも使える
- 色や余白の理由が説明されている
- 主色の濃淡、非強調色、アクセントの役割が説明されている
- 主色が資料テーマから選ばれており、固定色の惰性になっていない
- 参照資料から取り入れるものと、真似ないものが分かれている
- slide_role / layout_archetype / emphasis_pattern の選び方が説明されている
- navigation_title / message_headline / supporting_message の役割が分かれている
- visual_focus と emphasis_reason が説明されている
- タイポグラフィのウェイトが定義されている
- どこを後処理で揃えるかが明確
- ブランドルールと今回のデッキ固有情報が分かれている

## ロゴなしチェック

ロゴなしの場合:

- ロゴを作らない
- ロゴ使用ルールではなく、no-logoルールを書く
- 右上を空けるか、ユーザー指定のデッキマークだけにする
- 通常スライドの小タイトルと下線だけを固定要素にする

---

## Style Markdownチェック

Style Markdownがある場合:

- その内容を一次入力として扱っているか
- ロゴ/公式サイトとの矛盾があれば記録しているか
- 避けたい表現がnegative constraintsへ落ちているか
- 古い/抽象的な内容を事実扱いしていないか

---

## セッションで守ること

- タイトルとロゴ位置はImage2任せにしない。
- コードで直してよいのは通常スライドの小タイトル、下線、ロゴだけ。
- 本文エリア、表、図解、本文テキストをコードで直さない。
- コンタクトシートはデフォルトで作らない。
- 方向性確認用のサムネイル/ボードを作る場合も、完成物とは分ける。
- `layout_archetype` を明示し、構図の同型反復を避ける。
- 円、泡、線、チャート風要素は、意味やラベルがある場合だけ使う。
- 既存テンプレートを勝手に見ない。
- 原稿の情報を落とさない。多い場合は分割する。
