# 監視・コントロールフェーズ（親オーケストレーター）

実行で出た事実 → 巻き上げて状況報告にし、意思決定につなげるところまで。中身の知識は持たない。順番と停止だけを守る。
監視の本質は「見張る」ことより「**周回して舵を切る**」——実行と並走し、定期的に巻き上げ、ゲートで判断材料を出す。一度きりでなくサイクルで回る。

> **★ 現状: 子Skillは status-report（週次状況報告）・change-log（変更管理表）が実装済み ★**
> スコープ/スケジュール/コストのコントロール・品質コントロール等の子Skillはまだ無い。本書は監視・コントロールフェーズを構造として閉じるための型。
> 中身は実案件で「伴走→写し取り」方式（[[ai-project-leader-design-principles]]）で詰める。接続規約は `00_プロジェクト管理/接続規約_前段→本体_v0.2.md` の様式に準じる。

## 実行手順

1. **入力を読む**: `{案件名}/03_実行/引き継ぎ書_実行→監視.md`（あれば）。無ければ実行の生きた台帳を直接入力にしてよい＝`03_実行/01_課題管理表_vX`・`02_計画/01_統合計画書_vX.md`（§3リスク登録簿）・`project-context.md`。生データは `project-context.md` をポインタ経由で参照する。
2. **現在地を明示**: 「フェーズ＝監視・コントロール／本質は"巻き上げて舵を切る"／これは第N週」を提示。報告対象の期間と、いま居るローリングウェーブの段階を確認する。
3. **01 状況報告（#7 パフォーマンスの監視・コントロール）** を status-report で実行 → ✋提出版（docx）確定前に「数字・期日がぼけていないか」「下流（課題・リスク）と矛盾しないか」「要対応が先頭か」を確認（**必須停止**）。
4. **02 変更管理表（#8 変更の評価と実装）** を change-log で実行（ベースライン変更が出たとき）→ ✋計画へ反映する承認前に「ベースライン変更だけか（境界）」「影響が4面で評価されているか」を確認（**必須停止**）。issue-logの判断待ちのうちベースラインを変えるものをここへ移す（指名・人選は残す）。
   - （将来の子Skill・未実装）スコープ/スケジュール/コスト/品質のコントロールも要る案件で足す。
5. **周回と移行**: 週次・節目ごとに 3〜4 を回す。ゲートを通過し当該フェーズ/プロジェクトの終わりが見えたら、確定事実を `project-context.md` へ書き戻し、終結フェーズへ引き継ぐ。課題管理表・リスク登録簿・状況報告はそのまま終結（project-closeout）の入力になる。
   - 節目では **テーラリング判断ログの「保留」を逆転条件と突き合わせ**、条件を満たした項目はbacklog（成果物マップのSkill化候補）へ積む／着手を提案する（規約: `40_Stock/横断ガイドライン/テーラリング判断ログ.md`・D-04補強①）。

## ルール

- 各ステップの入力は **引き継ぎ書／課題管理表／リスク登録簿／統合計画書／前ステップの成果物／project-context.md のみ**。
- 出力先と命名: `{案件名}/04_監視コントロール/01_週次状況報告_vX.docx` 形式（フェーズ別サブフォルダ＋フェーズ内ローカル採番。番号空間はフェーズごとに独立）。
- 停止ポイントを勝手にスキップしない。承認・確認なしに提出版を確定しない。
- **状況報告・変更管理で参照するリスク/課題の状態は、計画§3リスク登録簿・課題管理表の最新に揃える**（課題化済リスクは「対応へ移管」で表示。下流と矛盾させない）（実証メモ§2-B）。
- **完了後の `project-context.md` への書き戻しを飛ばさない**（確定事実・🟡の解消を反映。書き戻し先は案件ルート直下の1箇所）（実証メモ§2-C）。
- 各ステップ開始時に現在地（第N週・段階）を明示する（例:「監視・コントロール 01／状況報告・第6週」）。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う。
