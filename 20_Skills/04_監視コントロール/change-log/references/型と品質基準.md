# 変更管理表 型と品質基準

出典: 山藤商事 変更管理表v0.1の伴走（2026-06-14）で言語化した手順・判断を写し取ったもの。
位置づけ: SKILL.md本体（薄い）の品質バー。該当ステップでのみ読む（Progressive Disclosure）。

---

## この成果物の正体

計画のベースラインを変える要求だけを、影響評価と承認を通してから計画へ反映する**関所**（#8）。
生きた台帳である点はissue-log（課題管理表）と同じだが、扱うものが違う——issue-logは「いま手を打つべきこと」、change-logは「決めた約束（ベースライン）を変えること」。
品質バー＝「ベースライン変更だけが載っているか（境界）」＋「影響が4面で評価されているか」＋「承認を経ずに計画へ反映していないか」＋「引き金が計画済/予期外で区別されているか」。

## 境界（何を載せ、何を載せないか）— 最重要

混ぜると関所が形骸化する。判定は「計画のベースラインを変えるか」の一点。

| 載せる（change-log） | 載せない（issue-logに残す） |
|---|---|
| 要求・スコープ・優先順位の変更 | 指名・人選など純粋な意思決定待ち（I-01責任者指名・I-05運用担当） |
| スケジュール・マイルストーンの変更 | 計画どおり進めるための作業課題 |
| 予算・コストベースラインの変更 | リスク対応アクション（#39。リスク登録簿→issue-log） |
| 計画方針（進め方）の変更 | ベースラインを変えない運用上の調整 |

- 連動は **ID（C-xx）のリンク**。引き金の課題（I-xx）・リスク（R-xx）・計画の節を「関連」で指す。中身はchange-logにだけ書き、二重管理しない。
- リスク↔課題（issue-log）と同じ思想を、変更にも一貫適用する。

## 列構成（1変更1行）

| 列 | 中身 | 品質バー |
|---|---|---|
| ID | C-01〜 | 連番。issue-log(I-xx)・リスク(R-xx)と名前空間を分ける。 |
| 起票日 | YYYY-MM-DD | 具体で。 |
| 対象ベースライン | 要求/スコープ/優先順位/スケジュール/予算/計画方針 | どのベースラインを変えるか1つ明示。 |
| 変更内容 | 何を→何に | before→afterで言い切る。 |
| トリガー・理由 | なぜ変えるか | 頭に **[計画済]／[予期外]**。計画済はR-xx等に対応物があるのが普通。 |
| 影響評価 | スコープ・日程・費用・リスクの4面を**1セルに要約** | 4面を必ず触れる。具体で・軽く。別列に割らない。 |
| 起案する反映 | 承認されたら計画へどう反映するか | 反映先（統合計画書のどの節・vX+1）まで書く。 |
| 意思決定者 | 誰が承認するか | ゲート権限者（多くはスポンサー）。 |
| 期限 | いつまでに判断 | ゲート・キックオフ等の相対表現可。 |
| 状態 | 起票/評価中/承認待ち/承認/却下/反映済 | 言葉。関所として進める。 |
| 関連 | R-xx・I-xx・計画§ | 双方向に辿れる状態。 |

## トリガーの区別（計画済／予期外）

- **[計画済]**＝計画に仕込んだ変更トリガーが引かれた（逆転条件の発火・ゲートでの見直し）。予見できた変更。
- **[予期外]**＝計画になかった外部要因（取引先要請・法改正・予算カット）。発生時に評価から始める。
- 区別を残すと、終結の教訓「どの変更が予見できたか」や次案件の計画精度に効く。

## 状態の流れ（関所）

思いつき反映を防ぐため、状態で進める。

1. 起票（トリガーと対象を明記）→ 2. 評価中（4面の影響を要約）→ 3. 承認待ち → 4. 承認/却下（条件付き承認もここ）→ 5. 反映済（計画書vX+1へ反映＋project-contextへ書き戻し＋反映先を関連に記録）。
承認された変更に伴う新しい作業は、change-logでなく **issue-logに起票**する（変更管理表に作業を書かない）。

## 記号・数値の作法

- **状態は言葉**（起票/評価中/承認待ち/承認/却下/反映済）。確度（●◐○）・採否（◎○△）を流用しない（`references/表と数値の見せ方.md` A節）。
- 状態セルは色で補助してよい（承認待ち＝黄系／承認＝緑系／却下＝灰系）。色は言葉の補助。
- 数字をぼかさない（影響の日程・費用・件数は具体で）。

## 出力（xlsx）

- 生きた台帳なので最終形は .xlsx（issue-logと同じ家の流儀）。状態で色分け・凡例シート。
- 運用: マスターはマークダウン、xlsx化は人の確認後。出力先 `{案件名}/04_監視コントロール/02_変更管理表_vX.xlsx`。

## やってはいけない

- ベースラインを変えないもの（指名・人選・作業）を変更管理表に入れる（境界の崩壊）。
- issue-log・リスク登録簿と内容を二重に持つ（リンクで持つ）。
- 承認を経ずに計画へ反映する（関所の意味が消える）。
- 影響評価を4面に触れず「影響小」とだけ書く（スコープ・日程・費用・リスクを具体で）。
- 状態に確度記号（●◐○）・採否記号（◎○△）を流用する。
- トリガーの計画済/予期外を区別しない（振り返りの材料が消える）。
