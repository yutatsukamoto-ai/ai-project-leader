# プロジェクトチャーター作成（子Skill）

立ち上げフェーズの基幹成果物。承認を得る書類ではなく「迷ったとき立ち戻る一枚」を作る。
中身の品質基準は `references/` を見る（Progressive Disclosure。本体は薄く保つ）。

## 手順

1. **入力を読む**: `{案件名}/00_前段/引き継ぎ書_前段→立ち上げ.md`（冒頭の✋人による確認が済か確認。未なら止める）。引き継ぎ書②のテーラリングで、今回のチャーターの濃淡を決める
   - `project-context.md`（案件ルート直下）があれば併せて読む。**無ければ引き継ぎ書を一次入力に進め、承認後の書き戻し（手順6）でこのファイルを新規作成する**（前段を終えた直後の案件はまだ存在しないのが普通。止めない）
   - 旧フラット構成の案件（00_前段サブフォルダが無く、ファイルが案件直下にある）では、引き継ぎ書を `00_前段/` に新設してから進める。既存ファイルの構造移行はD-01と合わせて別途判断（このSkillでは移行しない）
2. **11項目を埋める**: `references/型と品質基準.md` を読み、11項目を埋める。品質バー＝「チャーター必須項目 ∩ PMBOK#1で要る項目 ＋ 前提情報で優先すべきこと」が揃っているか
3. **推測には根拠を明示**: 推測を含む項目は出典（引き継ぎ書/0-4/context/PMBOK等）を併記する。未確認は🟡で残し、立ち上げ〜計画で詰める申し送りにする
   - 目標・成功基準（§3-4）は**SMARTをレンズに**点検する（S具体/M計量/A達成可能/R関連/T期限）。確定できない次元は🟡で§11学習・申し送りへ。数値・期限はでっち上げない（12ステップ#1が「SMARTに語れる状態」を要求）
4. **§11を背骨として必ず埋める**: 「固定する事項／学習に委ねる事項」はチャーター固有の価値。ここを空にしない
5. **✋承認前の照合（必須停止）**: `references/12ステップ要点.md` の定義4項目（最終目標/対象範囲/利害関係者/阻害要因）＋「迷ったとき立ち戻れる一枚か」を照合し、結果を添えて承認を求める
6. **承認後**: 確定事実を `project-context.md` へ書き戻す。出力先 `{案件名}/01_立ち上げ/01_プロジェクトチャーター_vX.md`

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う
- プロセス参照は `references/PMBOK第8版_40プロセス一覧.md` の #番号で行う
- 0-4計画提案書との内容重複は許容する（チャーターは別目的の正式成果物＝立ち戻る判断基準、0-4＝承認を得る提案）。差別化より必須項目が揃うことを優先
- 多段階で勝手に進めない。✋承認なしに次工程へ渡さない
- 各ステップ開始時に現在地を明示する（例:「立ち上げ 01／チャーター作成」）

正典は `40_Stock/横断ガイドライン/成果物Skill共通ひな形.md`。
