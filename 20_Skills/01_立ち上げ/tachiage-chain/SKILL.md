# 立ち上げフェーズ（親オーケストレーター）

前段の引き継ぎ書 → チャーター → ステークホルダー登録簿 の承認まで。中身の知識は持たない。順番と停止だけを守る。
立ち上げの本質は「始める」ではなく「引き受ける」——価値・判断・責任・前提条件を言語化する。

> 子Skill: `project-charter`（#1）・`stakeholder-register`（#23）＝いずれも実装済（伴走→写し取り）。接続規約は `00_プロジェクト管理/接続規約_前段→本体_v0.1.md`。

## 実行手順

1. **入力を読む**: `{案件名}/00_前段/引き継ぎ書_前段→立ち上げ.md`。冒頭メタの「✋人による確認: ☐済」が立っているか確認。**未確認なら止める**（凍結スナップショットの未確認版を本体に流さない）。生データ（クライアント・目的・制約）は案件ルート直下の `project-context.md` をポインタ経由で参照する
2. **現在地を明示**: 「フェーズ＝立ち上げ／本質は"引き受ける"」を提示。引き継ぎ書②のテーラリング（採用/省略の#プロセス）に従い、立ち上げで作る成果物を確定する（おまかせにしない）
3. **01 チャーター作成（#1 プロジェクトまたはフェーズの立ち上げ）** を実行 → ✋承認前に、12ステップ「定義4項目」＋「迷ったとき立ち戻れる一枚になっているか」を照合（**必須停止**）
4. **02 ステークホルダー登録簿（#23 ステークホルダーの特定）** を `stakeholder-register` で実行。チャーター§7のハイレベル一覧を**深掘り**（二重化しない）→ ✋承認前に検査ゲートを回す
5. **完了後**: 確定事実を `project-context.md` へ書き戻し、次のフェーズ境界の引き継ぎ書（立ち上げ→計画）を起案 → 次は計画フェーズであることを案内

## ルール

- 各ステップの入力は **引き継ぎ書／前ステップの成果物／project-context.md のみ**
- 出力先と命名: `{案件名}/01_立ち上げ/01_プロジェクトチャーター_vX.md` 形式（フェーズ別サブフォルダ＋フェーズ内ローカル採番。番号空間はフェーズごとに独立）
- 停止ポイントを勝手にスキップしない。承認・確認なしに次へ進まない
- 各ステップ開始時に現在地を明示する（例:「立ち上げ 01／全2工程」）
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う
