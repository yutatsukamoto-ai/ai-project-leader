# ステークホルダー登録簿 作成（子Skill）

立ち上げフェーズの基幹成果物（#23 ステークホルダーの特定）。
チャーター§7のハイレベルな関係者一覧を、誰を・現在どの関与度で・どこへ動かし・どう巻き込むかまで深掘りした生きた台帳にする。
中身の品質基準・登録簿の項目・判断の分かれ目は `references/型と品質基準.md` を見る（Progressive Disclosure。本体は薄く保つ）。

## 手順

1. **入力を読む**: `{案件名}/01_立ち上げ/01_プロジェクトチャーター_vX.md`（§7主要ステークホルダー・§8意思決定構造・§9リスク）・`project-context.md`・あれば案件設定/前段成果物。
   - チャーターの✋承認が済か確認する。未なら止める。
   - **チャーター§7は"立ち戻る一枚"として残す**。登録簿はそれを深掘りした別物＝二重化しない（課題管理表と状況報告の関係と同じ）。

2. **登録簿を埋める**: `references/型と品質基準.md` の項目・品質バーに沿う。
   - 関係者ごとに 区分／影響力／関心／重点度／**関与度（現在→目標）**／巻き込み戦略／状態 を埋める。
   - 【差し込みカード参照枠】`40_Stock/カード/レシピ_作成/ステークホルダー登録簿の作り方.md` があれば引く（無ければ素通り）。

3. **推測には根拠を明示**: 影響力・関心・関与度の評価は根拠を添える。未確定（指名未・人選未）は🟡で残し、確定タイミング（どのゲートで決めるか）を書く。
   - 記号は `references/表と数値の見せ方.md` の正典どおり（関与度・状態は言葉。確度●◐○／採否◎○△を流用しない）。

4. **巻き込み戦略を組む（停止・壁打ち）**: 関与度の現在→目標の**ギャップ**から戦略を引く。
   - 影響力×関心で**重点管理を絞る**（広げすぎると誰も動かせない）。
   - 最大の抵抗者がキーパーソンを兼ねうるときは**当事者化を第一候補**に（ただし腹落ち前に名ばかり任命しない＝効果の理屈）。
   - **可逆／不可逆の線引きを必ず通す**: 巻き込みの一手が可逆なら攻める（読み違いは直す）。不可逆・致命（情報漏洩・契約・安全・信頼の致命破壊）は賭けず関所（✋✋）に置く。

5. **検査ゲート（出力前・必須停止）**: 共通7項目＋本Skill固有を回す（共通ひな形§検査ゲート）。
   - 共通: ①チャーター✋確認済か ②必須項目・品質バー（未確定は🟡） ③記号は正典どおりか ④チャーター§7と二重化でなく深掘り・接続になっているか ⑤出力先パス・版番号 ⑥未確認の明示（確認しなかった範囲＋理由を一覧） ⑦サボり回収ループ（スキップ理由を再検証、サボりはやり直し）。
   - 固有: 意思決定者・キーパーソン・承認者に漏れがないか／各人に関与度（現在→目標）と戦略があるか／不可逆リスクに関わる関与に関所があるか。
   - 結果を ✅／🟡／❌ で出し、最終判断は本人に返す。✋承認を得る。

6. **承認後**: 確定事実（重点管理・任命の確定タイミング等）を `project-context.md` へ書き戻す。
   - 出力先 `{案件名}/01_立ち上げ/02_ステークホルダー登録簿_vX`（最終形は生きた台帳＝md→**xlsx**。issue-logと同じ家の流儀）。

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う。
- 記号は `references/表と数値の見せ方.md` の正典（関与度・重点度・状態は言葉。予約記号を流用しない）。
- プロセス参照は正典 `10_参考資料/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#23）。
- チャーター§7との重複は許容するが**二重化はしない**＝チャーターは一枚もの、登録簿は深掘り台帳。役割を分け、リンクで繋ぐ。
- 多段階で勝手に進めない。✋承認なしに次工程へ渡さない。
- 各ステップ開始時に現在地を明示する（例:「立ち上げ 02／ステークホルダー登録簿」）。
