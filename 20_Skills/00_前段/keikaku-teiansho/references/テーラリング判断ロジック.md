# テーラリング判断ロジック（計画提案書の心臓部）

v1.0（2026-06-13）。出典: 山藤商事（小規模・ユーザー主導DX）／テクノブリッジ（中堅SIer・統制重・受託）の2実走から写し取り。
基準資料: references/PMBOK第8版_40プロセス一覧.md（#番号）／references/PMBOK第8版_6原則一覧.md。
使う場面: 手順2で提案書を書く前にこのロジックで「成果物の取捨選択」を導き、第7セクションのテーラリング表と社内用付録の選択記録を同じ評価から埋める。

**記号と出力の使い分け**: 本文書では思考の略記として◎○△×を使う（内部参照）。ただし**提案書の本体（クライアント向け出力）では言葉で書く**（「しっかり作る」「簡易版」「今回なし＋理由」）。記号だけで状態を表さない——横断ガイドライン「状態は言葉」ルール準拠。

---

## 0. 大原則（動かさない）

- **全項目◎を目指さない**。意識的に「活かす/軽く/やらない」を選び、理由を残す。
- 判断基準は常に **「お客さんに価値が届くか」**。クライアントが求めない領域への深掘りは期待値ズレ。
- **やらない判断も記録すれば「漏れ」ではなく「選択」**（PMBOK第8版は40プロセスを非規範的＝illustrativeと位置づける）。
- 前段では**仮説**。確定は立ち上げ〜計画の合意時点。すべて**確度＋逆転条件**で書く。

## 1. 入力 — 6つの判断ドライバー（案件の「何を見るか」）

テーラリングは勘ではなく、案件を特徴づける6軸の評価から導く。すべて0-1／0-2から読み取れる。各ドライバーを **軽・中・重** で評価する。

| # | ドライバー | 読み取り元 | 「重」の例 |
|---|---|---|---|
| D1 | 規模・複雑さ | 0-1概要（人数・拠点・予算）／概算 | 数百名規模・複数部門・高額 |
| D2 | 説明責任の重さ（金額で説明する義務） | 0-1業種・契約形態・意思決定者 | 受託・契約金額の説明義務・対役員/対監査 |
| D3 | 統制・コンプラの重さ | 0-1制約の「絶対条件」 | 機密・規制（金融/官公庁）・セキュリティが死守 |
| D4 | スコープの安定性 | 0-2課題仮説の確度 | 確度低＝何をやるかが流動的 |
| D5 | スピード／リードタイム要求 | 0-1期待・役員評価軸 | 「立ち上げが遅い」が主訴 |
| D6 | 定着リスク（組織成熟度） | 0-1最重要発言／0-2の最大リスク | 過去のツール形骸化・現場の不信・キーパーソン依存 |

## 2. マッピング — ドライバー → 7ドメイン/6原則の強度

各ドライバーが「重」のとき、どこに重みが寄り、どこを軽くしてよいか。

| ドライバーが重い | ◎に寄る（厚くする） | 軽くしてよい |
|---|---|---|
| D1 規模大 | ファイナンス（#19-22）・スケジュール（#16-18）・資源（#30-34）・ガバナンス | — |
| D2 説明責任重 | ファイナンス（特に#22 財務の監視＝EVM）・ガバナンス（#7,8）・原則「説明責任あるリーダーシップ」 | — |
| D3 統制重 | ガバナンス（#6 品質保証）・リスク（#35-40）・原則「品質の組み込み」 | サステナビリティ（無関係なら△） |
| D4 スコープ流動 | スコープ（#11要求の引き出し・#14妥当性確認）・リスク・変更管理（#8）。反復前提 | 詳細スコープ構造（#13 WBS）・固定スケジュール（#17）を作り込まない |
| D5 スピード重 | 立ち上げを薄く速く・実行を厚く。スコープを絞る（#12 やらないこと） | 重い計画ドキュメント一式 |
| D6 定着リスク重 | ステークホルダー（#23特定・#24,26）・原則「文化のエンパワーメント」・リスク | — |

**常に◎（固定）**: 原則「価値への集中」（#2）／スコープの「やらないこと」定義（#12）。

## 3. 40プロセスへの落とし込み（◎○△× の付け方）

### ベースライン（小〜中規模・前段→立ち上げ段階で最低限触れる6つ）

案件特性に関わらず、まずこの6つを◎で置く（提案書は「開始の意思決定」のための文書だから）。

- #1 プロジェクト/フェーズの立ち上げ（チャーターの素地）◎
- #11 要求事項の引き出しと分析 ◎（曖昧な期待の要件化＝前段の勝負どころ）
- #12 スコープの定義（やらないことの明文化）◎
- #23 ステークホルダーの特定 ◎
- #36 リスクの特定 ◎
- #2 計画の統合と整合 ○

### 調整ルール（第2章を番号に落とす）

- D2/D1重 → #19-22ファイナンスを△→○、#22は説明責任が重ければ◎（EVM所在）
- D3重 → #6品質保証・#35-40リスクを○→◎
- D4流動 → #13スコープ構造・#17スケジュール作成は△（反復で育てる）。#8変更管理を意識
- D5重 → 計画系を薄く、#4実行を早く回す
- D6重 → #24,26ステークホルダーエンゲージメントを◎（定着の約束に直結）
- 規模小・前段段階 → #3ソーシング戦略・#20,21見積/予算・#30資源計画は基本△

## 4. 進捗管理手法の選択（テーラリングの一適用例）

進捗管理手法の選択は「#22財務の監視」「#18スケジュールの監視」をどの粒度でやるかのテーラリングそのもの。

- 目安（重→軽）: EVM（大規模・契約型・説明責任重＝D1D2重）／簡易EVM（中規模受託）／MTC（小〜中・節目報告で十分）／バーンアップ（スコープ流動＝D4重）／スループット（運用・保守）。
- **迷ったらMTCで開始**。説明責任が増したらEVM側、変動が増したらバーンアップ側へ。
- 書き方は確度＋逆転条件。例:「MTCで開始（確度:中）。月次コスト報告要求が出たら簡易EVMへ、スプリント運用に移行したらバーンアップへ」。

## 6. 承認ゲートでの効用

ドライバー評価を先にやることで、承認ゲートで「なぜこの形か」を問われたとき **「この案件はD3統制とD6定着が重いので、ガバナンスとリスクに寄せました」と即答できる**。
