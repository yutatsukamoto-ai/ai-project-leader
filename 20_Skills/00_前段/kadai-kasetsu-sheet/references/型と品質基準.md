# 課題仮説シート 型と品質基準

出典: 模擬案件・山藤商事の試運転（2026-06-11〜12）で実証した型。修正ログ・05振り返りより写し取り。

## 文書の型（セクション構成）

1. **Why — なぜやるのか**: 0-1の期待（W番号）を引き継ぎ、目的として構造化。確度と根拠発言を表で付ける。「手段（AI等）は目的ではない」の整理をここで固める
2. **What — 何が課題か**: 課題仮説をH1〜で採番。各仮説に確度（●高=発言で裏付け ◐中=状況証拠 ○低=一般論からの推測）と**検証方法**をセットで付ける
3. **How — どう進めるか**: 進め方の仮説+理由（例: 現状可視化を先行する理由、段階型にする理由、キーパーソンを巻き込む理由）
4. **優先順位の仮説+逆転条件**: どのWを先行するかの案と、**「何が分かったら順位が変わるか」を必ず明記**

## 判断基準

- **仮説と断定の区別**: 0-1の「推測」由来はすべて仮説。断定調で書かない
- **最大リスクの二重位置づけ**: 最大のリスク要因（例: 過去のシステム導入失敗）は「課題」と「リスク」の両方に位置づける。後の計画フェーズのリスク登録の種になる
- **確度と検証方法はセット**: 検証方法のない仮説は書かない（誰に何を聞けば確かめられるかまで)
- **制約の出どころを引き継ぐ**: 0-1で記録した①クライアント指定/②未定/③自社制約をHowの設計に反映する

## 合格チェック

- [ ] WhyのW番号が0-1と一致しているか
- [ ] 全仮説に確度+検証方法が付いているか
- [ ] 最大リスクが課題とリスクの両方に位置づけられているか
- [ ] Howに理由が付いているか（結論だけ書かない）
- [ ] 優先順位に逆転条件が明記されているか
- [ ] 断定調の文が混ざっていないか
