# 前段チェーン（親オーケストレーター）

新規案件の説明 → 計画提案書の承認まで。中身の知識は持たない。順番と停止だけを守る。

## 実行手順

1. 作業フォルダを作る: `30_Flow/{カテゴリ}/{案件種別}_案件名/`（カテゴリ＝`実案件/` `模擬案件/` `テスト/`、案件＝ライフ全体の単位）。案件ルート直下に `templates/project-context.md` を複製して `project-context.md` を置き冒頭情報を埋める。サブフォルダ: `_素材/`（受領資料の冷蔵庫）・`00_前段/`〜`05_終結/` を作成。前段成果物は `00_前段/` に置く（フォルダ規約: `30_Flow/README.md` と `00_プロジェクト管理/接続規約_前段→本体_v0.2.md`）
2. **0-1 anken-rikai-summary** を実行。本文作成前に入力品質3行判定（この入力で言えること／不足していること／追加確認したいこと）を出す → ✋理解確認（必須停止）
3. **0-2 kadai-kasetsu-sheet** を実行 → 軽い確認
4. **0-3 research-memo** を実行（スキップ判断あり）→ 軽い確認
5. **0-4 keikaku-teiansho** を実行 → ✋✋承認ゲート（必須停止）
6. 承認後: **引き継ぎ書を起案** → 人確認 → 案内（下記）
   - a. `00_前段/引き継ぎ書_前段→立ち上げ.md` を起案する（既存案件の引き継ぎ書を参考にする。例: `30_Flow/模擬案件/模擬案件_テクノブリッジ/00_前段/引き継ぎ書_前段→立ち上げ.md`）。0-4承認時点の**凍結スナップショット**。0-4を複製せず、承認の事実／チャーターへの再フレーム／運ぶ仮説・申し送りだけを書く。運ぶ仮説はAIが候補と理由を提示し人が選ぶ
   - b. **✋人による確認**（必須停止）。引き継ぎ書冒頭の「✋人による確認: ☐済」を立ててもらう。済むまで本体へ渡さない
   - c. 終了を宣言し、次は立ち上げフェーズ（**tachiage-chain**）であることを案内

## ルール

- 各ステップの入力は**前ステップの正典成果物（フェーズフォルダ）とproject-context.mdのみ**。`outputs/` は中間生成物に限る
- 各子Skillは確認/承認後、確定事実と決定を `project-context.md` に書き戻す。親はステップ間でcontextが更新されているかを確認する（決定の記録が空のまま進まない）
- 停止ポイントを勝手にスキップしない。承認・確認なしに次へ進まない
- 各ステップ開始時に現在地を明示する（例:「前段チェーン 0-2／全4工程」）
- `project-context.md` は案件ルート直下（`30_Flow/{カテゴリ}/{案件種別}_案件名/project-context.md`）＝全工程を通す生きた本体
- 前段成果物の命名: `00_前段/01_案件理解サマリー_vX.md` 形式（フェーズ別サブフォルダ＋番号+成果物名+版。番号空間はフェーズごとに独立）
- 引き継ぎ書は凍結スナップショット。承認後の新事実は引き継ぎ書でなく `project-context.md` に書き戻す

## 完了時の案内

前段完了時は、最後に必ず次を提示する。

- 次に使うchain: `tachiage-chain`
- 入力: `00_前段/引き継ぎ書_前段→立ち上げ.md` と `project-context.md`
- 停止条件: 引き継ぎ書の「✋人による確認: 済」が無ければ立ち上げへ進まない

## chain-trace.json

- 前段のみの段階では `chain-trace.json` は作らない。立ち上げ開始時に `tachiage-chain` が初期化する。
- 既に案件ルートに `chain-trace.json` がある場合は、承認ゲートや再実行など本チェーンのイベントを追記する（規約: `40_Stock/横断ガイドライン/チェーントレース規約.md`）。
