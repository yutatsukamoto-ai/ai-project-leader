# アジェンダ・議事録 標準規格 v0.1（ドラフト）

- 出典: 『両利きのプロジェクトマネジメント』プロマネツールキット（スライド19〜22）
- 位置づけ: 本プロジェクトの「入力の標準形式」（骨子・未決事項の1つに対する答え）。前段チェーン（ヒアリング・議事録 → 課題と期待の構造化）の起点となる規格
- 運用原則: **マスターはMarkdown、クライアント提出時にWord（.docx）へ変換**。docx変換は人間確認後に行う

---

## 1. ファイル運用ルール

- **1会議 = 1ファイル**。アジェンダと議事録は同じファイルに同居させる
  - 理由: 「終了条件」に対して「実際どうなったか（決定事項・議論メモ）」を構造的に突き合わせられる＝漏れ検知装置になる
- ファイル名: `YYYYMMDD_会議名.md`（例: `20260618_○○社_初回ヒアリング.md`）
- ライフサイクル:
  1. **会議前**: アジェンダ部（ゴール・議題・終了条件）を作成
  2. **会議中**: 議論メモ部に構造化箇条書きで記録。決定・ToDoには★
  3. **会議後**: 決定事項・Next Actionを確定 → 共有用サマリをチャットに投稿（清書はしない）

## 3. 会議後の共有フォーマット（チャット投稿用）

> 出典: 議事録フォーマット（会議後に、決定事項とToDoを共有するフォーマット）

```
■決定事項
・●●会議は●月●日の開催とする

■ToDo
▼報告書の内容について検討する
・●●資料のp10を修正する（@AAA、締切: 12/10）
・そのために、@BBBさんとクイックに5分MTGする（@AAA、@BBB）

▼クライアントとの懇親会の開催内容について議論する
・お店を予約して、関係者のカレンダーに登録する（@CCC）

■次回MTGアジェンダ
・アジェンダURL（URLでなくとも、テキスト記載でも良い）

■関連資料
・議事録 / スケジュール / 報告書資料（リンク付与）
```

構造ルール: ToDoは**アジェンダ（テーマ）ごとに▼で区切る**／ToDo単位で**担当にメンション＋締切**／該当資料には**リンクを付与**。

## 4. アジェンダ設計の思考フレーム（詳細版の観点）

> 出典: アジェンダフォーマット Googleスプレッドシート版（詳細版）

詳細版はテンプレとして二重管理せず、**アジェンダ設計時のチェック観点**として使う。各議題は以下の問いを通過していること:

| 観点 | 問い |
|------|------|
| 番号・優先順位 | どの順で扱うか。優先度（◎/○）は付けたか |
| 完了チェック | 終わった議題にチェックを付けて進行管理できるか |
| 何をするのか (What to Do) | この時間で何をするか |
| 目的 (Why) | なぜこの議題を扱うのか |
| どのように進めるか (How) | 進め方の手順（例: 1)共有 2)疑問点記入 3)解消、残りはNext Action明確化） |
| 誰に (Whom) | 誰に向けた議題か |
| 種別 | 発散か収束か |
| 担当 | 議題のオーナー |
| 資料リンク | 議題で使う資料への参照 |
| 完了の状態 | 終了条件（＝簡易版の「終了条件」欄に書く中身） |
| 想定アウトプット | 議題から生まれる成果物（例: 質問・疑問リスト） |
| 開始時間・所要時間 | タイムボックス（MTGゴールから逆算） |

※ キックオフ・ステアリングコミッティ等の重要会議では、詳細版（上記全項目の表）をフル出力するオプションを使う。MTGヘッダーには日付・開始/終了予定時間・参加者（会社ごと）・関連資料リンク・**MTGのゴール（実現したい状態）**を必ず置く。

## 5. 品質基準（Skill化するときの判断基準）

### 出典由来の注意点

- アジェンダは「話す話題」ではなく**「終了条件（完了とみなす状態）」で定義**する
- 議論メモは**議題番号ごとに区切り**、箇条書き機能で構造化する
- 決定事項やToDoは、議事録を取っているタイミングで**★マーク**を付けて、あとから確認しやすくする
- ToDoには**担当メンションと締切**を必ず付ける
- **次回MTGアジェンダを仮でも（最悪、枠だけでも）用意**しておく——関係者の目線が先を向き、行動が早くなる
- **詳細な議論メモの清書は不要**。会議後に共有するのは決定事項とToDoだけ
- 関連資料には**リンクを付与**し、関係者がアクセスしやすい状態にする

### 本プロジェクトの追加ルール

- **【要確認】ルール**: AIは文字起こし・メモから決定事項を推測で補完しない。断定できないものは`【要確認】`タグを付けて人間に返す（おまかせ不採用の思想）
- **docx出力は人間確認後**: Markdown版を人間がレビューしてからWord変換する
- 終了条件は「全部解決」を求めない書き方も許容する（例:「質問が出し切れていてアクションが明確なこと（解消まではこの場で求めない）」）

## 6. チェーン上の位置づけ

```
アジェンダ設計Skill →（会議実施）→ 議事録Skill → 課題と期待の構造化 → チャーター素材
                                      │
                                      └→ docx変換（クライアント共有用）
```

- フェーズ×ドメイン: 全フェーズ横断の運営系（主にステークホルダー／ガバナンス・ドメイン）
- この2つのSkillは汎用性が高く、立ち上げ〜終結まで使い回せる
