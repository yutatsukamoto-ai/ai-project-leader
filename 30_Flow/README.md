# 30_Flow — 作業中ドラフト・試運転案件

作業中（未完成）の成果物を置く場所。完成品は `40_Stock/` へ移す。

---

## 命名規約

```
30_Flow/
├── 現在の作業セッション.md  ← ローカル運用台帳（git/配布対象外）
├── 実案件/              ← クライアント案件
│   └── 実案件_○○/
├── 模擬案件/            ← 練習・実証用の模擬案件
│   ├── 模擬案件_テクノブリッジ/
│   └── 模擬案件_丸山製作所/
└── テスト/              ← Skill単体の検証・実験
    └── Skillテスト_0-1_協和精機/
```

ルール:
- 第1階層はカテゴリ（`実案件/` `模擬案件/` `テスト/`）。日付階層は使わない。
- 第2階層の接頭辞（実案件_/模擬案件_/Skillテスト_）で種別を識別する。
- Skill内で `{案件名}` と書く場合は、実体としては `30_Flow/{カテゴリ}/{案件種別}_案件名/` の案件ルートを指す。
- `現在の作業セッション.md` は複数案件の切り替え混在を防ぐローカル台帳。実案件名や現在地を含み得るためgit/配布対象外にし、成果物を書き込む前にアクティブ案件ルートと保存先の一致を確認する。

---

## 案件フォルダの内部構造

```
模擬案件_○○/
├── project-context.md          ← 案件の共有コンテキスト（SSOT）
├── _素材/                      ← 受領資料・参考資料（冷蔵庫。形式不問で放り込む）
│   ├── ヒアリング議事録_0612.docx
│   ├── 現行業務フロー.pptx
│   └── ...（形式不問、何でも放り込む）
├── _検査ログ.md                ← 自己検査の記録
├── 00_前段/                    ← フェーズ別の正典成果物
│   ├── 01_案件理解サマリー_v0.2.md
│   └── ...
├── 01_立ち上げ/
├── 02_計画/
├── 03_実行/
├── 04_監視コントロール/
├── 05_終結/
└── outputs/                    ← 中間生成物（下記参照）
```

### _素材/ の使い方

- クライアントからもらった資料（PPTX, DOCX, Excel, PDF, 議事録…）を形式不問で入れる。
- フェーズの進行に関係なく、いつでも追加・差し替えしてよい。
- Skillは必要な時にここを参照する。成果物をここに置かない（成果物はフェーズフォルダ）。
- `_素材/` の中身は案件フォルダの外（40_Stock/・配布物）に出さない（D-01ルール）。

### フェーズフォルダ vs outputs/

| | フェーズフォルダ（00_前段/ 等） | outputs/ |
|---|---|---|
| 位置づけ | **正典**。最終版の成果物 | **中間生成物**。作業途中の副産物 |
| 置くもの | 承認済み or 最新版の成果物 | サブエージェントの出力、検証レポート、旧版ドラフト |
| AIが参照すべき先 | こちらが常に優先 | 正典にない場合のみ参照 |
| 命名例 | `01_案件理解サマリー_v0.2.md` | `_verification_report_research.md`, `_research_axis1.md` |

**迷ったときの判断基準**: 「次のSkillの入力になるか？」→ Yes ならフェーズフォルダ、No なら outputs/。

### Git追跡ルール

`30_Flow/` は作業場なので、すべてをGitに残す場所ではない。

| 対象 | Git追跡 |
|---|---|
| フェーズフォルダ内のMarkdown正典成果物 | 可（実案件は除外） |
| `project-context.md` / `_検査ログ.md` / `chain-trace.json` | 可（実案件は除外） |
| `outputs/` 内の観察メモ・検証メモMarkdown | 必要な代表のみ可。次のSkill入力になるならフェーズフォルダへ昇格 |
| `export/` / `final-slides/` / `_generated/` / `_renders/` | 原則不可 |
| `_render/` / `_quicklook/` / `screenshots/` | 原則不可 |
| PPTX/docx/xlsx/png/jpg/pdfなどの生成バイナリ | 原則不可。教材として残す場合は `50_サンプル成果物/` に最小代表だけ置く |

理由:

- `outputs/` は中間材料であり、正典ではない。
- `outputs/` 内のMarkdownは「再現性・検証・観察に必要な代表」だけ残す。旧版ドラフトや重複した中間出力は残し続けない。
- 画像・HTML・PDF・PPTXは再生成できるものが多く、Git履歴と配布物を重くする。
- 例外として残す場合は「次回の検証に必要な最小代表」だけにする。

AI向け優先規則:
- 正典成果物は、作成中のv0.1であってもフェーズフォルダへ置く。
- `outputs/` は正典成果物を作るための中間材料に限る。
- `outputs/` 配下のMarkdownを参照した結果、それが後続Skillの入力・判断根拠になると分かったら、次版でフェーズフォルダ側の正典成果物へ統合する。`outputs/` を正典の置き場にしない。
- `project-context.md` は新規チェーン開始時に作る。古い前段停止案件で未作成の場合は、後続フェーズへ入る時点で作ればよい。
- 実案件で、ユーザーが「○○の件」「○○案件」など案件名だけで話しかけた場合は、まず `30_Flow/実案件/案件インデックス.md` が存在するか確認する。存在すれば、該当案件の `project-context.md` を読んでから返答する（実案件フォルダはgitignore配下）。読み取りだけなら `現在の作業セッション.md` のアクティブ案件は変更しない。
- 一般的な構造調査・Skill調査・README更新では、実案件名が明示されていない限り `30_Flow/実案件/` を横断検索しない。必要なら `30_Flow/README.md`、模擬案件、テスト、またはGit管理対象のファイルに探索範囲を絞る。
- 案件成果物を作成・更新する前、または `project-context.md` へ書き戻す前は、`現在の作業セッション.md` のアクティブ案件ルートと保存先が一致しているか確認する。不一致の場合は書き込まず、ユーザー確認を取る。
