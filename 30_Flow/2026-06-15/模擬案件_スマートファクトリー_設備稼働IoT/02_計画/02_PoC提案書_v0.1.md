## 1. 背景と目的

### 背景

生産管理システムの予定生産数と実績が乖離している。
原因は設備の計画外停止が報告に反映されておらず、稼働時間管理が作業員の終業時報告に依存していること。
FSで4方式を比較し、ゲートウェイデバイス方式を第一候補に選定済み。
会社として初のDXプロジェクトであり、要件整理もできていない状態が出発点。

### 目的

このPoCが担う3つの役割:
- **実現性の評価**: ゲートウェイデバイスで設備の稼働データを取得できるか（技術）、業務に使えるか（ビジネス）、運用できるか（運用）、コストは妥当か（コスト）を4軸で検証する
- **学びと発見**: 初のDXプロジェクトとして、机上では気づけない課題の発見と要件の具体化
- **予算確保**: 本番導入の費用見積り基礎情報の取得（W5）

## 3. スコープ

1工場の対象設備（🟡台数未確定）を対象に、ゲートウェイデバイスによるデータ取得からBIツールでの可視化までのE2Eを検証する。

検証対象:
- ゲートウェイデバイスとPLCの接続・データ取得
- セルラー通信（SORACOM Air）によるクラウドへのデータ送信
- AWS（S3+Kinesis+Athena）でのデータ蓄積・変換
- BIツール（JDBC）での稼働データ可視化
- 工場環境（振動等）でのデバイス安定稼働

## 4. やらないこと・やれないこと

| 対象 | いつやるか |
|---|---|
| 生産管理システムとの連携 | パイロット運用で検証 |
| 本番システムの設計・構築 | 本番導入時（外部パートナー） |
| 全工場への展開 | 本番導入時に策定 |
| 多様なデータ分析基盤（W3） | PoC後の将来ビジョンとして検討 |
| 予知保全・異常検知AI | 将来の拡張として検討可。今回は対象外 |
| 定量目標の設定（稼働率〇%向上等） | PoC結果を踏まえて設定 |

代替案: ゲートウェイ方式が不適合の場合、FSで検討した他3方式（作業者報告/PLCロギング/センサ外付け）に戻る。代替案の再評価はPoC Month1のH2判定後。

## 5. 評価軸および評価項目

### ノックアウトファクター（最優先）

**H2: ゲートウェイデバイスで対象設備（PLC経由）から稼働データを取得できるか**
→ 不可の場合、PoC全体の前提が崩れる。Month1で検証完了。

### 4軸の評価項目

| 軸 | ID | 評価項目 | 種別 | 評価方法 |
|---|---|---|---|---|
| **技術** | E-T1 | デバイスからPLC経由で稼働/停止データを取得できるか | 定量 | 取得テスト合否（Yes/No） |
| | E-T2 | デバイス→クラウド→BIツールのE2Eでデータが流れるか | 定量 | E2E疎通テスト合否 |
| | E-T3 | 工場環境（振動等）でデバイスが3カ月安定稼働するか | 定量 | データ欠損率・再起動回数 |
| | E-T4 | データフォーマットの変換（デバイス側 or クラウド側）の実現性 | 定性 | 変換処理の実装難易度・特性の把握 |
| **ビジネス・業務** | E-B1 | 取得データで設備稼働時間を正確に把握できるか | 定量 | 手動計測との照合（誤差〇%以内） |
| | E-B2 | 生産管理部門が「予実管理に使える」と評価するか | 定性 | BIツール閲覧後の評価（有用/条件付き/不可） |
| | E-B3 | PoCを通じて要件一覧が作成できたか | 定量 | 要件一覧の作成（Yes/No） |
| **運用** | E-O1 | 現場担当者だけでデバイスの設置・交換ができるか | 定性 | 設置体験後の評価（容易/要教育/困難） |
| | E-O2 | デバイスの遠隔管理（通信停止・ステータス確認）が機能するか | 定量 | 遠隔操作テスト合否 |
| | E-O3 | デバイス盗難リスクへの技術的対策が有効か | 定量 | IMEIロック・遠隔停止の動作確認 |
| **コスト** | E-C1 | セルラー通信の月額コスト（実機計測） | 定量 | 台数×通信単価×月数 |
| | E-C2 | クラウド利用費の月額コスト | 定量 | AWS従量課金の実績値 |
| | E-C3 | 本番規模（全工場）での年間ランニングコスト見積り | 定量 | PoC実績×スケールファクター |

## 6. システム構成・開発対象

```
[対象設備] → [PLC] → [ゲートウェイデバイス] → (SORACOM Air/セルラー) → [AWS]
                                                                              ├─ S3（データ格納）
                                                                              ├─ Kinesis Data Firehose（ストリーム処理）
                                                                              ├─ Athena（クエリ）
                                                                              └─ BIツール（JDBC接続・可視化）
```

**PoC用プロトタイプ構成**:
- 本番同等の構成は不要。評価のエッセンスを抽出した最小構成。
- デバイス: 🟡選定中（PLC仕様調査後に確定）
- 通信: SORACOM Air（au回線）。IMEIロック・API経由SIM管理
- クラウド: AWS（S3+Kinesis Data Firehose+Athena）。既存BIツールとJDBC接続

## 7. 評価の実施環境

- **場所**: メンバーが物理的にアクセスしやすい1工場を選定（クライアント指定済み）
- **対象設備**: 🟡台数未確定。PLCが接続されイーサネットポートがある設備
- **環境条件**: 空調完備（特殊温度要件なし）。振動への耐性が必要
- **データ取得条件**: 24時間稼働の設備を対象とし、稼働/停止の両状態を検証する

## 8. 遵守すべきレギュレーション

- **セキュリティ**: 社内に閉じたPoC。データはセンシティブでない。暗号化・認証認可は必要
- **ネットワーク**: 工場内LANへの接続不可（①指定）。セルラー通信で独立経路
- **デバイス管理**: IMEIロック＋遠隔通信停止で盗難対策
- **法的要件**: 特段のレギュレーション制約なし（製造業・非規制データ）

## 9. 実施スケジュール

| 時期 | マイルストン | 主な作業 |
|---|---|---|
| **Month1前半** | 準備完了 | PLC仕様調査・デバイス選定・調達・キッティング・クラウド環境構築 |
| **Month1後半** | **H2判定（ノックアウトゲート）** | デバイス設置・データ取得テスト・E2E疎通確認 |
| **Month2** | 中間報告 | BIツールでの可視化開始・生産管理部門評価開始・通信コスト計測開始 |
| **Month2〜3** | デモお披露目 | 現場＋関係者向けデモ・フィードバック収集 |
| **Month3** | **最終報告（4軸評価）** | PoC結果報告書作成・4軸評価・次段階移行判断 |
