# チェーントレース規約（正典 / SSOT）

最終更新: 2026-06-17（新設）
チェーン実行の全工程を1ファイルで俯瞰するための構造化ログ。
「何が・いつ・何を入力に・何を出し・検査はどうだったか」を1エントリ1 Skill実行で記録する。

## なぜ要るか（狙い）

既存の記録は3箇所にバラけている:

- `_検査ログ.md` … 検査結果だけ（入力・判断・タイミングがない）
- `project-context.md` の書き戻しログ … フェーズ遷移だけ（Skill単位の粒度がない）
- テーラリング判断ログ … 判断だけ（実行の文脈がない）

chain-trace.json はこれらを **補完する（置き換えない）** 統合ビュー。
既存3つはそれぞれの役割で残し、chain-trace.json は「チェーン全体を時系列で追える索引」として機能する。

## 置き場

`{案件名}/chain-trace.json`（案件フォルダ直下、1案件1ファイル）。
立ち上げフェーズの project-context 作成と同時に初期化する。
前段のみの段階では作らない（前段は軽量運用、本体に入ってから開始）。

## フォーマット

```json
{
  "案件": "丸山製作所 受注〜出荷プロセス改善",
  "開始日": "2026-06-16",
  "現在フェーズ": "05_終結",
  "entries": [
    {
      "seq": 1,
      "skill": "project-charter",
      "chain": "tachiage-chain",
      "phase": "01_立ち上げ",
      "timestamp": "2026-06-16T10:00",
      "inputs": ["00_前段/引き継ぎ書_前段→立ち上げ.md"],
      "output": "01_立ち上げ/01_プロジェクトチャーター_v0.1.md",
      "inspection": "✅7 🟡0 ❌0",
      "inspection_notes": "S-02中(SMART Aの根拠→修正済)",
      "tailoring": [],
      "duration_min": null,
      "notes": ""
    }
  ]
}
```

### フィールド定義

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `seq` | number | ✅ | 通し番号（1始まり、実行順） |
| `skill` | string | ✅ | 実行したSkill名（SKILL.mdのname） |
| `chain` | string | ✅ | 呼び出し元のchain名（手動起動なら `"manual"` ） |
| `phase` | string | ✅ | フェーズ（`01_立ち上げ` 等） |
| `timestamp` | string | ✅ | 開始時刻（`YYYY-MM-DDTHH:MM`、秒は不要） |
| `inputs` | string[] | ✅ | 入力ファイルの相対パス（案件フォルダ起点） |
| `output` | string | ✅ | 出力ファイルの相対パス |
| `inspection` | string | ✅ | 検査ゲート結果サマリー（`✅n 🟡n ❌n`） |
| `inspection_notes` | string | — | 検査での指摘要点（なければ空文字） |
| `tailoring` | string[] | — | このSkill実行中に記録したテーラリング判断（要点のみ。詳細はproject-contextのログ本体） |
| `duration_min` | number\|null | — | 所要分（計測できなければ `null`） |
| `notes` | string | — | 特記事項（差し戻し・再実行・スキップ理由等） |

### 特殊エントリ

通常のSkill実行以外に、以下のイベントもエントリとして記録する:

- **フェーズ遷移**: `skill: "phase-transition"`, `output: "→ {次フェーズ}"`
- **承認ゲート通過**: `skill: "approval-gate"`, `notes: "承認 / 条件付き承認 / 差し戻し"`
- **差し戻し・再実行**: 元のseqを `notes` に参照（例: `"seq3の差し戻し→再実行"`）

## ライフサイクル（誰がいつ書くか）

- **初期化**: `tachiage-chain` が project-context 作成直後に空の chain-trace.json を作成
- **追記**: 各成果物Skillが **検査ゲート通過後（✋承認の直前）** に1エントリ追記
- **chain**: オーケストレーターがフェーズ遷移エントリを追記
- **閲覧**: `build.sh --trace {案件パス}` で整形表示（後述）

### 共通ひな形との接続

成果物Skill共通ひな形の手順6「承認後」に、trace追記を1行足す:
> 6-b. `chain-trace.json` に本Skillのエントリを追記（検査結果・出力先・テーラリング判断を含む）

検査ゲートのレシート→チャット提示→✋承認→trace追記→project-context書き戻し、の順。

## build.sh --trace の出力仕様

`bash _tools/build.sh --trace {案件フォルダパス}`

整形出力例:

```
=== チェーントレース: 丸山製作所 受注〜出荷プロセス改善 ===
現在フェーズ: 05_終結  |  エントリ: 12件  |  開始: 2026-06-16

 #  Skill                    Phase           検査     時間   備考
 1  project-charter          01_立ち上げ     ✅7      —     S-02中→修正済
 2  stakeholder-register     01_立ち上げ     ✅7      —
 -  [→ 02_計画]
 3  integrated-plan          02_計画         ✅7      —
 4  risk-plan                02_計画         ✅7      —
 ...
=== 検査: ✅12 🟡0 ❌0  |  テーラリング判断: 3件 ===
```

- 検査の集計（全体で✅/🟡/❌がいくつか）
- テーラリング判断の一覧
- 差し戻し・再実行があれば強調表示

## 既存記録との関係（二重化しない）

| 記録 | 役割（残す） | chain-trace.jsonとの関係 |
|---|---|---|
| `_検査ログ.md` | 検査証跡の人間可読台帳 | traceの `inspection` / `inspection_notes` が索引、詳細は検査ログ本体 |
| project-context 書き戻しログ | 確定事実の蓄積 | traceのフェーズ遷移エントリが対応。書き戻し内容はproject-context側 |
| テーラリング判断ログ | 採否判断の正典 | traceの `tailoring` は要点のみ。詳細・理由はproject-context側のログ本体 |

chain-trace.json は **索引・俯瞰ビュー** であり、各記録の **詳細の正典を奪わない**。
