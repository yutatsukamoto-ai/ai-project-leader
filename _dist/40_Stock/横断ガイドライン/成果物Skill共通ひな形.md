# 成果物Skill 共通ひな形（型の型）

各フェーズの成果物子Skillが共有する骨格と、出力前の検査ゲートの正典。
薄く保つ。共通核だけここに置き、成果物固有（可変）は各SkillのSKILL.md＋referencesに置く。
穴埋め用スケルトンは `20_Skills/99_メタ/_seikabutsu-template/SKILL.md`（新規はこれをコピー）。

## 対象

- 成果物を作る子Skill＝前段成果物、チャーター、各種管理計画、WBS、課題・変更・状況報告、PoC/終結系の成果物。
- オーケストレーター（chain）は順番管理が役割なので対象外。
- 前段4本（案件理解〜計画提案）は別チェーン・独自ゲートあり。寄せるかは別途判断。

## 固定核：SKILL.mdの並び

A. **frontmatter** … name／description（フェーズ＋役割＋入力→成果物＋トリガー＋PMBOK#）
B. **見出し＋リード** … 「{フェーズ}の基幹成果物（#N）。{前工程}を入力に{目的}を作る」
   ＋ Progressive Disclosure一文（中身は `references/型と品質基準.md`・本体は薄く保つ）
C. **## 手順（6ステップ）**
   1. 入力を読む … 前工程の成果物＋`project-context`／前工程の✋確認が済か→未なら止める／無ければ次フェーズへ繰越
   2. 中身を埋める … `references/型と品質基準.md` の型・品質バー｜**【差し込みカード参照枠】**ドメインカードがあれば引く・無ければ素通り
   3. 推測は根拠明示・未確定は🟡 … 記号は §参照する正典 のとおり。Markdown正典の冒頭には `出力フォーマット規約.md` の「成果物冒頭メタ」を置く
   4. 停止・壁打ち … 巻き上げ／照合の素材を出す｜曖昧・不可逆な分岐が出たら `grill-me`（`20_Skills/90_横断/`）で1問ずつ詰める
   5. **【検査ゲート】** … 出力前にセルフチェック（次の§）→結果を出す→✋承認（必須停止）
   6. 承認後 … 確定事実を `project-context` へ書き戻し＋出力先 `{案件名}/{フェーズ}/NN_{成果物}_vX.{md|docx}`＋ **`chain-trace.json` にエントリ追記**（規約: `40_Stock/横断ガイドライン/チェーントレース規約.md`）
D. **## ルール** … 文体＝文章作法6ルール／記号＝表と数値の見せ方の正典／プロセス参照＝PMBOK#／多段階で勝手に進めない＝✋／各ステップで現在地を明示
E. **> 配布TODO** … referencesへの同梱物

## 検査ゲート（手順5・共通7項目）

出力（✋承認）の前に回す。
結果は ✅／🟡／❌ で出し、**最終判断は本人に返す**（直すか承認かは本人が決める）。

1. 入力の✋確認は済か … 未確認の前工程に乗っていないか
2. 必須項目・品質バーは埋まったか … 埋まらない所は🟡で明示したか
3. 記号は正典どおりか … 確度`●◐○`／採否`◎○△`／状態は言葉。混用していないか
4. 正典を二重化していないか … 要件は統合計画書"要求"の続きとして深掘り（M-01／M-02）
5. 出力先パス・版番号・冒頭メタは規約どおりか
6. **未確認の明示** … この成果物で確認・検討しなかった範囲を一覧にする（項目名＋スキップ理由）。「全部確認した」は疑う——入力が不足していた項目、前提を置いた項目、深掘りしなかった項目を洗い出す。本人が追加入力するか、手動で確認するか、次工程へ繰り越すかを判断できる材料にする
7. **スキップ理由の妥当性チェック（サボり回収）** … ⑥で出したスキップ理由を1件ずつ再検証する。理由が「入力が物理的にない」「本フェーズのスコープ外」なら妥当→そのまま。理由が「見落とし」「手間を省いた」「確認可能だが飛ばした」ならサボり→**その場でやり直して結果を更新**。やり直し後、再度⑥⑦を回す（妥当な理由だけになるまでループ）。ループ上限は2回——2回で収束しなければ残りを🟡として本人に返す

※成果物固有の検査は各Skillがこの§に足してよい（共通7項目は削らない）。

### 検査の独立エージェント委譲

設計基準: `40_Stock/横断ガイドライン/サブエージェント設計基準.md` に準拠。

#### なぜ分離するか

成果物を**書いた本人が自己検査する**と、自分の前提を疑えず盲点が残る。
独立したサブエージェント（Agentツール）に委譲することで、成果物を初見で読む目が入る（原則A-1: 検査専任、原則A-2: コンテキスト分離）。
検査の**中身（7項目+固有検査）は変わらない**。誰がやるかだけが変わる。

#### 起動方法

手順5で、メインエージェントは成果物ファイルを書き終えた直後に**Agentツールを1つ**呼び出す。
検証エージェントは成果物を読み取り専用で検査し、レポートをファイルに書き出す。
メインエージェントはそのレポートを読んでレシートを組み立て、✋承認に進む。

#### 検証エージェントの仕様

**prompt に含める情報:**
- 検査対象の成果物ファイルパス
- 共通7項目（この§の①〜⑦をそのまま転記）
- 当該Skillの固有検査項目（SKILL.mdに定義がある場合）
- レポートの出力先: `{案件名}/_inspection_{成果物名}.md`（成果物名を含め上書き防止）

**promptテンプレート:**

```
あなたは成果物の独立検査担当です。成果物の作成には関与していません。

## 検査対象
- ファイル: {成果物ファイルパス}

## 検査項目（共通7項目）
①入力の✋確認は済か — 未確認の前工程に乗っていないか
②必須項目・品質バーは埋まったか — 埋まらない所は🟡で明示されているか
③記号は正典どおりか — 確度●◐○／採否◎○△／状態は言葉。混用していないか
④正典を二重化していないか — 要件は統合計画書"要求"の続きとして深掘りされているか
⑤出力先パス・版番号・冒頭メタは規約どおりか
⑥未確認の明示 — 確認・検討しなかった範囲を洗い出す。「全部確認した」は疑う
⑦スキップ理由の妥当性チェック — ⑥の理由を1件ずつ再検証。「見落とし・手間省略」はサボり→やり直し。ループ上限2回

## 固有検査項目
{当該Skillの固有検査があればここに転記。無ければ「なし（共通7項目のみ）」}

## 制約
- 成果物ファイルは読み取りのみ。編集しない
- 検査レポートを {レポート出力先: _inspection_{成果物名}.md} に書き出すこと
- レポートは項目ごとに ✅／🟡／❌ と判定理由を記載
- ⑥⑦でサボりを検出した場合、やり直し内容と結果もレポートに含める
- メインへの返答は「✅n／🟡n／❌n、詳細はレポート参照」の1行のみ
- 10ステップ以内で完了すること
```

#### メインの後処理

検証エージェントの完了後、メインは以下を行う:
1. `_inspection_{成果物名}.md` を読む
2. レポートの内容をレシート書式（下記§）に整形して提示
3. ❌がある場合: メインが成果物を修正し、再度検証エージェントを起動（このループ上限は2回。⑦のサボり回収ループとは別カウント — ⑦は検証エージェント内部で回る）
4. 全項目✅または🟡のみ: ✋承認に進む
5. 承認後、`_inspection_{成果物名}.md` は削除する（証跡は `_検査ログ.md` に1行残す）

### ⑥⑦の設計意図（出典と原理）

AIは「確認した」と自己申告するが、実際には飛ばしている項目がある。直接「全部確認せよ」と指示してもAIは別の理由を付けて回避する。**一度スキップ理由を含めて全部出させ、最後にそのサボりだけを回収する**方が網羅率が上がる（出典: 清水えん @haruruuun0405「ユーザビリティ評価のSkill精度が上がった8つの方法」⑥⑦を成果物Skillの文脈に再構成）。

### レシート（結果の出し方・2026-06-15）

検査ゲートの結果は、毎回この固定書式の「レシート」で必ず提示する（守られた証拠を毎回見せる＝出す／出さないの自己申告にしない）。

書式（例）:

`── 検査ゲート｜{成果物名} {版} ──`
`✅ ①入力の✋確認  ✅ ②必須項目・品質バー  ✅ ③記号は正典どおり  ✅ ④正典の二重化なし  ✅ ⑤出力先・版番号・冒頭メタ`
`⑥未確認: {件数}件（下記）  ⑦サボり回収: {回収件数}件やり直し済 / {残}件は妥当`
`  - {項目A}: {スキップ理由}（妥当）`
`  - {項目B}: やり直し済→{結果の要約}`
`🟡 残: {🟡／❌があれば内容と申し送り先を1行で。全✅なら省略}`
`→ 判断はお任せします（このまま承認 ／ 修正）`

置き場（使いやすさ優先・2026-06-15決定）:

- **必ずチャットに出す**（✋承認の直前）＝その場で見える安心感。これが主役。
- **証跡**: `{案件名}/_検査ログ.md` に「日付・成果物・版・結果（✅n／🟡n／❌n）＋🟡❌の要点」を1行追記する（無ければ新規作成）。全✅でも1行残す。
- **客先提出物（.docx 等）の本体には埋めない**＝レシートは内部の安心感・証跡用。
- 固有検査を足したSkillは、その項目もレシートに含める。

## chain-trace.json 追記（手順6の一部）

検査ゲート通過後・✋承認の直前に、`{案件名}/chain-trace.json` へ1エントリを追記する。
フォーマット・フィールド定義は `40_Stock/横断ガイドライン/チェーントレース規約.md` に従う。

追記タイミング: レシート提示 → ✋承認 → **trace追記** → project-context書き戻し。
chain-trace.json が案件フォルダに存在しない場合（前段のみ等）はスキップしてよい。
