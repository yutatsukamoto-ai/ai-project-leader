# eval 目視層 judgeプロンプト（LLM判定者用）

目的: 機械層（`eval.sh`＝型）では見えない「中身の質」を、LLM判定者で点検できる形にしたもの。
今（Cowork）は単発のLLM/人が手で回す。Claude Code版では subagent 化して自動実行・結果記録する（覚書§3）。
判定基準の正典は `合格チェックリスト.md`。本書はそれを「1ゴールデンずつ機械的に判定させる」ための指示文。

---

## 使い方

**自動生成（推奨）**: `bash _tools/eval-judge.sh` で全ゴールデンのjudgeプロンプトを `_tools/eval/judge-results/` に一括生成。1件だけ回すなら `bash _tools/eval-judge.sh --prompt <skill> <case>` で標準出力に出せる。

**手動**: 1ゴールデン（成果物md）ごとに、下のプロンプトに「対象Skill名」「ゴールデン本文」を差し込んで判定者に渡す。

判定者は共通チェック＋該当Skill固有チェックを順に見て、各項目 `pass`／`fail`／`n/a` と一文の根拠を返す。
`fail` が1つでもあれば、そのゴールデンは目視層FAIL。原因が「Skill劣化」か「仕様変更（ゴールデン更新すべき）」かを人が切り分ける（M-09）。

---

## プロンプト本体（差し込んで使う）

```
あなたはプロジェクトマネジメント成果物の品質監査者です。
以下の成果物を、チェックリストの各項目について pass / fail / n/a で判定し、各項目に一文で根拠を付けてください。
判定は「型（節が在るか）」ではなく「中身の質・判断の妥当性」を見ます。節が在っても中身が浅い・誤っていれば fail。
最後に、成果物全体の総合判定（pass/fail）と、fail があればそれが「Skillの指示の劣化」か「仕様が正しく変わった可能性（ゴールデン更新候補）」かの見立てを1〜2文で述べてください。

# 対象Skill
{skill名（例: project-closeout）}

# 共通チェック（全Skill）
- 状態・達成度・優先度を、確度記号(●◐○)や採否記号(◎○△)で代用していないか（言葉で書いているか）
- 件数・期日・金額・目標値が具体か（ぼかしていないか）
- 一文一改行か。だらだら続く長文パラグラフが無いか
- 見出しがラベル型で、流し読みで目次になるか
- やらないこと・停止ポイントが落ちていないか
- 前工程の成果物・project-contextの確定事実と矛盾していないか

# Skill固有チェック
{該当Skillの固有チェック項目を 合格チェックリスト.md からそのまま貼る}

# 判定対象の成果物
{ゴールデン本文をそのまま貼る}
```

---

## 出力フォーマット（判定者に守らせる）

```
- [共通1] pass — （根拠一文）
- [共通2] fail — （根拠一文）
- [固有1] pass — （根拠一文）
...
総合: fail
切り分け見立て: Skillの指示の劣化（達成度を○×で書いており、3値の指示が効いていない）。
```

## 運用メモ

- 機械層（eval.sh＝型）が緑でも、judgeで質のfailが出ることがある。これが急所2（型は通るが中身が劣化）への網。
- judgeの判定はモデル依存。判定者には**回す本番より強いモデル**を割り当てるのが望ましい（覚書§5の使い分け）。
- Cowork版: `eval-judge.sh` でプロンプト生成→Claude（同会話 or 別会話）に渡して判定→結果は `eval/judge-results/` に保存。
- Claude Code版: `eval-judge.sh --prompt <skill> <case>` を subagent に渡して自動実行→結果を `eval/judge-results/` に追記（覚書§3）。
