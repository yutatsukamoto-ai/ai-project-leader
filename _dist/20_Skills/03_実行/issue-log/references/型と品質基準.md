# 課題管理表 型と品質基準

出典: 山藤商事 課題管理表v0.1の伴走（2026-06-13）で言語化した手順・判断を写し取ったもの。
位置づけ: SKILL.md本体（薄い）の品質バー。該当ステップでのみ読む（Progressive Disclosure）。

---

## この成果物の正体

実行フェーズで「いま手を打たないと進まないこと」を一元管理する生きた台帳。
計画を動かすと計画どおりに進まない事柄が必ず出る。それを落とさず次の一手に変えるのがこの表。
品質バー＝「全課題に次の一手・担当・期限があるか」＋「リスクと混ざっていないか」＋「連動が双方向に辿れるか」。

## 列設計

| 列 | 中身 | 注意 |
|---|---|---|
| ID | I-01… | 連番。消さない。 |
| 区分 | 作業課題／判断待ち／リスク対応 | 必ず1つ（下の定義）。 |
| 課題 | 何が起きている／何が要るか | 一文。事実で書く。 |
| 影響 | 放置するとどうなるか | 優先度の根拠になる。 |
| 優先度 | 高／中／低 | 記号でなく言葉。律速は高。 |
| 次の一手 | 具体的アクション | これが書けないなら課題ではない。 |
| 担当 | 人名（仮置き時は役割PL等） | 「みんな」は不可。一人に名指す。 |
| 期限 | 日付（未定なら段階の相対表現） | キックオフ後に日付へ確定。 |
| 状態 | オープン／対応中／完了 | 記号でなく言葉。完了でも行は残す。 |
| 関連 | R-xx・計画の節 | 登録簿IDや出どころを指す。 |

## 区分の定義（3種・必ずどれか1つ）

- **作業課題** ＝ 計画した作業を進める上で解決が要ること。
- **判断待ち** ＝ 誰かの意思決定・指名で止まっていること。
- **リスク対応** ＝ リスク登録簿の対応策を"今実行する"アクション（PMBOK #39）。事故化ではなく先回りの実行。

## リスク登録簿との連動ルール（最重要）

課題管理表とリスク登録簿（計画§3）は別物。混ぜず、リンクでつなぐ。

- **役割の違い**: 登録簿＝まだ起きていない不確実なこと（兆候つきの記録・消さない）。課題表＝すでに起きている／今手を打つべきこと（担当・期限つきで閉じる）。
- **移すトリガー**: リスクが「もう"かもしれない"でなく今向き合うべき」に変わったら、課題表に1行立てる。
- **二重管理しない**: 中身は課題表にだけ書く。登録簿には残さない。連動はコピーでなく **ID（R-xx）のリンク**。
- **移管理由は2分岐**:
  - リスク対応＝事故前に先回りで対応策を実行。登録簿の状態＝「対応へ移管（移管先I-xx）」。
  - 顕在化＝リスクが実際に事故化。登録簿の状態＝「顕在化済」。
- **双方向トレース**: 課題側は「関連」列でR-IDを指す／登録簿側は「状態」列で移管先IDを指す。
- **前提**: これが成り立つには登録簿の各行にIDが要る。無ければ統合計画書のリスク表にID列＋状態列を足す（版up）。

## seedの引きどころ（計画書から）

- §4の🟡（確定すべき事項）→ 「判断待ち」になりやすい。
- §3リスクのうち今動くもの → 「リスク対応」（登録簿の状態を更新）。
- ゲート通過条件 → 段階出口までに閉じる課題。
- 対象業務が未確定 → 仮置きで通し、特定自体を律速の1課題に立てる（決め打ちでズレるのを防ぐ）。

## 出力（Excel）

最終形は .xlsx＝生きた台帳。

- 「課題管理表」シート: ヘッダー固定・オートフィルタ・優先度で色分け（高=赤系/中=黄系/低=灰）。課題・影響・次の一手は折り返し。
- 「凡例」シート: 区分3種・優先度・状態・連動ルール・出典を載せる（提出物では記号でなく言葉の凡例を再掲）。
- 出力先 `{案件名}/03_実行/01_課題管理表_vX.xlsx`。壁打ち中はマークダウン、固まってからxlsx化。

## やってはいけない

- 次の一手・担当・期限の無い行を残す（それは課題でなくリスクか願望）。
- リスクと課題を1つの表に混ぜる／登録簿と課題表で同じ中身を二重に持つ。
- 優先度・状態に確度記号（●◐○）・採否記号（◎○△）を流用する。
- 対象業務をヒアリング前に決め打ちする（仮置き→現場が選定）。
- 完了した課題の行を消す（記録・振り返りの材料）。
