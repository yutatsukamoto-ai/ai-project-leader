# gyoumu-flow-designer — 方法論に基づく業務フロー設計・描画

業務フローを**設計し、描画ツールに出力する**横断Skill。
「ただの描画マシン」ではなく、目的確認・レイヤー選定・入力充足チェック・3Wブロック切り・L3記載項目ルールといった設計知識をreferencesに持ち、設計品質を担保したうえで描画に渡す。

## アーキテクチャ: 2層分離

```
設計層（Step 0〜2）… ツール非依存・共通
  references/設計方法論.md
描画層（Step 3〜4）… ツール依存・差し替え可能
  references/figma-draw.md  ← FigJam（デフォルト）
  references/miro-draw.md   ← Miro
  references/pptx-draw.md   ← PPTX（未実装）
```

設計知識と描画APIを分離することで、描画先の追加・変更が設計層に影響しない。

## いつ使うか

- **明示呼び出し**: 「業務フローを作って」「フロー図を描いて」「As-Is/To-Beフローを作って」「業務の流れを可視化して」「フロー図にして」。
- **AI能動サジェスト**: 業務改善型プロジェクトでAs-Is/To-Beの可視化が要ると判断したとき、「業務フローを作りますか？」と一声かける。
- **成果物選定ガイドでのランク**: 任意。

## やり方（背骨）

### Step 0: 目的確認・前提チェック

`references/設計方法論.md` を読み、以下を確認する。

1. **目的の確認**: 「何のために作るのか」「誰が使うのか」を明示的に聞く（§1）。
2. **レイヤー選定**: L1概要 / L2部門・役割 / **L3処理・データ（推奨）** / L4手順書（別ドキュメント扱い）。未指定ならL3を推奨する（§2）。
3. **As-Is / To-Be**: どちらを作るか、または両方か。両方の場合は同一ボード内にAs-Is左・To-Be右で並列配置。未指定なら確認する。
4. **描画先選定**: FigJam（デフォルト） / Miro / PPTX。未指定ならFigJam。
5. **入力充足チェック**: 選択したレイヤーの必要情報が揃っているか確認する。L3の場合は§3-3の7つの記載項目に対して不足を指摘し、追加ヒアリングを促す。

不足が多い場合は先にL1〜L2の概要フローを作り、それをベースに再ヒアリングしてL3に進む段階的アプローチを提案する。

### Step 1: 入力分析

入力テキスト（議事録・ヒアリングメモ・箇条書き・口頭書き起こし等）から以下を抽出する。

| 抽出項目 | 手がかり |
|---|---|
| アクター（Who） | 「〜が」「〜部」「〜担当」の主語 |
| タスク | 「〜する」「〜を確認」などの動詞 |
| 判断ポイント | 「〜の場合」「承認/却下」 |
| システム/ツール | 固有名詞（Salesforce、Excel等） |
| データソース | DB名、ファイル名、参照元 |
| ドキュメント | 帳票名、報告書名、成果物 |
| サイクル/タイミング | 日次/週次/月次/随時、具体的な曜日・日付・時間 |

### 3Wブロック切り

抽出した情報を `設計方法論.md §3` の3W基準でブロックに分割する。

- **Who**: 担当者が変わる → 切る
- **When**: 日/時間をまたぐ → 切る
- **What**: 成果物が変わる → 切る

出力: ブロック一覧（仮。Step 2で座標テーブルに昇格）。

### Step 2: フロー設計

ブロック一覧を座標テーブル＋接続計画に昇格させる。

1. **スイムレーン設計**: アクターごとにレーンを割り当てる。
2. **座標テーブル作成**: 各ブロックにノードタイプ・座標(x, y)・サイズを割り当てる。ノードタイプと配色は `設計方法論.md §4` に従う。
3. **接続計画**: ブロック間の接続線（接続元ref・接続先ref・接続点magnet）を設計する。接続点の重複禁止。
4. **フロー方向**: 横書き（左→右）固定。

座標テーブルの書式例:

```markdown
## 業務: [業務名]
レーン1(Y=100): [アクター名]
レーン2(Y=370): [アクター名]

| ref | ノード名 | タイプ | X | Y | 幅 |
|-----|---------|--------|---|---|---|
| start | 開始 | 開始 | 100 | 195 | 60 |
| t_input | データ入力 | タスク | 260 | 190 | 140 |
| ...

## 接続計画
| 元ref | 元magnet | 先ref | 先magnet | 種別 | ラベル |
|-------|---------|-------|---------|------|--------|
| start | RIGHT | t_input | LEFT | 通常 | — |
| ...
```

ここで**ユーザーに設計を提示し、確認を取る**。

### Step 3: 描画

描画先に応じたreferencesを読み、座標テーブル＋接続計画を実際の描画コマンドに変換する。

| 描画先 | 読むreferences | 描画手段 |
|---|---|---|
| FigJam（デフォルト） | `references/figma-draw.md` | Figma MCP `use_figma` でPlugin APIコードを実行 |
| Miro | `references/miro-draw.md` | Miro REST API（curl） |
| PPTX | `references/pptx-draw.md`（未実装） | python-pptx |

各referencesに描画コードのテンプレート・座標系の違い・トラブルシュートが記載されている。

### Step 4: 検証

描画結果のスクリーンショットを取得し、設計意図との整合を確認する。

チェック観点:
- ノードの重なりがないか
- 接続線の交差が最小限か
- ラベルの可読性（テキスト切れがないか）
- スイムレーンからのはみ出しがないか
- 接続先の正しさ

問題があればStep 2-3を部分的にやり直す。

### 付録: 更新フロー設計

Step 4完了後、`設計方法論.md §9` に基づいて更新フロー設計を付録として提示する。

| 項目 | 記入内容 |
|---|---|
| 更新トリガー | いつ更新するか（仕様変更時・四半期レビュー等） |
| 更新責任者 | 誰が管理するか |
| 更新手順 | どうやって更新するか |
| 最新版の置き場 | どこに保管するか |
| 影響チェックリストへの組み込み | 変更管理との連動 |

## 残し方（出力）

| 出力 | 形式 | 説明 |
|---|---|---|
| 業務フロー図 | FigJam / Miro / PPTX | 描画先に応じた形式。URLまたはファイルとして提供 |
| 座標テーブル＋接続計画 | Markdown（Step 2の中間成果物） | gyoumu-list-excelへの接続点として再利用可能 |
| 更新フロー設計（付録） | テキスト | 運用ルールの提案 |

## してはいけないこと

- 目的・利用者を確認せずにいきなり描画する。
- L3を選んだのにL3の7記載項目の充足を確認しない。
- いきなりL3から書き始める（L1〜L2→L3の段階を飛ばす）。
- ノードタイプの色を設計方法論.md §4と異なる色で描画する。
- 同じノードの同じ接続点に複数のコネクタを接続する。
- 座標テーブルなしで描画コードを書く（設計と描画を分離する）。
- 更新フロー設計の提示を省略する。
- 描画後のスクリーンショット確認を省略する。
