# スケジュール管理計画 作成（子Skill）

計画フェーズの基幹成果物（#16 スケジュールマネジメントの計画 / #17 スケジュールの作成）。
統合計画書のクリティカルパス記述を**具体的なベースラインとバッファ管理方針**に展開する。
中身の品質基準・節構成は `references/型と品質基準.md` を見る（Progressive Disclosure。本体は薄く保つ）。

## 位置づけ

統合計画書§2はスコープと簡易WBS・クリティカルパスを記述している。
本Skillはその記述を**スケジュールベースライン＋バッファ管理＋進捗測定方法**に展開する。
wbs-builder（`20_Skills/02_計画/wbs-builder`）が出力する `wbs.json` と連動し、WBSの日程情報をベースラインに引き上げる。

## 手順

1. **入力を読む**: `{案件名}/02_計画/01_統合計画書_vX.md`・`{案件名}/02_計画/NN_WBS_wbs.json`（あれば）・`project-context.md`。
   - 統合計画書の✋承認が済かを確認する。未なら止める。
   - WBS（wbs.json）がある場合はそこから日程・依存関係を引く。無い場合は統合計画書§2の簡易WBSから作成する。
   - `project-context.md` が無ければ止めず、承認後の書き戻し（手順7）で新規作成する。

2. **スケジュール方針を定義する**: `references/型と品質基準.md` の型に沿う。
   - 見積技法（類推・パラメトリック・三点見積等）を案件に合わせて選定する。
   - スケジュール管理の単位（日・週）・ツール・更新頻度を明示する。
   - 【差し込みカード参照枠】ドメインカードがあれば引く（無ければ素通り）。

3. **マイルストーン一覧とクリティカルパスを特定する**: 
   - WBSまたは統合計画書§2からマイルストーンを抽出し、完了基準を付与する。
   - タスク間の依存関係からクリティカルパスを特定する。
   - **スケジュールベースライン**（計画日程の確定版）を定義する。

4. **推測には根拠を明示**: 見積の根拠（類推元・前提条件）を添える。未確定（見積精度が低い等）は🟡で残す。
   - 記号は `references/表と数値の見せ方.md` の正典どおり。

5. **停止・壁打ち**: バッファ管理方針の妥当性（バッファ量・配置・消費ルール）とクリティカルパスの前提条件の素材を出す。
   - 曖昧・不可逆な分岐が出たら `grill-me`（`20_Skills/90_横断/`）で詰める。

6. **検査ゲート（出力前・必須停止）**: 共通7項目＋本Skill固有を回す（共通ひな形§検査ゲート）。
   - 共通: ①入力✋確認済か ②必須項目・品質バー（未確定は🟡） ③記号は正典どおりか ④正典を二重化していないか ⑤出力先パス・版番号 ⑥未確認の明示（確認しなかった範囲＋理由を一覧） ⑦サボり回収ループ（スキップ理由を再検証、サボりはやり直し）。
   - 固有: ⑧クリティカルパスが特定されているか ⑨マイルストーンに完了基準があるか ⑩WBSとの整合性（wbs.jsonがある場合）。
   - 結果を ✅／🟡／❌ で出し、最終判断は本人に返す。✋承認を得る。
   - **結果はレシート書式で必ず提示する**（チャットに出す＋`{案件名}/_検査ログ.md`へ1行追記）。

7. **承認後**: 確定事実を `project-context.md` へ書き戻す。
   - 出力先 `{案件名}/02_計画/NN_スケジュール管理計画_vX.md`。

## ルール

- 文体は `references/文章作法.md`（6ルール）に従う。
- 記号は `references/表と数値の見せ方.md` の正典（他軸へ流用しない）。
- プロセス参照は `references/PMBOK第8版_40プロセス一覧.md` の #番号で行う（#16/#17）。
- WBS（wbs.json）との**二重化をしない**＝wbs.jsonは事実（予定/実績の日付）のJSON、本計画はベースライン＋管理方針の文書。日程データはwbs.jsonを正典とし、本計画は参照する。
- 多段階で勝手に進めない。✋承認なしに次工程／提出版を確定しない。
- 各ステップ開始時に現在地を明示する（例:「計画 スケジュール管理計画」）。
