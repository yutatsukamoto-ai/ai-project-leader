# PoC評価報告書 型と品質基準

出典: 『IoTプロジェクトで学ぶ実践PoC』Ch5（評価-判定）＋Ch10（実践編）より再構成。
アイデアの再構成であり、表現・図版の転載はしない。
位置づけ: SKILL.md本体（薄い）の品質バー。該当ステップでのみ読む（Progressive Disclosure）。

---

## この成果物の正体

PoC提案書で設定した評価軸に実績を対照し、Go/NoGoの判断材料を意思決定者に返す文書（#9・PoC特化）。
PoC提案書（計画フェーズ）が「これから検証する」文書なのに対し、PoC評価報告書は「検証した結果を閉じる」文書。
**ワンショット評価**＝生きた台帳でなくPoCフェーズを閉じるコミュニケーション文書。
品質バー＝「提案書の基準どおりに判定しているか」＋「ノックアウトファクターの結論が出ているか」＋「Go/NoGoの根拠と条件が明確か」。

## 文書の型（6節・この順で組む）

| 節 | 中身 | 品質バー |
|---|---|---|
| §1 評価概要 | PoC目的の再確認（提案書§1から引用）、実施期間・範囲・体制の概要 | 提案書の目的と一致しているか。スコープの変更があれば明示。 |
| §2 評価結果（4軸） | 評価軸ごとに、評価項目・成功基準・実績・判定（合格／条件付き／不合格）を対照表で示す | 4軸すべてに記載。定量は数値、定性は観察事実。判定は3値で言葉。 |
| §3 ノックアウトファクター判定 | 提案書で特定したノックアウトファクターの結果を独立節で | 合格／不合格を明確に。不合格なら§5のGo/NoGoに直結。 |
| §4 課題と対策 | PoC実施中に発見された課題、対策案、本番移行への影響 | 課題は事実ベース。対策は実現可能性を示す。本番への影響度を明示。 |
| §5 Go/NoGo推奨 | §2・§3に基づく推奨（Go／条件付きGo／NoGo）。条件付きの場合は条件を列挙 | 推奨の根拠を§2・§3の結果で裏付け。条件は具体的・検証可能に。 |
| §6 本番移行への提言 | Go判定の場合: 本番化に向けたスコープ・体制・スケジュール・概算コストの方向性 | GoまたはNoGoで内容が変わる。NoGoなら撤退計画・代替案。 |

## 評価結果の対照表（§2の書き方）

提案書の評価項目をそのまま左列に持ってくる。基準を後から変えない。

| 軸 | 評価項目 | 成功基準（提案書） | 実績 | 判定 |
|---|---|---|---|---|
| ビジネス・業務 | {提案書から} | {提案書から} | {実測値・観察事実} | 合格／条件付き／不合格 |
| 技術 | … | … | … | … |
| 運用 | … | … | … | … |
| コスト | … | … | … | … |

- 判定は**合格／条件付き／不合格の3値を言葉で**書く。○×の二択にしない。
- 条件付きの場合、「何をすれば合格になるか」を右列に付記する。
- 定量項目は数値で。定性項目は観察事実を根拠に。
- 確度（●◐○）・採否（◎○△）を判定に流用しない（`references/表と数値の見せ方.md`）。

## ノックアウトファクターの扱い（§3）

提案書で「これが駄目なら先に進めない」と特定した要素の結果を独立節で示す。

- ノックアウトファクターが不合格 → Go/NoGo推奨はNoGo寄りになる（自動的にGoにはならない）。
- 不合格でもGoにする場合は、リスク受容の根拠と代替策を✋✋で確認する（不可逆判断）。
- 提案書にノックアウトファクターが設定されていなかった場合、その事実を記載し、実質的にノックアウトだった要素があれば遡及的に明示する。

## Go/NoGo推奨の3値（§5）

| 推奨 | 意味 | 必要条件 |
|---|---|---|
| Go | 本番移行を推奨 | 4軸の主要評価項目が合格。ノックアウトファクターが合格。 |
| 条件付きGo | 条件を満たせば本番移行可 | 不合格項目があるが、対策により解消可能。条件を列挙。 |
| NoGo | 本番移行を推奨しない | ノックアウトファクター不合格、または複数軸で不合格。撤退計画・代替案を§6に。 |

- 推奨はあくまで「材料」。最終判断は意思決定者が行う。
- 条件付きGoの条件は**具体的・検証可能・期限付き**にする。「頑張れば何とかなる」は条件ではない。

## PoCの学びの扱い

PoCの3つの役割（PoC提案書 `references/型と品質基準.md` より）:
1. 実現性の評価 → §2・§3・§5に反映
2. 学びと発見 → §4に反映（課題だけでなく「想定外の発見」も記録）
3. 予算確保 → §6の概算コストに反映

「失敗」も成果。期待と異なる結果は正直に報告し、その学びを§4に残す。
AP-04（失敗が許されないPoC）に陥っていないかセルフチェックする。

## アンチパターンチェック（評価段階）

PoC提案書の `references/PoCアンチパターン.md` 7パターンのうち、評価段階で特に注意すべきもの:

| パターン | 評価段階での兆候 |
|---|---|
| AP-04 失敗が許されないPoC | ネガティブな結果を矮小化している。不合格項目を「条件付き」に無理やり格上げしている。 |
| AP-06 やりっぱなしPoC | 結果は出たが整理されていない。次アクションが不明確。 |
| **ゴール後付け**（評価段階固有） | PoC実行後に成功基準を変えている。「最初からこの基準だった」と辻褄を合わせている。 |
| AP-03 際限なきPoC | 「もう少し追加検証すれば合格になる」が繰り返される。評価を先延ばしにしている。 |

ゴール後付けは最も危険なアンチパターン。提案書の基準が正。結果に合わせて基準を変えてはならない。
基準自体に問題があった場合は、「基準の問題」として§4に記録し、次回のPoC計画に申し送る。

## §6の書き分け（Go/NoGoで変わる）

**Go・条件付きGoの場合**:
- 本番化に向けたスコープの方向性（PoCからの拡張点）
- 必要な体制・リソースの概算
- スケジュールの目安（次のフェーズゲートまで）
- 概算コスト（PoCのコスト実績をベースに）
- PoC環境から本番環境への移行で注意すべき点

**NoGoの場合**:
- 撤退の判断根拠（§2・§3の結果から）
- 代替案があれば提示（別アプローチ・別技術・スコープ縮小等）
- PoC資産（コード・データ・ナレッジ）の保全方針
- サンクコストに引きずられない旨を明記

## 品質バー（検査ゲートで使う）

- [ ] 提案書の評価軸4軸すべてに実績と判定があるか
- [ ] ノックアウトファクターの判定が明示されているか
- [ ] Go/NoGo推奨に根拠（§2・§3の結果）と条件（条件付きGoの場合）があるか
- [ ] 成功基準がPoC提案書と一致しているか（ゴール後付けでないか）
- [ ] 定量項目に数値、定性項目に観察事実の根拠があるか
- [ ] §6がGo/NoGoの結果に応じた内容になっているか
- [ ] アンチパターン（特にAP-04・ゴール後付け）に陥っていないか

## ✋✋承認前チェック（不可逆判断）

Go/NoGo判定はプロジェクトの方向を決める不可逆判断。通常の✋に加え二重停止（✋✋）を設ける。

承認前に意思決定者へ以下を明示する:
- Go/NoGo推奨とその根拠（1〜2文で）
- ノックアウトファクターの結果
- 条件付きGoの場合: 条件のリスト＋条件を満たす見通し
- この判断を覆すのに必要なこと（可逆性の説明）
