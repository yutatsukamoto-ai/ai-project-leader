# スライド作成（slide-craft）

## このスキルの目的

スライド作成の「正解」は装飾ではなく**伝わること**にある。
「伝わらない」スライドの最大の原因はデザイン以前に「何を伝えたいか（メッセージ）」が曖昧なまま作られていること。

このスキルは実務家の知見を体系化し、次の順番でスライドを支える:

**メッセージ → 構造型 → レイアウト → デザイン（白黒→配色）**

この順番には理由がある。スライドが「伝わる」かは、PowerPointを触る前のコンセプト段階で8割決まる。

**横断ガイドライン参照:** `40_Stock/横断ガイドライン/スライド生成の原則.md` を全ステップの判断基準として使う。特に§7（基本ルール5則）、§8（構造型）、§9（白黒先行）、§10（仕上げの引き算）は毎回参照すること。

## 配色方式 — ブランドアンカー起点

配色は常にブランドアンカーから都度生成する（横断ガイドライン §1）。固定パレットは持たない。

**フォールバック順:**
ロゴ → コーポレートカラー → 業種慣習色 → 白基調＋アクセント1色（安全なデフォルト）

**デフォルト配色（白基調クリーン）:** 白〜薄灰がベース（95%）、ダークグレーがテキスト、アクセント1色（5%以下）。メインカラー（25%帯）は基本的に不要。色でなくフォントサイズと太さでメリハリを作る。

## ページ種別ガイド

コンテンツの性質に応じて、以下の種別を割り当てる。32パターン辞書（layout-patterns.md）から具体的な座標を引くときの入り口になる。

| コンテンツの性質 | ページ種別 | 対応パターン例 |
|--------------|----------|-------------|
| スライド冒頭 | **表紙** | 1-1（完全中央）、1-2（上部配置） |
| セクション区切り | **扉** | 1-1（中央）、8-1（フルブリード）、3-1（L字）|
| 3〜4つの並列ポイント | **カードグリッド** | 2-3（3カラム）、2-4（4カラム＋グリッド） |
| 手順・プロセス・ステップ | **フロー** | 6-3（ステップフロー）、6-1（横タイムライン） |
| 現状vs改善、Before/After | **比較** | 2-1（左右分割）、3-3（Z字） |
| 概念・事例をビジュアルで並べる | **イメージ並列** | 2-3 + 画像エリア、8-5（カード型） |
| まとめ・単独メッセージ | **強調** | 7-1（ミニマル）、1-1（完全中央） |
| 数値表・詳細データ | **高密度** | 7-3（高密度）、テーブル型 |

**割当ルール:**
- セクション区切りは3枚以上のコンテンツが続く場合に自動挿入
- 同じ種別を3枚以上連続させない（単調になる）
- 迷ったらカードグリッド（最も汎用的）

## タイポグラフィ基準

フォントとサイズの階層を統一する。ブランドアンカーで色は変わっても、この階層は固定。

| 用途 | フォント | サイズ | ウェイト |
|------|--------|------|--------|
| スライドタイトル（主張文） | 游ゴシック | 28pt | bold |
| カード見出し | 游ゴシック | 16pt | bold |
| 本文・説明文 | 游ゴシック | 12pt | normal |
| 強調メッセージ | 游ゴシック | 14pt | bold |
| 丸囲み番号（01, 02...） | 游ゴシック | 14pt | bold |
| 扉タイトル | 游ゴシック | 36pt | bold |
| 注釈・出典 | 游ゴシック | 9pt | normal |

**16pt未満を本文に使わない**（読めない=言っていないのと同じ）。注釈・出典のみ9ptを許容。

## 制作モード（テキスト → PPTX）

入力テキストを受け取ったら、次の流れで進める。各チェックポイントでユーザーの確認を取り、OKが出るまで次へ進まない。

### Step 1 — メッセージ設計

スライド全体と各ページの「たった1つのメッセージ」を固める。これが最重要。

1. テキストを読み、**Why / What / How** の3層で全体の狙いを言語化する:
   - **Why**: このスライドで達成したい読後ゴール（読んだ人に何をしてほしいか）
   - **What**: それを叶える主要メッセージ・主張
   - **How**: Whatを実現する構成の方向性（ページの並び）
2. 推定した「読み手」「トーン」「想定枚数」も添える
3. 「この解釈で合っていますか？」と**1回だけ**確認する。テキストから読めることは自動で埋める

**メッセージの作り方（design-principles.md より）**: 書き散らかし→削ぎ落とし。言いたいことを全部出してから、伝わるギリギリまで削る。1ページ＝1メッセージ。2つ言いたいなら2ページに分ける。

**インプット不足の診断（横断ガイドライン §4）**: 読み手・主張・数字の3点が揃っていない場合は、不足を指摘して追加を依頼する。

### Step 2 — 構成案（構造型判定つき）

1. Why/What/Howを基に全体構成（表紙・アジェンダ・本文・まとめ）を生成する
2. 各ページに以下を1行で添える:
   - 番号・タイトル・そのページのメッセージ（主張）
   - **構造型**（横断ガイドライン §8: 順番/対比/仕分け/問題→打ち手/原因→結果/結論先出し）
3. 「この構成で進めますか？」と確認する

枚数の目安: 未指定ならHowの論点数を基準に5〜15枚で提案。

**省略せず、スライドを分ける（変えないmd準拠）:** 情報が多いときに要約して1枚に詰め込まない。「3つのルール」→ 概要1枚＋ルール1＋ルール2＋ルール3の4枚。人間が作ったらそうするはず、という構成に従う。

### Step 3 — レイアウト選択

各ページについて、構造型→パターン辞書（`references/layout-patterns.md`）の順に選択する。

1. Step 2で決めた構造型から候補パターンを絞る（横断ガイドライン §8 の対応表）
2. 候補の中からページのメッセージと情報の数・性質で最適を選ぶ
3. 迷うページだけ3択を簡潔に提示。明確なページは最推奨を当てて進める
4. 同じパターンを3枚以上連続させない
5. **基本ルール5則（横断ガイドライン §7）を適用**: 2軸で並べる、同じ言葉は軸に吸収、軸は2つまで、大小は面積で、事実/根拠/解釈は置き場所で分ける

### Step 4 — デザイン決定

**白黒先行（横断ガイドライン §9）:**
1. まず白黒で構造を組む（テキスト＋図形＋余白）
2. 白黒のまま「3秒で何を言っているか分かるか」を確認
3. 成立したら配色を載せる

**配色（ブランドアンカー起点）:**
デフォルトは白基調クリーン（95:0:5）。コーポレートカラーがある場合はそれをアクセントに。色の印象（赤=情熱、青=知性、緑=安心等）はdesign-principles.md §4を参照。

**ルール:**
- アクセント（赤やビビッド）は「ここを見て」の1点だけ
- 視線: 左上→右下。良い事象・結論は右。ハイライトの周りに大事な情報を固める
- 整列: 縦軸・横軸で要素を揃える。意図しないズレはノイズ
- 白黒印刷でも区別が付くよう、色だけに依存せず明度差・ラベルでも区別

**パーツ3点セット（変えないmd準拠 — 全コンテンツページで統一）:**
1. **丸囲み（アイコンまたは番号）**: アクセント色の丸背景＋白線アイコン or 白抜き番号。並列ならアイコン、順序があるなら番号
2. **薄灰カード**: 薄い灰色の角丸四角形（枠線なし・影なし）。情報のグループ化に使う
3. **底部まとめバー**: ページ最下部に角丸の帯＋左にアイコン＋「So What」の一文。ページの結論を強制する装置。アイコンはページの役割で使い分ける:
   - **✓**（デフォルト）: 結論・まとめ。そのページの要点を締めるとき
   - **!**: 注意・強調。「ここを見落とすな」という警告的メッセージ
   - **→**: 次のアクション。読み手に行動を促すとき
   - **★**: 重要な気づき。意外性のあるインサイトを伝えるとき

この3点を全コンテンツページで繰り返すことが統一感の正体。ページごとにあしらいを変えない。

**仕上げの引き算（横断ガイドライン §10）:**
- 主役を1点に絞り脇役のトーンを落とす
- 囲み線でなく余白の近さでグループ化
- 矢印は1つの相手にだけ
- 注釈は欄外に逃がす
- 簡易テスト: 装飾を全部消して構造が崩れないなら、それらは不要

### Step 5 — PPTX生成

1. pptxgenjs（Node）でスクリプトを生成・実行する。各ページは確定したパターンの座標ガイド（layout-patterns.md）に従う
2. 出力先: ワークスペースに `[日付]_[タイトル].pptx` として保存し、`present_files` で提示

**pptxgenjs の必須ルール（守らないとファイル破損）:**
- カラーコードに `#` を付けない（`'E65A00'` であって `'#E65A00'` ではない）
- 8桁カラーコード（透明度込み）を使わない。透明度は `transparency` 等の別プロパティで指定
- shadow は使い回さず毎回新規生成する
- 改行は `breakLine: true` を使う
- 日本語フォントは「游ゴシック」「Meiryo」「Noto Sans JP」など環境にあるものを指定

実行手順:
```bash
cd <作業ディレクトリ>
npm install pptxgenjs >/dev/null 2>&1
node build_slides.js
```

### Step 6 — 提出前レビュー（横断ガイドライン §12）

生成後、ユーザーに渡す前に以下5項目を自己チェックする:

1. **形** — 各ページの構造型が内容と合っているか
2. **主張** — 各ページの上部1/3に主張文が28pt以上で置かれているか。30字以内で言い切れているか（§11）。主張文がないページは作らない
3. **数字** — 主張の裏付けとなる数字・事実があるか
4. **並べ方** — 基本ルール5則に違反していないか
5. **仕上げ** — 引き算ルールの簡易テストを通すか

問題があればStep 3〜4に戻って修正してから出力する。

## レビューモード（既存スライド → 改善提案）

ユーザーが既存スライド（PDF/PPTX/画像）や「このスライドどう？」と相談してきたら、診断する。

1. **メッセージ診断**: 各ページの「たった1つのメッセージ」は明確か。羅列になっていないか
2. **構造型診断**: 情報の論理構造（§8）とレイアウトが合っているか。比較なのにフロー型で描いていないか
3. **視線診断**: 3秒で「どこを見ればいいか」が分かるか。強調が機能しているか
4. **基本ルール診断**: 5則（§7）に違反していないか。軸が3つ以上ないか
5. **情報量診断**: Busy（うるさい）になっていないか。引き算ルール（§10）を適用できないか
6. **デザイン診断**: 配色は白基調クリーン（95:0:5）に収まっているか。白黒で成立するか（§9）

診断は「良い点 → 直すと効く点（優先度順）→ 具体的な直し方」の順で伝える。詳細な判断基準は `references/design-principles.md` を参照。希望があれば修正版PPTXまで作る。

## 判断のコツ

- **メッセージが先、コンテンツとデザインは後**。コンテンツ作りは楽しくて時間を溶かすが、メッセージを表していなければ意味がない
- **構造型で迷ったら「結論先出し」**: 最も汎用的で外れが少ない
- **削る勇気**: 残しすぎが「伝わらない」最大の原因。情報量は「少なすぎ」より「多すぎ」がほぼ常態
- **色は諸刃の剣**: 多色は視線を散らす。目立たせるのはアクセント1色だけ。迷ったら白基調クリーン
- **揃っていることは想像以上に効く**: 整列はセンス不要で誰でもできる、最もコスパの高い改善
- 質問は最小限に。テキスト/スライドから読み取れることは自動で解釈し、確認は要所だけ
- **見出しは30字以内で言い切る**: 「〜について」で止めない。主張を込める

## お手本（インプット源）

伝わるスライドの型を学ぶなら企業のIR・決算説明資料が最良の教材（時間をかけて作られ・メッセージがあり・定期更新される）。配色ではなく「見せ方」を参考にする。

| 会社 | 見せ方の強み |
|------|------------|
| 資生堂 | 文字サイズと情報量のコントロール。高密度でも整理されている |
| DeNA | 左サイドバー＋右メインの構造的レイアウト。論理的な図表 |
| Retty | 余白広め・情報量を絞り1メッセージを大きく見せる |
| SMBC | 細かい数字の見せ方が上手 |
