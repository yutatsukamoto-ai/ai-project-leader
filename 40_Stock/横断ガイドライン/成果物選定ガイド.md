# 成果物選定ガイド（全Skill共通）

プロジェクト開始時に「この案件では最低限何を作るか」を決めるためのガイド。
成果物マップ（`20_Skills/成果物マップ.md`）の母集合に対し、必須/推奨/任意のランクを付与する。
テーラリング前提＝毎回すべてを作るわけではないが、「必須」は原則省略しない。

確定経緯: 3案件目完走後のふりかえりで「必要な成果物の作り忘れ防止」の仕組みとして確定（2026-06-16）。

---

## A. 成果物ランク表

### 前段チェーン（0-x）

| 成果物 | ランク | 根拠 |
|---|---|---|
| 案件理解サマリー（0-1） | 必須 | 全案件の入口。事実/推測/不明点の構造化がないと後工程が空回り |
| 課題仮説シート（0-2） | 必須 | Why/What/Howの整理がないと計画提案に根拠がない |
| リサーチメモ（0-3） | 推奨 | 既知の領域・短期案件ではスキップ可。スキップ判断を明示する |
| 計画提案書（0-4） | 必須 | 意思決定者の承認を得る唯一の文書。承認なしに立ち上げへ進まない |

### 本体チェーン（01〜05）

| フェーズ | 成果物 | ランク | 根拠 |
|---|---|---|---|
| 01 立ち上げ | プロジェクトチャーター | 必須 | 「迷ったとき立ち戻る一枚」。これがないとスコープが宙に浮く |
| 01 立ち上げ | ステークホルダー登録簿 | 推奨 | 関係者が3名以下なら省略可。ただし「誰の承認が要るか」は必ず決める |
| 02 計画 | 統合計画書 | 必須 | 要求・スコープ・リスクの統合。簡易版でもゼロにしない |
| 02 計画 | WBS | 推奨 | 統合計画書§2の簡易WBSで足りる案件もある。独立WBSは作業が10以上で有効 |
| 02 計画 | PoC提案書 | 任意 | PoC型プロジェクトのみ。段階設計でPoCを含む場合に作成 |
| 03 実行 | 課題管理表 | 必須 | 「いま手を打つべきこと」の台帳。全案件で必要 |
| 03 実行 | コミュニケーション管理 | 任意 | ステークホルダーへの情報伝達を実績追跡する生きた台帳。外部関係者が多い案件で有効 |
| 03 実行 | リソース管理 | 任意 | メンバーの稼働・スキル充足を追跡する台帳。チーム規模が大きい案件で有効 |

| 05 終結 | 終結報告書 | 必須 | 閉じないプロジェクトは存在しない。フェーズ終結でも必須 |

### 未整備領域（将来Skill化候補）

| 成果物 | ランク | 着手トリガー |
|---|---|---|
| チームチャーター | 任意 | 大規模混成チーム（複数社・複数部門の混成でメンバー10名超）で、働き方の合意が必要な案件。小規模ならキックオフMTGまたは統合計画書内で代替可 |

※品質計画（#6/#33）は独立Skillにしない。品質管理は各Skillの型と品質基準.md・検査ゲート共通7項目・受入基準に構造的に内包済み。規制業界等で独立文書が契約納品物として求められる場合はアドホックで作成する。
※要件定義書は独立Skillにしない。scope-planの要求事項一覧（ID・優先度・受入基準・トレーサビリティ付き4区分）に内包済み。粒度が必要な場合はscope-planを厚く書くことで対応する。

## B. テーラリング6ドライバーとの連動

計画提案書（0-4）の手順2で評価する6ドライバーが、成果物の厚みを決める。

| ドライバー | 重いとき（→成果物を厚くする方向） |
|---|---|
| D1 規模 | WBS独立化、スケジュールBL、コスト見積もり、ソーシング戦略 |
| D2 説明責任 | コスト見積もり・コストBL、状況報告の頻度UP、ソーシング戦略 |
| D3 統制 | 変更管理表の厳格運用、リスク管理計画 |
| D4 スコープ安定性 | 低い→変更管理表必須、スコープ定義書 |
| D5 スピード | スケジュールBL、WBS独立化 |
| D6 定着リスク | コミュニケーション計画、ステークホルダー登録簿必須化 |

**原則**: ドライバーが軽い案件は「必須」だけで始める。重いドライバーが出てきたら、対応する成果物を追加する。

## C. 能動サジェストのルール

Claudeがプロジェクト開始時（前段チェーンの0-1完了後、または立ち上げのチャーター作成時）に以下を能動的に提示する:

### タイミング

1. **0-1案件理解サマリー承認直後**: 案件の概要が見えた時点で、必須成果物リスト＋推奨の要否を確認
2. **0-4計画提案書のテーラリング判断時**: 6ドライバー評価の結果を見て、任意成果物の追加を提案
3. **実行中に逆転条件が発動したとき**: 「関係者が増えたのでコミュニケーション計画を作りませんか？」等

### 提示フォーマット

```
この案件の成果物チェックリスト:

【必須】（原則省略しない）
- [ ] 案件理解サマリー（0-1）
- [ ] 課題仮説シート（0-2）
- [ ] 計画提案書（0-4）
- [ ] チャーター
- [ ] 統合計画書
- [ ] 課題管理表
- [ ] 状況報告
- [ ] 終結報告書

【推奨】（この案件では？）
- [ ] リサーチメモ（0-3）→ {スキップ可の条件に該当するか}
- [ ] ステークホルダー登録簿 → {関係者の数}
- [ ] WBS → {作業数}
- [ ] 変更管理表 → {スコープ変動の見込み}

【この案件で検討】（6ドライバーから）
- {D2重 → コスト見積もりを追加？}
- {D6重 → コミュニケーション計画を追加？}

どこまで作りますか？
```

### やらないこと

- 全成果物を毎回作ることを強制しない。必須以外は「提案して本人が決める」
- 未整備のSkillを「ないから作れない」としない。Skillがなくてもアドホックで作れる。Skillは品質を安定させる装置であって、前提条件ではない
