## 1. ブランドアンカー起点でデザインシステムを作る

自社テンプレートに縛られない。聞き手にとってテンプレ準拠は関係なく、「分かりやすさ」と「ブランドらしさ」が必要十分。

**ブランドアンカーの優先順位:**
1. **ロゴがある場合** → ロゴの色・雰囲気からパレットとスタイルを自動派生する
2. **ロゴがない場合** → 以下のいずれかをアンカーにする:
   - コーポレートカラー（Webサイト・名刺等から抽出）
   - 業種の慣習色（例: 医療=白/青、建設=緑/茶、IT=青/グレー）
   - クライアントが普段使っている資料の色味（過去資料から抽出）
   - 上記もない場合 → ネイビー+白+アクセント1色の安全なデフォルトパレットを適用

**原則:** テンプレートに従うのではなく、ブランドアンカーから都度デザインシステムを生成する。
これにより自社テンプレートとAI生成の相性問題が消える。

## 2. 省略しない構成設計

長文は要約せず、構成マップで全情報をスライドへ割り振る。

**AIの典型的な失敗:** 要約して大事な手順・条件・注意書きが消える。

**正しいアプローチ:**
- 見出し、表、チェックリスト、注意書きを適切な形式に割り振る
- 情報が多すぎる場合は1枚に詰めず、スライドを分ける
- 入っていないことは出さない。書かれている内容が全て大事という前提で構成する

**具体例（分割ルール）:**
「セキュリティの3つのルール」のような構造化された長文:
- NG: 1ページに全部ぶち込む（AIがやりがちだが人間はこうしない）
- OK: 概要1枚（3つあります、それはこれとこれとこれ）+ 各ルール1枚ずつ

**判断基準:** 「人間が作ったらこうするはず」に従わせる。

## 3. 禁止事項の明文化で出力を安定させる

AIが勝手に品質を落とす挙動は、事前に禁止しておく。
「やらないでね」を明文化すると、毎回の出力が安定する。

**Cowork版の禁止事項:**
1. **1枚に複数トピックを詰め込まない** — 1スライド1メッセージを厳守。情報過多なら分割する
2. **入力にない情報を勝手に追加しない** — 原稿に書かれていない補足や例を捏造しない
3. **変に要約しない** — 表やリストの項目数を勝手に減らさない。全項目を構成に入れる
4. **装飾過多にしない** — 不要なアイコン・イラスト・グラデーションを入れない

## 4. 100点のインプット前提

良いスライドが出るかは、原稿の整理度合いで決まる。

**「100点のインプット」の定義:**
- テキストを読むだけでその内容が全て分かる状態
- 伝えたいことが自分の言葉で書き切られている状態
- 構成（何をどの順番で話すか）が確定している状態

**インプット不足の診断（つまずいたらここを疑う）:**
- 「読み手」が決まっていない → 誰に見せるかを毎回明記する
- 「主張」がない → 「だから何？」に一文で答えられるか
- 「数字」がない → 主張の裏付けとなる事実・数値が添えられているか
上の3つが揃っていない原稿からは、どんなSkillでも良いスライドは出ない。

**このスキルの位置付け:**
スライドSkillは「整理された原稿を、分かりやすいスライドに変換する」ツール。
テーマだけ渡して中身もAIに考えさせるアプローチとは異なり、中身は人間が100%コミットしている前提。
→ 結果として中身の漏れ・歪みがなく、見た目だけをAIが最適化する構造になる。

**フロー:** 整理された原稿 → 分かりやすい構成 → 使えるスライド

## 5. プレビュー運用（段階的生成）

多枚数資料は最初の数枚で方向性を確認してから残りを生成する。

**ステップ:**
1. 全体構成を先に設計する（構成マップを作る）
2. 最初の3〜5枚を生成する
3. デザイン方向・文章精度・レイアウトを確認する
4. 承認後に残りを一括生成する

**メリット:**
- 方向性のずれを早期に検知できる
- 全枚数生成してからやり直すコストを避けられる
- 修正が必要な場合「N枚目だけ作り直す」指示で済む

## 6. ヘッダー・フッターの統一補正

出力後にヘッダーのタイトル位置とブランドアンカー（ロゴ等）の位置を補正し、見た目を揃える。

**なぜ必要か:**
AIが1枚ずつ生成すると、微妙にサイズや位置がずれることがある。
ヘッダー・ブランド表示だけを統一するだけで、全体の統一感が大幅に向上する。

**Coworkでの適用:**
pptxgenjs等でコード生成する場合は、ヘッダー部分を定数化して全スライドに同一座標で配置することで構造的に解決できる。
→ 画像生成方式と違い、コード生成方式ではこの問題が起きにくい（Coworkの強み）。

## 7. レイアウト基本ルール5則

スライドのレイアウトを決める前に、以下の5つを必ず適用する。パターン辞書（32パターン等）は「形」を提供するが、形の中で情報をどう配置するかはこの5則が決める。

1. **2本の軸で並べる** — スライドを「表」として捉える。縦軸と横軸を設定し、情報を行列で整理する。軸があると読み手は比較・対照ができる。
2. **同じ言葉を二度書かない** — 繰り返される共通部分は軸のラベルに吸収する。各セルには差分だけを置く。冗長さが消え、情報密度が上がる。
3. **1枚に詰める軸は2つまで** — 縦×横の2軸が限界。3軸目が必要なら別のスライドに分ける。軸が多いと読み手の認知負荷が跳ね上がる。
4. **数の大小は図形の大きさで見せる** — 数値の比較は、数字だけでなく面積・長さ・太さで直感的に伝える。「3倍」なら面積3倍の図形にする。
5. **事実・根拠・解釈は置き場所で分ける** — 事実（数字・ファクト）は見出しに、根拠（図・表・グラフ）は中央に、解釈（So What）は別枠（吹き出し・注釈欄）に置く。混ぜると読み手が「これは事実？意見？」と迷う。

## 8. 6つの構造型 — パターン選択の上位判断

32のレイアウトパターンを選ぶ前に、そのページの情報が「どの論理構造か」を判定する。構造型が決まればパターンの候補は3〜5個に絞れる。

6つの構造型は2層に分かれる。**表現系**（情報の並べ方）と**意味系**（論理の組み立て方）。表現系は「どう見せるか」、意味系は「なぜそう並ぶか」の判断に使う。

### 表現系（並べ方）

| 構造型 | いつ使う | 向くパターン例 |
|-------|---------|-------------|
| **順番** | 手順・プロセス・時系列 | 6系（タイムライン・フロー） |
| **対比** | AとBの比較・Before/After | 2-1（左右分割）、Before/After型 |
| **仕分け** | カテゴリ分類・グルーピング（MECE） | 2-3/2-4（カラム）、4系（中央集中） |

### 意味系（論理）

| 構造型 | いつ使う | 向くパターン例 |
|-------|---------|-------------|
| **問題→打ち手** | 課題提起→解決策→効果 | 3-3（Z字）、2-1（左右：問題/打ち手） |
| **原因→結果** | 因果関係・Why/So what | 6-3（ステップフロー）、ツリー型 |
| **結論先出し（ピラミッド）** | 主張→根拠の並べ。帰納法（事実→結論）にも演繹法（結論→根拠）にも使える | 1系（単一配置で結論）→2-3/2-4（根拠をカラムで） |

**使い方:** ページのメッセージを読み→表現系/意味系のどちらが主か判定→構造型を選択→候補パターンから選ぶ。迷ったら「結論先出し」が最も汎用的。

## 9. 白黒先行の検証原則

配色は最後に行う。まず白黒（グレースケール）で構造が成立するか検証してから色を載せる。

**なぜか:**
色に頼って構造を伝えようとすると、色を変えた瞬間に伝わらなくなる。白黒で成立する構造は、どんな配色でも壊れない。

**手順:**
1. レイアウトを白黒で組む（テキスト＋図形＋余白だけ）
2. 白黒のまま「3秒で何を言っているか分かるか」を確認する
3. 分からなければ構造の問題。色ではなく配置・サイズ・余白を直す
4. 白黒で成立したら、§9a のルールに従って色を載せる

**Coworkでの適用:**
pptxgenjsで生成する際、まず構造（座標・サイズ・テキスト配置）を先に確定し、カラーコードの適用を最終ステップにする。コードの構造としても自然。

## 9a. アクセント1色族の原則（配色の鉄則）

70:25:5で色を載せるとき、**デッキ全体でアクセント色は1色族**に限定する。
「1色族」とは、アクセント色本体＋その淡色版（tint）の2段階のみ。

**4つの鉄則:**

1. **並列アイテムは同色** — 対等な項目（01/02/03、カード並列、フロー内の等価ステップ）を色で区別しない。位置（左/中/右）で区別は十分伝わる。並列に異なる色を割り当てると、意図しない優先順位やカテゴリの印象を生む。
2. **色で区別していいのは意味の差があるときだけ** — Before/After、Go/No-Go、現状/提案のように意味が異なる場合のみ色を変える。その場合も「アクセント色 vs 無彩色（グレー）」の2値で表現し、第3の色を持ち込まない。
3. **あしらい（装飾要素）はアクセント色で統一** — バッジ、コールアウト帯、矢印コネクタ、アイコン背景など、構造を補助する装飾要素はすべてアクセント色（または淡色版）で揃える。あしらいに別の色を使うと統一感が崩れる。

**あしらいの標準パターン（推奨）:**

| 要素 | 色の使い方 |
|------|----------|
| 番号バッジ（01/02/03） | アクセント色の塗り＋白テキスト。並列は全て同色 |
| コールアウト帯（ページ下部） | アクセント淡色の背景＋アクセント色バッジ＋濃色テキスト |
| 矢印・コネクタ | アクセント色テキスト（→）またはアクセント色の線 |
| カード背景 | 無彩色（F5F5F5等）。枠線は原則なし |
| 強調カード（1枚だけ） | アクセント色の枠線で差別化。塗りは変えない |
| テーブルヘッダー行 | **メイン色（navy等）の塗り＋白テキスト**。アクセント色は使わない |
| ハイライト行 | アクセント色の塗り＋白テキスト（1行だけ） |

**4. ヘッダーとアクセントは役割で分ける** — テーブルヘッダーは「構造」（毎テーブルに存在）なのでメイン色（25%ゾーン）を使う。アクセント色は「注目」（ここを見て）の1点にだけ使う。同じ色を使うとヘッダーと強調行の区別がつかなくなり、特にヘッダー直後の行を強調すると完全に一体化する。

**アンチパターン（やりがち）:**
- 3カラム並列で左バーの色を緑・青・金に変える → 全部同じアクセント色にする
- Go=緑、No-Go=赤 → Go=アクセント色、No-Go=グレーにする
- フロー図のステップごとに色を変える → 全ステップ同色、強調したい1つだけアクセント色
- テーブルヘッダーにアクセント色を使い、強調行も同じアクセント色にする → ヘッダーはメイン色、強調行だけアクセント色にする
- 並列カードの1枚だけ枠線を追加して強調する → あしらい（枠・バー・背景）は全カード同一。強調は色（アクセントvs無彩色）で表現し、枠線の有無で差をつけない

## 10. 仕上げの引き算ルール

スライドの仕上げは「足す」ではなく「削る」。AI生成は装飾を足しがちなので、意識的に引き算する。

**5つの引き算:**
1. **主役を1点に絞り、脇役のトーンを落とす** — 全要素が同じ目立ち方をしていると、何も目立たない。主役（メッセージ・結論）だけを太字/アクセント色にし、残りはグレーや小さい文字に落とす。
2. **囲み線ではなく余白の近さでグループ化する** — 罫線・枠線を引くと「線」自体がノイズになる。近接（近くに置く）と遠隔（離す）だけでグループは伝わる。
3. **矢印は1つの相手にだけ向ける** — 1本の矢印が複数に分岐すると視線が散る。複数の関係を示したいなら、矢印を増やすのではなく構造（ツリー・フロー）で表現する。
4. **注釈は欄外に逃がす** — 本文エリアに注釈を混ぜると情報の階層が崩れる。注釈・出典・補足はフッター近く（y=5.0以降）に小さく（9pt）まとめる。
5. **底部ゾーンは1要素だけ** — ページ下部のコールアウトバー（So What帯）は1つで十分。「ポイント」ボックスとコールアウトバーなど同じ役割の要素を2段積みすると密着して読みにくく、情報の階層も崩れる。表の集計行（合計・小計）も底部の結論的要素としてカウントする — 集計行の直下にコールアウトバーを置くと表の一部に見える。複数の結論的要素は1つに統合する。

**簡易テスト:** スライドから装飾（枠線・影・アイコン・バッジ）を全部消してみて、情報の構造が崩れないなら、それらは不要。

## 11. 見出しの書き方（スライド固有）

横断ガイドライン/文章作法の一般ルールに加え、スライドの見出しには追加の制約がある。

- **30字以内で言い切る** — 見出しは一目で読める長さにする。30字を超えたら削るか分割する
- **体言止めか断定で終わる** — 「〜について」「〜に関して」で止めない。「売上30%増」「コスト構造の転換が必要」のように主張を含める
- **「だから何？」に答えている** — 見出しを読むだけでページの結論が分かる状態が理想。見出しが目次的（「背景」「課題」）なら、結論を足す（「背景：市場が縮小している」）

## 12. 提出前レビュー5項目

スライド完成後、提出前に以下の5項目を順に確認する。

1. **形** — 各ページの構造型（§8）が内容と合っているか。並列なのにフロー型で描いていないか
2. **主張** — 各ページの見出しに主張が入っているか（§11）。「〜について」で止まっていないか
3. **数字** — 主張の裏付けとなる数字・事実があるか。根拠なき主張になっていないか
4. **並べ方** — 基本ルール5則（§7）に違反していないか。軸が3つ以上ないか
5. **仕上げ** — 引き算ルール（§10）の簡易テストを通すか。不要な装飾が残っていないか
