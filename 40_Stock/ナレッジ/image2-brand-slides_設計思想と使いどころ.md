# image2-brand-slides 設計思想と使いどころ

作成: 2026-06-19
出典: 社内外勉強会での解説・Q&Aを一般化・再構成したナレッジ

---

## このナレッジの前提

- **対象**: image2-brand-slides の考え方と使いどころ。特定ツール実装の操作手順書ではない
- **生成方式**: 編集可能なPPTXファイルを作るのではなく、**スライド1枚を1画像として生成する**方式。後からPowerPointで細かく編集したい場合はこのアプローチは向かない
- **向いている用途**: 見た目の完成度と伝達力が重要な資料（勉強会・発表・社外向け説明資料等）
- **向いていない用途**: 上司レビューで数字・文言を繰り返し修正する資料、複数人がPowerPoint上で共同編集する資料
- **ツール依存**: 現状 Codex デスクトップアプリ（Mac）でのみ安定動作。CI/CD環境やClaude.ai Web版では画像生成の前処理が欠落するため使用不可

---

## 1. なぜテンプレートを捨てるか（発想転換）

### 従来の問題

多くの組織は「会社の固定テンプレート」を起点にスライドを作ることを求める。このテンプレートをAI生成に持ち込むと2つの問題が起きる。

1. **ズレへの注意が向く**: テンプレートに少しでもズレがあると「ここが違う」という視線が生まれ、AIで作ったこと自体への信頼が落ちる
2. **テンプレートの品質上限に縛られる**: 古い・デザインが微妙なテンプレートをベースにすると、どれだけ頑張っても品質の上限がテンプレートで決まる

### 発想転換

> 会社らしさの最小単位は**ロゴ**だけでよい。聞き手はテンプレートが会社の正式版かどうかより、内容と見た目が一致しているかを見ている。

テンプレートを捨て、ロゴだけを渡してその都度デザインシステムを生成する。これにより上限がなくなり、毎回その案件・発表に最適なビジュアルを作れる。

---

## 2. 入力の作り方：100点テキストの定義

image2-brand-slides は**入力テキストが100点であることを前提**に設計されている。

### 100点テキストとは

> **そのテキストを読むだけで、内容のすべてが理解できる状態。**

- 伝えたいことを自分の言葉で書き切っている
- 構成（順番・流れ）が整理されている
- 省略や「あとで口頭で補足する」がない

### なぜ要約させないか

AIにテキストを渡して「要約してスライド化」させると、何が重要かの判断がAIに移る。タイトルも本文テキストもすべて重要であり、省略すると意味が失われる。そのため、このSkillは要約を行わず、与えられたテキストをそのまま構成最適化してスライドに割り振る設計になっている。

### 実務での作り方

1. 話したい内容をMarkdownで書き切る（箇条書き・表・コード等を含めて可）
2. 読み返して「これを読めば話の全体がわかるか」を確認する
3. OKになったものを入力として渡す

## 3. アーキテクチャ：ロゴ＋内容MD → デザインシステム → Image 2

### インプット

```
ロゴファイル（PNG等）  ＋  内容テキスト（Markdown）
```

### 処理フロー

```
1. ロゴを解析 → カラーパレット・フォントトーン・余白ルールを決定（デザインシステム生成）
2. 内容テキストを解析 → スライド構成を最適化（1枚に収める量・分割ルールを適用）
3. GPT Image 2 でスライド1枚ずつを画像として生成
4. タイトル・ロゴの上揃え補正（位置ズレの後処理）
```

### GPT Image 2 を使う理由

GPT Image 2 は現状最強クラスの画像生成モデルであり、スライド1枚を1画像として生成する用途での精度が特に高い。テキストの配置・余白・整列を正確に再現できる点で従来モデルと大きく異なる。

---

## 4. 生成フロー（5枚確認 → 並列生成 → 補正）

### ステップ1: 方向性確認（最初の5枚）

最初に5枚を生成して方向性を確認する（所要約5分）。デザイン・レイアウト・情報密度が想定通りか確認し、問題なければ続行。

### ステップ2: 並列生成

サブエージェントを使った並列処理で残り全枚を生成する。1枚ずつ逐次生成すると多大な時間がかかるため並列化が重要。ただしサブエージェント数が多すぎると統一感が崩れるため、最初の5枚で統一感を確認してから全体に進む。

実績値: 47枚スライドを約20分で生成。

### ステップ3: 後処理（タイトル・ロゴ補正）

生成後、タイトルとロゴの上揃え補正を行う。これをしないと枚によってロゴ位置・タイトル位置がわずかにずれ、全体の統一感が崩れる。細かいが品質に直結するポイント。

### 修正の方法

特定ページだけ修正したい場合は「N枚目だけやり直して」と指示することで、そのページのみ再生成できる。全体を作り直す必要はない。

## 5. 使いどころと制限

### 使いどころ

| 用途 | 適否 |
|---|---|
| 勉強会・社内発表 | ◎ |
| 外部向け説明資料 | ◎ |
| 提案書（内容が固まった後） | ○ |
| 営業資料（繰り返し修正が前提） | △（向かない） |
| 上司レビューで数値・文言を繰り返し直す資料 | ✕ |

### 技術的制限

| 制限 | 内容 |
|---|---|
| 動作環境 | Codex デスクトップアプリのみ安定動作 |
| OS | 現状 Mac のみ（Windows 未検証） |
| 出力形式 | 画像（PNG）→ PDFエクスポート。PPTXとしての直接出力はなし |
| SVG | GPT-4.5 はSVG精度が低いため原則使わない（Font Awesome等は例外） |

## 6. スライドである必要はあるか（目的再定義）

このSkillは「スライドを高品質に作る」ことを支援するが、そもそも**スライドが必要かどうかを問うこと**も重要な視点。

### 問い直しのポイント

- 聞き手が本当に必要としているのは「スライド」か、それとも「内容と口頭説明の一致」か
- スライドでなくMarkdown/Notionページを投映しながら話すスタイルでも目的が達成できる場面は多い
- 資料の目的（説明・承認・保存）によって最適なフォーマットは変わる

### 判断の軸

> **ロゴが入っていて見た目の統一感があれば、聞き手にとっては十分であることが多い。テンプレートの正確な再現は、作る側の都合であって聞き手の都合ではない。**

image2-brand-slides を使うかどうかよりも先に「この資料の目的は何か、スライドである必要はあるか」を確認することが、結果として良い資料につながる。
