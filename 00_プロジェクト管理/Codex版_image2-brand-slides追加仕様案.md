# Codex版 image2-brand-slides 追加仕様案

作成日: 2026-06-18
状態: **MVP実装済み（2026-06-18）。**

---

## 1. 背景

Codex版のスライド生成は、既存 `slide-craft` のPPTXコード生成よりも、参考資料 `image2-brand-slides` の思想を主軸にする。

参考にするもの:

- `10_参考資料/スライド・プレゼン/CodeXによるシン・スライド生成/勉強会メモ_CodeXによるシン・スライド生成.md`
- `10_参考資料/スライド・プレゼン/CodeXによるシン・スライド生成/image2-brand-slides.zip`
- `20_Skills/90_横断/image2-brand-slides/`
- `40_Stock/横断ガイドライン/スライド生成の原則.md`
- `20_Skills/90_横断/slide-craft/`
- Zenn記事「会社員のためのCodex資料作成術: Skillで統一感のあるプレゼン資料を作る実践ガイド」（2026-06-16公開、2026-06-17更新）

狙いは、編集可能なPPTXを最初からコードで組むことではない。ロゴやブランドアンカーからデザインシステムを作り、Image2で1スライド1画像として生成し、必要に応じて画像をPPTXへ貼り付ける。

Zenn記事は「構成」「視覚」「仕上げ」を分ける考え方が参考になる。ただし、本仕様ではImage2生成を主成果物にするため、記事の「全体イメージ画像→編集可能PPT化」を主フローにはしない。取り込むのは、工程分離・Style Markdown・人間レビューの考え方に限定する。

## 2. 方針

### 2.1 新Skillを主役にする

新Skill `image2-brand-slides` を追加し、Codex版で「スライドを作って」と言われた場合の主入口にする。

既存 `slide-craft` は補助扱いに下げる。

| Skill | 位置付け | 主な用途 |
|---|---|---|
| `image2-brand-slides` | **Codex版スライド生成の主役** | ロゴ/公式サイト/雰囲気からブランドルールを作り、Image2で1枚ずつスライド画像を生成する |
| `slide-craft` | 補助・旧来用途 | 編集可能PPTXが必要な場合、既存PPTXレビュー、Image2が使えない場合の代替 |
| `slide-quality-check` | 検査 | 生成画像/PPTXの品質検査。必要に応じて新Skillの検査項目を追加する |

### 2.2 出力の主成果物は画像

主成果物はPNG画像群とする。

PPTXは、必要な場合だけ「画像を貼る器」として作る。PPTX内で本文・図表を再編集可能にすることは初期スコープにしない。

### 2.3 ブランドアンカーの入口を広げる

ロゴがある会社だけを前提にしない。

ブランドアンカーの優先順位:

1. ロゴ画像
2. 既存のStyle Markdown / 社内デザインルール
3. 公式サイトURL
4. 既存資料・スクリーンショット
5. ユーザー指定の雰囲気・業種・用途
6. 短い壁打ちで作る暫定ブランド仮説

ロゴがない場合でも止めず、公式サイトまたは壁打ちから `design_system.md` を作る。

### 2.4 工程ごとの役割分担

工程を分けることで、AIへの丸投げを避ける。

| 工程 | 担当 | 役割 |
|---|---|---|
| 構成 | Codex / `image2-brand-slides` | 目的・読者・主張・全体構成・カバレッジを固める |
| 視覚 | Image2 | `design_system.md` とスライド別プロンプトから、1スライド1画像を生成する |
| 補正 | スクリプト | 通常スライドの小タイトル・下線・ロゴだけを揃える |
| 仕上げ | 人間 + 必要に応じて補助Skill | 内容・誤字・社内資料としての違和感を確認し、必要なら再生成またはPPTX化する |

この分担を崩さない。特に、Image2の本文エリアをコードで描き直すことはしない。

- 本文エリアは絶対に補正対象にしない。

## 4. 入力

### 必須

- スライドにしたい原稿、構成案、議事録、メモ、またはMarkdown

### ブランドアンカー

以下のいずれか:

- ロゴ画像
- 公式サイトURL
- 既存資料/PDF/スクリーンショット
- Style Markdown / 社内デザインルール
- ブランドの雰囲気指定
- 壁打ちによる暫定ブランド指定

何もない場合は、3問だけ確認して暫定ブランドルールを作る。

確認する3問:

1. 誰に見せる資料か
2. どんな印象にしたいか
3. 避けたい印象は何か

## 5. ワークフロー

### Step 1: 入力整理

原稿を読み、以下を抽出する。

- デッキタイトル
- 想定読者
- 目的
- 主要メッセージ
- セクション候補
- 表、チェックリスト、注意書き、手順、引用、補足

不足が大きい場合は、スライド生成へ進まず追加確認する。

### Step 2: ブランドアンカー確認

分岐:

```text
ブランドアンカー確認
├─ ロゴあり
│  └─ ロゴから色・形状・雰囲気を抽出
├─ ロゴなし・公式サイトあり
│  └─ 公式サイトから色・余白・言葉遣い・写真/図版傾向を抽出
├─ ロゴなし・Style Markdownあり
│  └─ Style Markdownをデザインシステムの一次入力にする
├─ ロゴなし・既存資料あり
│  └─ 資料から色・構成・トーンを抽出
├─ ロゴなし・雰囲気指定あり
│  └─ 指定語から暫定デザインシステムを作る
└─ 何もなし
   └─ 3問だけ壁打ちして暫定デザインシステムを作る
```

公式サイトを使う場合は、Web検索/閲覧が必要になるため、ユーザーから明示されたURLを優先する。勝手に古い企業情報を推測しない。

公式サイト・既存資料・スクリーンショット・Style Markdownを使った場合は、`input_manifest.md` に以下を残す。

- 入力種別（ロゴ / Style Markdown / 公式サイト / 既存資料 / 雰囲気指定 / 壁打ち）
- 参照URLまたはファイル名
- 取得日・確認日
- 抽出した色（可能ならHEX）
- 抽出したトーン・雰囲気
- 推測と事実の切り分け
- 参照しなかったもの（既存テンプレート、過去出力など）

Style Markdownがある場合:

- `design_system.md` の一次入力として扱う。
- 既存ルールとロゴ/公式サイトの印象が矛盾する場合は、勝手に統合せず差分を記録する。
- 社内資料として避けたい表現、余白、配色、図解傾向が書かれている場合は、プロンプト制約へ反映する。
- Style Markdownが古い・抽象的・不足している場合は、暫定解釈として扱い、必要最小限だけ確認する。

### Step 3: `deck_structure.md` 作成

全体構成を先に作る。

必須項目:

- デッキ目的
- 想定読者
- 全スライド一覧
- 3枚以上の場合、2枚目は目次
- 各スライドのタイトル
- 各スライドの1メッセージ
- 各スライドに割り当てた元情報
- ソースカバレッジマップ
- 初回プレビュー対象

原則:

- 入力にある重要情報を勝手に落とさない
- 情報が多い場合は要約で消さず、スライドを分ける
- 1スライド1メッセージ
- 表示タイトルにはページ番号や `01` / `第1章` を入れない

### Step 4: `design_system.md` 作成

ブランドアンカーから、今回のデッキだけでなく再利用可能なデザインシステムを作る。

必須項目:

- ブランド原則
- ロゴ使用ルール、またはロゴなしルール
- 色の役割
- タイポグラフィ
- 余白・配置
- 情報設計
- 表・チャート方針
- 画像・アイコン・図版方針
- 避ける表現

デフォルトは、シンプルなコンサルティング資料風にする。

### Step 5: `prompts/` 作成

1スライドにつき1プロンプトを作る。

必須制約:

- 1枚の16:9スライド画像だけを生成する
- コンタクトシートを作らない
- 1画像に複数スライドを入れない
- 本文エリアはImage2で生成する
- 本文・図表をSVG、HTML、コード描画で作らない
- 既存テンプレートを模倣しない
- ページ番号を表示しない
- 表示タイトルに連番を入れない
- 日本語テキストはできるだけ正確で読みやすくする

プロンプト作成時は、`deck_structure.md` と `design_system.md` を分けて扱う。

- `deck_structure.md`: 何を、どの順番で、どのスライドに載せるか
- `design_system.md`: どのトーン、余白、配色、文字階層で見せるか

この分離により、構成変更と見た目変更を混ぜない。

任意で、ユーザーが明示的に求めた場合のみ、全体の方向性確認用に「サムネイル一覧/方向性ボード」プロンプトを作ってよい。ただし、これは完成物ではなく、通常フローの1スライド1画像生成を置き換えない。デフォルトでは作らない。

### Step 6: 初回プレビュー生成

フルデッキでも、最初に生成するのは3〜5枚まで。

初回プレビュー対象:

- 1枚目: 表紙
- 2枚目: 目次（3枚以上の場合）
- 3〜5枚目: 主要本文スライド

ここで停止し、ユーザー確認を取る。

確認観点:

- ブランド感は合っているか
- 文字の読みやすさは許容できるか
- 情報量は多すぎないか
- 画像生成の方向性は続行可能か
- 修正したいトーンはあるか

承認されたら、`input_manifest.md` または `deck_structure.md` に `## プレビュー承認記録` を追記する。

記録する項目:

- 承認日
- 承認者（ユーザー本人でよい）
- 承認対象（何枚目までのプレビューか）
- 修正条件（あれば）
- 残りスライド生成へ進んでよいか

この承認記録がない場合、残りスライド生成へ進まない。

### Step 7: 残りスライド生成

初回プレビュー承認後に残りを生成する。

サブエージェントが使える場合は、安定した `deck_structure.md` / `design_system.md` / `prompts/` を渡して並列生成する。

並列生成の境界:

- 1サブエージェントあたり1〜6枚程度
- 各サブエージェントは構成やデザインシステムを変更しない
- 生成物の命名、ヘッダー補正、最終検査はメイン側で行う

失敗時の戻し方:

- スタイルがズレたスライドだけ再生成する。
- `deck_structure.md` と `design_system.md` は固定し、再生成時に勝手に変更しない。
- 再生成では同じプロンプトをベースに、ズレた制約だけを強める。
- 並列生成でばらつきが大きい場合は、残りを小さいバッチに分け直す。
- 元画像は残し、再生成版は別名または上書き前バックアップを残す。

### Step 8: ヘッダー補正

通常スライドが2枚以上ある場合は、`image2-fixed/` を作る。

補正対象:

- 通常スライドの小タイトル
- 小タイトル下の短い下線
- ロゴがある場合は小ロゴ

補正してはいけないもの:

- 本文エリア
- 図表
- 本文テキスト
- ページ番号

ロゴなしの場合:

- ロゴを捏造しない。
- トップ右側は空けるか、ユーザーが指定した短いデッキマークのみ使う。
- 補正は小タイトルと短い下線だけに限定する。
- `titles.json` には `file` と `title` のみを入れ、ページ番号は入れない。

### Step 9: 検査

最低限の検査:

- 各スライドが別ファイルか
- 目次が必要な場合にあるか
- ページ番号が表示されていないか
- タイトルに連番が混ざっていないか
- 通常スライドのタイトル/ロゴ位置が揃っているか
- 入力情報が構成上どこかに割り当てられているか
- 文字が読めるか
- 画像生成特有の誤字がないか
- デザインが過度に装飾的になっていないか

必要に応じて `slide-quality-check` を拡張し、画像スライド用の検査レシートを追加する。

本文エリアに誤字・文字崩れ・意味崩れがある場合:

- ヘッダー補正スクリプトで直さない。
- 該当スライドを再生成する。
- 誤字が繰り返される場合は、該当スライドの本文量を減らす、表現を短くする、または分割する。
- 人間が読めない文字が残る場合、そのスライドは未完成として報告する。

### Step 10: 任意のPPTX化

PPTX化は初期MVPの主成果物ではない。必要な場合だけ、最終PNGを16:9の各スライドへ全面配置する。

初期判断:

- 画像貼り付けPPTXのみ対応する。
- PPTX内の本文・図表を編集可能にすることはしない。
- `pptxgenjs` が使える環境ならNodeで生成する。
- `pptxgenjs` が未導入の場合は、依存追加の承認を取るか、PPTX化を未実施としてPNG群だけ渡す。
- PPTX生成に失敗しても、PNG群が完成していれば画像成果物としては完了扱いにできる。

将来拡張:

- ユーザーが「編集可能なPPTX」を明示的に求める場合のみ、`deck_structure.md`、`design_system.md`、承認済みPNGを参考に、別工程で編集可能PPTXへ再構成する。
- この場合もImage2成果物をそのまま上書きせず、別成果物として扱う。
- 画像生成で確認したデザイン方向を、PPTX側で完全再現できるとは限らないことを事前に説明する。

### Step 11: 人間レビューと微調整

最終成果物を渡す前に、人間レビュー前提の確認レシートを出す。

確認項目:

- 伝えたいメッセージとズレていないか
- 社内/顧客向け資料として違和感がないか
- 表現が過度にカジュアル、装飾的、広告的になっていないか
- 読み手にとって修正したい点が特定しやすいか
- 再生成が必要なスライド番号が明確か

レビュー結果は、必要に応じて `review_notes.md` に残す。

## 6. 出力フォルダ

案件内で使う場合:

```text
{案件ルート}/outputs/image2-brand-slides/{デッキ名}/
├── input_manifest.md
├── deck_structure.md
├── design_system.md
├── prompts/
│   ├── 01-title.md
│   ├── 02-normal.md
│   └── ...
├── image2-original/
│   ├── 01-title.png
│   ├── 02-normal.png
│   └── ...
├── image2-fixed/
│   ├── 01-title.png
│   ├── 02-normal.png
│   └── ...
├── titles.json
├── review_notes.md
└── export/
    └── {デッキ名}.pptx   # 必要な場合のみ
```

生成画像の保存ルール:

- Image2が返した画像は、まず `image2-original/` に安定ファイル名で保存する。
- ファイル名は `01-title.png`, `02-normal.png` のように2桁連番にする。
- 生成ツールが別の保存先を返す場合は、元ファイルを残したままコピーする。
- 画像をローカル保存できない場合は、作業を止めて保存不能を報告する。保存できないまま `image2-fixed/` やPPTX化へ進まない。
- 再生成した場合は、旧版を `image2-original/_rejected/` などに退避するか、ファイル名に `_v2` を付けて履歴を残す。

案件外のテストでは `30_Flow/テスト/Skillテスト_image2-brand-slides/` を使う。

正典成果物にする場合は、画像群やPPTXをフェーズフォルダへ移す前にユーザー確認を取る。

## 7. 既存 `slide-craft` との住み分け

Codexでスライド生成を頼まれた場合、原則は `image2-brand-slides` を提案する。

`slide-craft` を使う条件:

- 編集可能なPPTXが必須
- Image2/画像生成が使えず、ユーザーが代替生成を承認した
- 既存PPTXのレビューやコード修正が主目的
- ユーザーが明示的に `slide-craft` を指定した

`slide-craft` 側には、将来的に「CodexではImage2方式を推奨。PPTXコード生成は補助」と追記する候補がある。

Image2が使えない場合は自動で `slide-craft` にフォールバックしない。まず止めて、以下を報告する。

- 画像生成ツールが使えない理由
- PNG生成なしで進める場合の品質差
- `slide-craft` / PPTXコード生成へ切り替えるかどうかの確認
