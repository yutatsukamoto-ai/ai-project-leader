# GitHub公開リポジトリ 対応指示書

起点: Codexによるセキュリティ/配布レビュー（2026-06-17）
状態: **public repoとしてpush済み。最新ツリーの脱識別対象は2026-06-17時点の実態に更新済み。**
含意: 実社名がGitHub履歴に入っている可能性は残る。最新コミットの修正だけでは`git log`や過去コミットのdiffから辿れる。

---

## §0. 履歴の扱いを決める（先決事項）

実社名（クライアントA・クライアントBの実名）がpush済みpublic repoの履歴に入っている可能性がある。選択肢は3つ。

| 方式 | 作業 | リスク |
|------|------|--------|
| A. 最新コミットだけ修正 | 該当箇所を置換してcommit & push | 過去コミットを掘れば実社名が見える。ただしカジュアルな閲覧では気づかない |
| B. git filter-repo で履歴書き換え | 全履歴から文字列を除去、force push | 完全除去。ただしfork/cloneした人の手元には残る。SHA全変更 |
| C. リポジトリ削除→新規作成 | 現リポジトリを削除し、クリーンな履歴で新規push | 最も確実。star/fork/リンクは切れる |

推奨: **A方式で十分な可能性が高い**。理由は以下。
- クライアントA・クライアントBの実名は`.gitignore`やtree図など「インフラ系ファイル」に出ていた。実案件フォルダ自体（中身の成果物）はGit管理外
- つまり漏れているのは「社名」であって「案件データ」ではない
- 守秘契約上「社名が出ること自体がNG」ならB/Cが必要。「社名＋内部データのセットが問題」ならAで実害はほぼない

**判断基準**: その会社との守秘契約で「社名の公開」自体が禁止されているか？
- Yes → B or C
- No / 不明だが実データは出ていない → A で進め、念のため先方に一報
