# 案件切り替えセーフティ仕様案

作成: 2026-06-20
状態: 最小実装済み（Hook強制は未実装）
対象: W-04（セッション中の案件切り替え混在）

## 位置づけ

複数案件を同じAIプロジェクトリーダー基盤で扱うとき、セッション中に案件Aから案件Bへ話題が移り、成果物や `project-context.md` の書き込み先が混ざるリスクを防ぐための仕様案。

本書は設計メモ兼実装メモ。2026-06-20に最小実装として `30_Flow/現在の作業セッション.md`、`AGENTS.md` / `CODEX.md` / `30_Flow/README.md` の運用ルール、`.gitignore` の除外を追加した。各Skill本文への反映とHook実装はまだ行わない。

## 背景

- 案件フォルダ分離、案件インデックス、相対プレースホルダ方式により、複数案件を独立して動かす構造はある。
- ただし、`案件インデックス.md` は「案件を探す」には効くが、「書き込み先を間違えない」ガードにはならない。
- 特に、既存案件の立ち上げフェーズと新案件の前段チェーンが同じセッションで並走すると、成果物保存先や `project-context.md` 書き戻し先の取り違えが起きやすい。

## 発動条件

次のいずれかが起きる直前に、本仕様の運用を必ず確認する。

1. 実案件または模擬案件の2案件目を、既存の活動中案件と並行して開始する。
2. 活動中案件の作業中に、別案件について成果物作成・ファイル編集・`project-context.md` 書き戻しを伴う依頼が来る。
3. 一時停止した案件へ戻り、フェーズ成果物の作成または更新を再開する。

読み取りだけのステータス確認や、方針相談のみの場合は、案件切り替えを確定しない。ただし回答冒頭で「読み取りのみで、アクティブ案件は変更しない」と明示する。

## 最小仕様

### 1. 作業セッション台帳

`30_Flow/現在の作業セッション.md` を置き、現在の作業対象を1箇所に集約する。実案件名や現在地を含み得るため、`.gitignore` でGit/配布対象外にする。

想定フィールド:

```md
# 現在の作業セッション

最終更新: YYYY-MM-DD

## アクティブ案件

- 案件名:
- 案件ルート:
- フェーズ:
- 次の一手:
- 書き戻し先:

## 一時停止中の案件

| 案件名 | 案件ルート | 停止位置 | 再開条件 |
|---|---|---|---|

## 切り替えログ

| 日時 | from | to | 理由 | 確認 |
|---|---|---|---|---|
```

### 2. 切り替え宣言

案件を切り替えるときは、作業前に短く宣言する。

定型:

```text
案件を切り替えます。
現在: {旧案件名} / {旧フェーズ}
次: {新案件名} / {新フェーズ}
以後の書き込み先: {新案件ルート}
```

旧案件が途中の場合は、一時停止中の案件に停止位置と再開条件を残す。

### 3. 書き込み直前確認

成果物作成、成果物更新、`project-context.md` 書き戻しの直前に、次を確認する。

```text
アクティブ案件: {台帳の案件ルート}
保存先: {これから書くパス}
判定: 一致 / 不一致
```

不一致の場合は書き込まず、ユーザーへ確認する。

## 環境別の扱い

| 環境 | 当面の扱い | 将来の強制 |
|---|---|---|
| Claude Cowork | メモリーと作業セッション台帳による運用ガード | Hook強制は前提にしない |
| Codex | 作業前の読み取りと書き込み直前確認を運用で実施 | 必要ならローカル検査スクリプト化 |
| Claude Code | 同じ台帳を参照 | PreToolUse Hookで `30_Flow/` 書き込み先を強制検査できる |

## 実装時の波及先

実装するときは、以下を同時に確認する。

- `AGENTS.md`: 実案件名で呼ばれたときの読み取り手順に、アクティブ案件確認を追加する。→ 最小実装済み。
- `CODEX.md`: Codex作業時の書き込み直前確認を追加する。→ 最小実装済み。
- `30_Flow/README.md`: `現在の作業セッション.md` の位置づけを説明する。→ 最小実装済み。
- `.gitignore`: `30_Flow/現在の作業セッション.md` を除外する。→ 最小実装済み。
- `20_Skills/*/*-chain/SKILL.md`: 親chainの開始時に作業セッション台帳を読む手順を追加するか。→ 未実装。2案件目の実運用で必要なら反映。
- Claude Code Hook: `30_Flow/` 配下への書き込み前に、保存先がアクティブ案件配下か検査するか。→ 未実装。PreToolUse Hook化候補。

## 採用しないこと

- 案件インデックスだけを強化して済ませる。
- すべてのステータス確認で案件切り替えを発生させる。
- 終結済み案件をメモリーに残し続ける。
- ユーザー確認なしに活動中案件を自動で切り替える。

## 完了条件

実装時は次を満たしたら完了とする。

1. `30_Flow/現在の作業セッション.md` が作成され、アクティブ案件と一時停止中案件を区別できる。→ 完了。
2. 案件切り替え時の宣言文が `AGENTS.md` または `CODEX.md` に明記されている。→ 完了。
3. 成果物書き込み直前に、アクティブ案件ルートと保存先の一致を確認する手順がある。→ 完了。
4. Claude Code版では、可能ならPreToolUse Hook化の要否を判断している。→ 候補として記録。未実装。
5. 実装後に `bash _tools/build.sh --verify` と `bash _tools/build.sh --check` が通っている。→ 実装後に確認する。
