# 再点検: 現状の弱点と穴（2026-06-20）

目的: AIプロジェクトリーダーの強み（再現性・停止ゲート・横展開・配布・学習還流）を前提に、実運用で次に壊れやすい箇所を再点検した記録。

本書は実装指示ではなく、後続作業の論点リスト。修正は別途ユーザー確認のうえ行う。

---

## 1. 今回の見立て

弱点の多くは「基盤の品質が低い」からではなく、短期間で強くなった結果、守る対象が増えたことから起きている。

特に次の5系統が壊れやすい。

| 系統 | 何が起きるか | 代表例 |
|---|---|---|
| 正典の鮮度差 | 実体は進んだが、骨子・索引・READMEの数値や方針が古くなる | 実体37 Skillに対し、骨子に31 Skill時点の記述が残る |
| 検査対象外 | `build.sh --verify` が緑でも、別種の危険物が残る | `tmp/` 配下の追跡済み中間画像 |
| Flowの混在 | 正典成果物・中間物・生成物・教材が同じ層に混ざる | `30_Flow/テスト/` の画像生成出力やPDF/画像 |
| ログのばらつき | project-context / chain-trace / 検査ログの構造が案件ごとに違う | chain-traceは丸山案件のみ |
| 環境分岐 | Cowork / Claude Code / Codex の役割差が文書・CIで非対称になる | Codex配布smokeは手動、Claude Code配布smokeはCIあり |

---

## 2. 現時点で見つかった弱点・穴

### H-01. Git追跡禁止パスの検査が弱い

`tmp/` は `.gitignore` に入っているが、既にGit追跡下のファイルが266件ある。

確認結果:

- `git ls-files tmp | wc -l` → 266
- `du -sh tmp` → 約19MB
- `bash _tools/build.sh --release-check` はPASS

含意:

- `.gitignore` は新規追加を防ぐだけで、既に追跡済みのファイルは防げない。
- 現行の `_tools/_lib.sh` の `check_forbidden_tracked_paths()` は `tmp/` を禁止対象にしていない。
- 「検査が緑でも、履歴に入れたくない再生成物が残る」状態。

2026-06-20対応:

- `check_forbidden_tracked_paths()` に `tmp/` / `_dist/` / `30_Flow/実案件/` / `30_Flow/現在の作業セッション.md` などを追加。
- 追跡済み `tmp/` 266件は `git rm --cached -r tmp` でGit管理から外した。ローカルファイル自体は削除していない。
- `bash _tools/build.sh --verify` で「Git追跡禁止パスなし」まで確認済み。

### H-02. `30_Flow/` 配下の生成バイナリが追跡されている

`30_Flow/` 配下で画像・PDF・docx・pptx・xlsxなどのバイナリ成果物が一部Git追跡下にある。

2026-06-20対応:

- `30_Flow/README.md` にGit追跡ルールを追加し、`export/` / `final-slides/` / `_generated/` / `_renders/` / `_render/` / `_quicklook/` / `screenshots/` を原則Git追跡不可とした。
- `_tools/_lib.sh` の追跡禁止検査へ同パターンを追加。
- 追跡済み37件は `git rm --cached --ignore-unmatch` でGit管理から外した。ローカルファイル自体は削除していない。
- `outputs/`配下のMarkdown観察メモ・検証メモは一律削除対象にせず、必要な代表だけ残す方針とした。

確認結果:

- `git ls-files '30_Flow/**/*.png' ...` → 44件

含意:

- 模擬案件やテストデータなので即座に実案件漏洩とは言えない。
- ただし、再生成可能物・中間生成物・正典成果物の境界がGitレベルでは曖昧。
- 配布・公開・リポジトリ肥大化のリスクが残る。

### H-03. `project-context.md` の構造が案件ごとに揺れている

確認した構造の例:

- 初期テンプレ型: 基本情報 / 制約条件 / 関係者 / 決定事項ログ
- 実案件型: 現在地サマリー / 制約・前提 / 重要仮説・リスク / 成果物ポインタ / 次の一手 / 変更ログ
- 丸山型: 基本情報 / 現在のフェーズ / 主要文書ポインタ / 書き戻しログ
- 山藤型: 確定事実 / 未確定・申し送り / テーラリング判断ログ / 主要文書 / 変更ログ

含意:

- 人間が読む分には問題ない。
- AIやツールが横断的に「現在地」「成果物ポインタ」「未確定事項」「テーラリング判断」を拾うには弱い。
- D-05として標準構造の設計が必要。

### H-04. chain-trace / 検査ログが横断分析できる母数に育っていない

確認結果:

- `chain-trace.json` は丸山案件のみ。
- `_検査ログ.md` は丸山案件と一部テスト案件のみ。

含意:

- 仕組みとしては良いが、学習還流・横断分析のデータソースとしてはまだ薄い。
- 過去案件に遡及する必要は薄いが、今後の本線案件では標準化したい。

### H-05. README / 骨子 / 決定の索引の鮮度差

実体:

- `20_Skills` の `SKILL.md` は37件。
- 回帰evalは50件。

鮮度差:

- `プロジェクト骨子.md` は31 Skill / ゴールデン37件時点の記述が残る。
- `決定の索引.md` にはchainの.skill化方針について古い表現が残る。

含意:

- 実行にはすぐ影響しない。
- 初見ユーザーや後続のAIが「どれが現在地か」を誤解する。
- 手書き数値を減らすか、実数との差分検査を追加したい。

### H-06. evalカバーが成果物Skill中心で、横断・スライド系は薄い

確認結果:

- 37 Skill中、機械eval対象は25種。
- 未カバーは主にchain、メタSkill、`grill-me`、`ichi-kakunin`、`image2-brand-slides`、`slide-quality-check`、`skill-kabeuchi-kickoff`、雛形。

含意:

- 成果物Skill中心のevalとしては自然。
- ただしCodex版で重要な `image2-brand-slides` と、品質担保の `slide-quality-check` は最低限のsmokeがあってもよい。
- すべてを同じ強度でevalする必要はない。

### H-07. Codex配布のCI確認が手動寄り

確認結果:

- README / AGENTS / CODEXには `package-dist.sh --target codex` → `test-dist-codex.sh` の手順あり。
- GitHub ActionsではClaude Code配布 smoke はあるが、Codex配布 smoke はない。

含意:

- Codexを併用・入口として使うなら、Codex配布もCIで見る方が自然。
- Claude Code側だけ強くなると、Codex配布版の事故に気づきにくい。

### H-08. スライド生成機能が複数環境にまたがり、責務が重なりやすい

現状:

- Codexでは新規スライド生成は `image2-brand-slides` が主ルート。
- `slide-craft` は編集可能PPTXや既存PPTXレビューの補助。
- Cowork / Claude Code / Codex のどこで「考える」「作る」「検査する」を担うかが、今後さらに重要になる。

含意:

- Claude側で生成機能まで厚く持つと、Codex側と二重化する。
- スライドは「設計・レビュー」と「生成」を分けた方が運用が安定する。

## 3. 既存の強みを踏まえた解釈

今回の穴は、以下の強みの副作用でもある。

| 強み | 副作用 |
|---|---|
| Skill / chain / 横断GLが増えた | 正典・索引の鮮度差が出やすい |
| build / eval / hookが強くなった | 検査対象外に穴が移る |
| Flowに模擬・テスト・実案件を集めた | 正典/中間/生成物が混ざりやすい |
| 伴走→写し取り方式 | 実運用で育った型がテンプレへ戻るまで時差が出る |
| Codex / Claude Code / Cowork併用 | 環境ごとの責務分離が必要になる |

したがって、次の改善は「機能を増やす」より「守る対象を決める」ことが中心になる。
