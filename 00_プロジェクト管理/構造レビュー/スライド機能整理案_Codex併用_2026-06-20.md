# スライド機能整理案（Codex併用前提 / 2026-06-20）

目的: Codexを併用する前提で、AIプロジェクトリーダーにおけるスライド機能の責務を整理し、重複や過剰維持を減らす。

本書は提案。実装・Skill削除・README更新は別途ユーザー確認後に行う。

---

## 1. 結論

スライド機能は「なくす」より、環境ごとに責務を分けるのがよい。

推奨:

| 環境 | 主担当 |
|---|---|
| Claude Cowork | スライドの意味設計・構成案・メッセージ整理・壁打ち |
| Claude Code | 既存PPTXレビュー、軽い修正、Hooks/CIを含む開発寄り検証 |
| Codex | 新規スライド生成、Image2生成、HTMLデッキ化、ファイル生成補助 |

原則:

> Claude側は「スライドを考える」。  
> Codex側は「スライドを作る」。  
> どちらにも同じ生成機能を厚く持たせない。

---

## 2. なぜ整理するか

AIプロジェクトリーダーの本線は、PM案件を入口から終結まで進めること。

スライドは重要だが、本線の中核は以下である。

- 何を伝えるか
- どの順で合意を取るか
- どの成果物から資料化するか
- どの論点を人間が判断するか

したがって、スライド生成機能を全環境で厚く持つ必要はない。

Codexが使えるなら、生成・ファイル化・HTML化はCodexに寄せた方がよい。

---

## 3. 残すもの

Claude側にも残すべき機能:

| 機能 | 理由 |
|---|---|
| スライド構成案作成 | PM文脈・合意形成・論点整理に近い |
| 1枚ごとの主張・根拠・図解方針 | 成果物からスライドへ変換する設計部分 |
| 提案書/報告書のスライド向け要約 | 案件成果物との接続が重要 |
| 既存スライドレビュー | 生成よりレビューの方がClaude側でも有効 |
| slide-quality-check観点 | 生成物の検査基準として再利用できる |

残すべきSkill:

| Skill | 推奨位置づけ |
|---|---|
| `image2-brand-slides` | Codex版の新規スライド生成主ルート |
| `slide-quality-check` | スライド品質検査の観点。生成環境を問わず使える |
| `slide-craft` | 既存PPTXレビュー、編集可能PPTXが明示された場合の補助 |

---

## 4. 減らしてよいもの

減らしても実運用上の問題は小さい。

| 減らす対象 | 理由 |
|---|---|
| Claude側でPPTX/HTMLスライドを直接生成する主導線 | Codexと二重化し、運用がぶれる |
| 複数の新規スライド生成Skillの並列運用 | 入口が増えるほど選択ミスが起きる |
| 旧Skill名・旧ルート | 正典参照や起動文が古くなる原因 |
| 大量のサンプルPPTX/画像のGit管理 | 再生成可能物でリポジトリが重くなる |
| 生成済みスライドをFlowへ残し続ける運用 | 正典成果物と生成物が混ざる |

---

## 5. 推奨する責務分離

### 5.1 新規スライド生成

標準ルート:

1. Cowork/Claude/Codexのどれでも、まず意味設計を作る。
2. Codexでは `image2-brand-slides` の流れへ進める。
3. `message_design.md` → `deck_structure.md` → `design_system.md` → `prompts/` → Image2生成 → HTMLデッキ化。
4. 最初の3〜5枚で確認を取る。

### 5.2 編集可能PPTXが必要な場合

`slide-craft` を使う。

条件:

- ユーザーが「編集可能PPTX」「PowerPointで編集できる形」と明示した場合。
- 既存PPTXのレビュー・修正が主目的の場合。
- Image2/HTMLではなくPPTX納品が必要な場合。

### 5.3 既存スライドレビュー

`slide-quality-check` または `slide-craft` のレビュー観点を使う。

レビュー対象:

- 主張が1枚1メッセージになっているか。
- 情報量が過多でないか。
- 配色・強調・余白が破綻していないか。
- 提案書/計画書の内容と矛盾していないか。

---

## 6. 変更候補

### S-01. README / AGENTS / CODEX の整理

明記する内容:

- 新規スライド生成は原則 `image2-brand-slides`。
- `slide-craft` は編集可能PPTXまたは既存PPTXレビューの補助。
- Claude側はスライド設計・レビュー、Codex側は生成・ファイル化を主担当にする。

現状、AGENTS / CODEXには近い方針が既にある。READMEや成果物マップの説明も必要なら同期する。

### S-02. 生成物の保存ルール

Flow内で生成物が散らばらないようにする。

候補:

```text
{案件ルート}/outputs/slides/
{案件ルート}/outputs/renders/
{案件ルート}/outputs/image2/
```

またはスライドSkill専用に:

```text
{案件ルート}/outputs/image2-brand-slides/{deck-name}/
```

方針:

- 正典の提案書・構成案はフェーズフォルダ。
- 生成画像・HTML・PDFは `outputs/` または `_generated/`。
- Git追跡は原則しない。代表サンプルだけ例外。

### S-03. slide系evalの軽量化

全てを機械evalに載せない。

最低限:

- `image2-brand-slides`: 生成フォルダ構造のsmoke
- `slide-quality-check`: サンプルスライドに対するレビュー項目の存在確認
- `slide-craft`: 既存の構成案ゴールデン1件を維持

### S-04. 旧資産の扱い

古いスライド生成Skill名や旧ルートは、正典からは除去する。

ただし、履歴文書に残る旧名は原則放置でよい。

---

## 7. 減らす判断の安全性

減らしても問題が小さい理由:

- PM案件の本線は成果物・合意形成・判断ログであり、スライド生成そのものではない。
- スライドの「意味設計」は残す。
- 生成・ファイル化はCodexが得意。
- 既存PPTXレビューや編集可能PPTXの逃げ道として `slide-craft` を残す。
- 品質観点は `slide-quality-check` と横断ガイドラインに残る。

減らしすぎると困るもの:

- 提案書からスライド構成へ変換する考え方。
- スライドレビュー観点。
- 編集可能PPTXが明示された場合の対応ルート。

---

## 8. 推奨実行順

1. まず運用方針として「Claude側は考える、Codex側は作る」を採用する。
2. README / 成果物マップ / AGENTS / CODEX の記述を同期する。
3. `slide-craft` を主役から補助へ降格する。
4. スライド生成物の保存先を `outputs/` 配下へ寄せる。
5. サンプルPPTX/画像の追跡方針を決める。
6. 必要なら `image2-brand-slides` のsmokeを追加する。
