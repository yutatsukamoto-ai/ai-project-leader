# 対策設計 (2026-06-20)

原因分析の3つの構造的パターンに対する再発防止策。
既存の設計思想（構造ファースト・build.shが唯一の入口・可逆性軸）に沿って設計する。

---

## 設計方針

対策は3段に分ける。

1. **検査の拡張**（build.sh --verify に足す）: 機械で検知できるものは機械で。最もテコが効く。
2. **手順の追加**（心得・チェックリストに足す）: 機械化が割に合わないものは手順で。既存の「規約を新設するときの心得」を拡張する。
3. **運用の習慣**（Claudeの能動サジェストに載せる）: 節目でClaudeが声をかける。Cowork版はメモリー駆動、CC版はHook駆動。

---

## パターン1: 改名・新設の波及範囲が追跡されない

該当: H-01（旧Skill名残存）, H-02（選定ガイド未反映）, H-04（ルール遡及漏れ）

### 対策A: Skill名の幽霊参照チェック（検査の拡張）

build.sh --verify に `check_skill_name_ghosts` を追加する。

ロジック:
1. `20_Skills/` 配下の全ディレクトリ名を「正規Skill名一覧」として取得
2. 主要文書（横断GL全ファイル・テンプレート全ファイル・プロジェクト骨子.md・成果物マップ.md）を全文Grep
3. Skill名っぽいパターン（ハイフン区切りの英小文字）で抽出し、正規一覧に存在しないものを警告

効果: H-01（旧Skill名が横断GLに残る）を機械検知できる。今回のtsutawaru-slide残存は全てこれで捕まる。

実装コスト: 低（grepベース、20行程度）。

### 対策B: 心得の拡張 — 変更種別ごとのチェックリスト（手順の追加）

既存の「規約・枠を新設するときの心得」（3点）を拡張し、**変更種別ごとの波及先チェックリスト**を追記する。

追記内容:

```
## 変更種別ごとの波及チェック（追記 2026-06-20）

変更を行ったとき、以下の波及先を確認する。

| 変更の種類 | 確認する波及先 |
|---|---|
| Skill改名・統合・削除 | ①成果物マップ ②出力フォーマット規約(A節) ③成果物選定ガイド ④プロジェクト骨子 ⑤全文Grep（build --verify が検査） |
| Skill新規追加 | ①成果物マップ ②成果物選定ガイド（ランク付け） ③親chainのSKILL.md ④出力フォーマット規約(A節・該当フォーマットの行) |
| 横断GLルール新設 | ①適用対象のテンプレート（40_Stock/テンプレート/） ②適用対象のreferences ③横断GL README索引表 |
| テンプレート修正 | ①元になった横断GLルールとの整合 |
```

効果: H-02（選定ガイド未反映）とH-04（テンプレート遡及漏れ）を手順で防ぐ。対策Aの機械検知と組み合わせて二重防御。

### 対策C: 能動サジェスト — Skill変更時の波及確認（運用の習慣）

Claudeが Skill の改名・追加・統合を行った直後に、対策Bのチェックリストを提示して確認を促す。

発動条件: 20_Skills/ 配下のディレクトリ名に変更があったとき。
実装: Cowork版はメモリーに記録（本対策設計自体がメモリーに載る）。CC版はauto-build hookの延長で検知可能。

## パターン2: 同一情報の複数所在（SSOT不徹底）

該当: H-01, H-02（出力フォーマット規約と成果物選定ガイドが成果物マップと独立にSkill一覧を持つ）

### 対策D: 選定ガイドと成果物マップの整合チェック（検査の拡張）

build.sh --verify に `check_selection_guide_coverage` を追加する。

ロジック:
1. 成果物マップからフェーズ別のSkill名一覧を抽出
2. 成果物選定ガイドのテーブルからSkill名一覧を抽出
3. 差分を警告

効果: H-02（選定ガイドにSkill漏れ）を機械検知できる。新Skill追加時に選定ガイドへの反映漏れが自動で見つかる。

実装コスト: 中（テーブルのパースが要るが、既存のcheck_map_consistencyのパターンを流用可能）。

### 対策E: SSOT参照への書き換え（構造の改善）— 保留

出力フォーマット規約(A節)と選定ガイドの独自テーブルを廃止し、「成果物マップを参照」に書き換える案。

判断: **現時点では保留**。理由は2つ。(1) 各テーブルは成果物マップとは異なる軸の情報を持っている（出力フォーマット規約は「形式の出し分け」、選定ガイドは「テーラリング判断のランク付け」）。単純に「マップを見よ」にはできない。(2) 対策A+Dの機械検知で同期漏れは捕まるので、SSOT化の緊急度は低い。

## パターン3: 一回限りの大規模操作が日常検査のスコープ外

該当: H-03（フォルダ移行で孤児化）, H-05（.gitignore漏れ）

### 対策F: .gitignoreの防御的パターン追加（即時対処）

Codex指示書のH-05で `tmp/` を追加済み。加えて、一般的な一時ファイルパターンを予防的に追加する。

```
# 一般的な一時ファイル
tmp/
*.tmp
.DS_Store
```

※ `.DS_Store` は既存の `.gitignore` に含まれているか確認の上。

### 対策G: 大規模操作時のチェックリストテンプレート（手順の追加）

「規約・枠を新設するときの心得」に並べて、「大規模操作（移行・統合・リネーム）の心得」を新設する。

内容:

```
## 大規模操作（移行・統合・構造変更）の心得

1. 影響範囲の棚卸し — 移行対象のファイル一覧を作り、「案件紐づき」と「案件非紐づき（テンプレート・規約・参照先）」に分類する。
2. 移行後の差分確認 — 移行元ディレクトリに残ったファイルがないか確認する。
3. 参照パスの更新 — メモリー・懸念マスター・横断GLの中で旧パスを参照している箇所をGrepで洗い出し更新する。
```

効果: H-03（フォルダ移行での孤児化）の再発防止。頻度は低いが不可逆度が高い操作なのでチェックリスト化する価値がある。

### 対策H: 定期監査の習慣化（運用の習慣）— 発動条件付き

今回の監査のような全体チェックを定期的に行う案。

判断: **即時の定期化はしない**。理由は (1) 対策A+Dの機械検知が入れば日常的な漏れは捕まる、(2) 今回の監査は「プロジェクトが一段落したタイミング」で行ったのが適切。

発動条件: **フェーズ完了（実案件の立ち上げ完了、計画完了など）のタイミングで、Claudeから「前回の監査から変更が多いので全体チェックしますか？」と能動提案する。** 既存の「節目でコミット提案」と同じパターン。

## 対策の優先順位

| 対策 | 種別 | 効果 | コスト | 優先 |
|---|---|---|---|---|
| A. Skill名幽霊参照チェック | 検査拡張 | H-01完全防止 | 低 | **即時** |
| B. 心得に波及チェックリスト追記 | 手順追加 | H-02,H-04防止 | 極低 | **即時** |
| C. Skill変更時の能動サジェスト | 運用 | A+Bの発動支援 | 極低 | **即時** |
| D. 選定ガイド/マップ整合チェック | 検査拡張 | H-02完全防止 | 中 | 次回build.sh改修時 |
| E. SSOT参照への書き換え | 構造改善 | 根本解消 | 高 | 保留 |
| F. .gitignore防御パターン | 即時対処 | H-05防止 | 極低 | **即時**（Codex指示書に含む） |
| G. 大規模操作の心得 | 手順追加 | H-03防止 | 低 | **即時** |
| H. 定期監査の能動サジェスト | 運用 | 全般 | 極低 | 発動条件付き |

即時対処: A, B, C, F, G（全て低コスト）
次回改修時: D
保留: E, H
