# 再発防止設計案（2026-06-20）

目的: 再点検で見つかった穴を、今後起きにくくするための設計方針を残す。

本書は提案。実装はユーザー確認後に行う。

---

## 1. 基本方針

すべてを同じ強度で守らない。

AIプロジェクトリーダーは、すでにSkill・chain・横断GL・Flow・Stock・配布・Hooks・evalを持つ。全部を厳密に維持しようとすると、運用コストが本線案件を圧迫する。

したがって、守る対象をA/B/Cに分ける。

| ランク | 対象 | 守り方 |
|---|---|---|
| A | 壊れると案件運用・配布・実データ保護が危ないもの | 機械検査で強く守る |
| B | 鮮度差が溜まると誤解・学習還流低下につながるもの | 節目棚卸しで守る |
| C | 実験・補助・表現系。壊れても本線は止まらないもの | smokeまたは手動確認で十分 |

---

## 2. Aランク: 機械検査で守るもの

### A-1. 追跡禁止レジストリ

`.gitignore` とは別に、Git追跡下にあってはいけないパスを `_tools/_lib.sh` の `check_forbidden_tracked_paths()` で検査する。

候補:

| 対象 | 理由 |
|---|---|
| `tmp/` | PDF変換・画像生成などの中間物。再生成可能 |
| `_dist/` | 配布生成物。再生成可能 |
| `_backups/` | 大容量スナップショット。履歴不要 |
| `30_Flow/実案件/` | 実案件データ。Git/配布除外 |
| `30_Flow/現在の作業セッション.md` | 実案件名・現在地を含み得るローカルSSOT |
| `*.skill` | buildで再生成可能 |
| `*.bak` | 一時バックアップ |

追加検討:

- `30_Flow/**/outputs/` の追跡禁止化
- `30_Flow/テスト/**/export/` や `final-slides/` の追跡禁止化
- バイナリ拡張子の追跡を警告にする（docx/pptx/xlsx/png/jpg/pdf）

判断:

- 模擬案件の成果物バイナリを教材として残す場合は例外リストが必要。
- まずは `tmp/` と `_dist/` を強く検査するのが低リスク。

### A-2. Flow内の正典/中間/生成物ルール

`30_Flow/README.md` のルールを機械検査へ落とす。

推奨整理:

| 置き場 | 役割 | Git追跡 |
|---|---|---|
| フェーズフォルダ | 正典成果物。次Skillの入力になるもの | 原則可（実案件は除外） |
| `_素材/` | 受領資料・raw | 実案件は不可。模擬/テストは要判断 |
| `outputs/` | 中間生成物 | 原則不可または警告 |
| `_generated/` / `_renders/` | 画像・PDF・HTML生成物 | 原則不可 |
| `_検査ログ.md` | 内部検査の証跡 | 可 |
| `chain-trace.json` | 実行ログ | 可 |

これにより、画像生成やPDF変換の出力が正典成果物と混ざりにくくなる。

### A-3. `project-context.md` 標準構造

D-05として、育った実運用の形を標準化する。

推奨セクション:

1. 基本情報
2. 現在地サマリー
3. 制約・前提
4. 重要仮説・リスク
5. 成果物ポインタ
6. 決定事項ログ
7. テーラリング判断ログ
8. 書き戻しログ / 変更ログ
9. 未確定事項・申し送り
10. 次の一手

ルール:

- 必須セクションと任意セクションを分ける。
- 既存案件には無理に遡及しない。
- 新規案件と3案件目実証から適用する。
- 初期テンプレは薄く、運用後に育てるための空欄を持つ。

### A-4. Codex配布smokeのCI化

現在のCIはClaude Code配布smokeを持つが、Codex配布smokeは手動。

Codexを併用する前提なら、CIに以下を足す。

```bash
bash _tools/package-dist.sh --target codex
bash _tools/test-dist-codex.sh
```

注意:

- `_dist/` を上書きするため、Claude Code配布smokeとの順番を分ける。
- 可能ならジョブを分ける。

## 3. Bランク: 棚卸しで守るもの

### B-1. ドキュメント鮮度

README / 骨子 / 決定の索引の数値は古くなりやすい。

対策候補:

1. 手書き数値を減らす。
2. 「詳細は成果物マップ参照」に寄せる。
3. 数値を残す場合は `build.sh --verify` で実数との差分を警告する。

対象:

- Skill数
- chain数
- ゴールデン件数
- 模擬完走件数
- `.skill` パッケージ件数

### B-2. chain-trace / 検査ログの運用

目的は「全過去案件を完璧に揃える」ことではなく、「今後の本線案件から学習可能なログを残す」こと。

方針:

- 既存案件への遡及はしない。
- 新規本線案件では `chain-trace.json` と `_検査ログ.md` を標準運用にする。
- 3案件目実証で、どの粒度が実用的かを見る。

### B-3. 案件教訓のタグ

案件教訓は現状4件。まだ検索設計を厚くするには早い。

軽い追加で十分:

```md
- 業種:
- 課題タイプ:
- 案件タイプ:
- 参照した成果物:
```

発動条件:

- 案件教訓が8件以上溜まったら、タグ体系と参照ログを設計する。

## 4. Cランク: 減らしてよいもの・軽く守るもの

### C-1. 全Skill同一強度のeval

不要。

成果物Skillは機械evalと目視層で守る。
chain / メタSkill / 思考道具は、成果物トークン検査に向かない。

代替:

- chain: 入出力・停止ポイントの静的検査
- grill-me / ichi-kakunin: サンプル対話または起動条件smoke
- slide系: 代表ケースのsmoke

### C-2. サンプルバイナリの大量保持

不要。

代表だけ残し、生成物は再生成可能に寄せる。

候補:

- PPTX/docx/xlsx/pngは最小代表だけ残す。
- 大量のレンダー画像やPDF変換中間物はGit追跡しない。

### C-3. 旧履歴文書の全更新

不要。

履歴は当時の記録として残す。正典ファイルだけ最新ならよい。

例:

- 過去監査結果に旧Skill名が残るのは履歴として許容。
- ただしREADME、骨子、成果物マップ、横断GL、懸念マスター、決定の索引は最新化対象。

## 5. 推奨実行順

1. ✅ `check_forbidden_tracked_paths()` に `tmp/` と `_dist/` を追加する。
2. ✅ 追跡済み `tmp/` 266件をGit管理から外す。
3. ✅ `project-context.md` 標準構造案を作る。
4. ✅ Flow内の `_generated` / `_renders` / `export` / `final-slides` ルールを決める。
5. ✅ Codex配布smokeをCIへ追加する。
6. ✅ README / 骨子 / 決定の索引の鮮度を棚卸しする。
7. 3案件目実証で `project-context` / `chain-trace` / `_検査ログ` の標準運用を試す。

---

## 6. やらない方がよいこと

- 既存の全模擬案件へ `chain-trace.json` を遡及生成する。
- すべてのSkillを同じ形式のevalに載せる。
- 旧履歴文書の古い表現を全件修正する。
- スライド生成機能をCowork / Claude Code / Codexすべてで厚く持つ。
- WBS Viewer本体の同梱を、配布方針が固まる前に急ぐ。
