# Claude Code版を作るときの覚書 ── モデル堅牢性・取りこぼし防止

作成: 2026-06-14 ／ 目的: 「Opus/Sonnet級の別モデルでも見落とさない」状態を、Claude Code版を作るときに**取りこぼさず**実装するための覚書。
位置づけ: 設計判断の凍結メモ。Claude Code版に着手したら、まず本書のチェックリスト（§3）から拾う。

> 注意: Claude Codeの正確なHook種別・設定キー・slash/subagentの記法は更新されうる。**着手時に最新の公式ドキュメントで裏取り**する（本書は「何をどの機構で担保するか」の意図を固定するもので、API名の正典ではない）。

## 0. 大前提（なぜモデル非依存を狙えているか）

この設計は「モデルが賢く推論しないと成立しない」部分を意図的に減らし、「**決まった構造をたどれば置き場所が分かる／読むべき品質バーがファイルに書いてある**」に寄せてある。
固定パス・ローカル採番（予測可能性＞賢さ）、薄い親＋子Skillのreferences、引き継ぎ書＋project-context、回帰eval。これらは構造と点検で担保＝モデルを替えても効く。
**「回す（実行）」はモデル寛容、「作る（写し取り・テーラリング設計）」はモデル依存**、という非対称が基本認識。

## 1. モデル堅牢性の3急所（網の粗い所・効く順）

1. **停止ポイントが自己申告（M-05・Hooks未実装）。** 粗さの伝播を防ぐ根っこが✋ゲートなのに、今は指示だけで構造強制でない。弱い／別モデルは承認ゲートを飛ばしうる＝最大の見落としリスク。
2. **機械evalは「型」しか見ない。** 核トークン＝節が在るかは見るが中身の質は見ない。構造は完成・中身が浅い/誤りの出力は機械層で緑になる。判断の重い所（期待分解・仮説と断定・正直な達成度・テーラリング）はモデル推論に直依存し、下げると静かに劣化。目視/LLM-judge層がまだ人手＝穴。
3. **作る系はモデル依存。** 新Skillの「写し取り」＝何を品質バーに書くかが高度判断。弱いモデルで起こすと型の質が落ちる。

## 2. 今（Cowork）でやる手当て ── 本セッションで着手

- **モデル使い分け運用ルール（下記§5に明記・即運用）**: 作る系は強モデル／回す系は別モデル可／モデル交代時は `bash _tools/eval.sh` 全件＋主要Skillの目視を必ず流す。
- **eval目視層をLLM-judge-readyに**: `_tools/eval/合格チェックリスト.md` をLLM判定者が回せる judge プロンプト化（`_tools/eval/judge-prompt.md`）。Coworkでは手動/単発LLMで回せ、Claude Codeではsubagent自動化へそのまま移植。急所2への一次対応。

## 3. Claude Code版で必ずやること（取りこぼさないチェックリスト）

各項目に「効く急所」を併記。着手時はここから拾う。

- [ ] **停止ポイントをHooksで構造強制**（急所1）。まず**不可逆・高stakesの✋✋だけ**先に（前段0-4承認・計画承認・終結の提出版docx確定）。承認マーカー（例: 直近成果物に承認記録）が無ければ次フェーズ着手や提出版書き出しをブロックする型のHookを置く。全停止の鍵化は後。
- [ ] **eval.shをHook/CIで自動実行**（急所2）。成果物の書き込み後、またはモデル交代時に機械層を自動で回し、FAILで止める。
- [ ] **LLM-judge（subagent）で目視層を自動化**（急所2）。`judge-prompt.md`をsubagent化し、ゴールデンを質の観点で判定→結果を記録。機械層（型）と二段で守る。
- [ ] **build/sync/verifyのHook化**（基盤防御）。コミット前に `build.sh --verify`、正典編集後に `sync.sh sync` を自動。手作業忘れ（Save skill手動・同期し忘れ）を構造で潰す。
- [ ] **M-08 ポータビリティの確定**。`.skill`形式・description トリガー・停止の前提のうち、Claude Codeで変わる所／そのまま動く所を実地で確認し、移植で壊れる箇所を洗う。
- [ ] **モデル使い分けを設定で表現**。作る系タスクは強モデル指定、回す系は寛容、をプロジェクト設定/サブエージェントのモデル指定で固定（急所3）。

## 4. 移植の対応関係（Cowork機構 → Claude Code）

| 守りたいこと | 今（Cowork） | Claude Code版 |
|---|---|---|
| 停止ポイントの遵守 | 指示（自己申告） | **Hooksで構造強制**（不可逆ゲート優先） |
| 出力の型の維持 | `eval.sh` を手動 | Hook/CIで自動実行・FAILで停止 |
| 出力の質の維持 | 合格チェックリストを人手 | **LLM-judge(subagent)で自動**＋記録 |
| 基盤の健全性 | `build.sh --verify`手動＋毎朝タスク | コミット前Hook＋定期 |
| Skill再登録 | Save skill手動（マウント権限制約） | パッケージ/配置の自動化余地 |
| 構造（固定パス・採番・SSOT） | 規約＋ツールで担保 | **そのまま移植可**（既にモデル非依存） |

## 5. モデル使い分け運用ルール（即運用）

- **作る系（強モデル必須）**: 新Skillの伴走→写し取り、references/型と品質基準の起草、テーラリング設計、計画提案書の設計意図。判断の質がそのまま資産の質になる工程。
- **回す系（別モデル可・寛容）**: 既存Skillで成果物を生成する、定型の巻き上げ。型に守られている工程。
- **モデルを交代したら**: 着手前に `bash _tools/eval.sh` 全件＋主要Skillの目視（または judge）を必ず流す。型の回帰を先に潰してから本番に使う。

## 6. 優先順位

1. 今: モデル使い分けルール（§5・即）＋ judge-prompt（§2）。
2. Claude Code版 着手時: §3の上2つ（停止のHooks強制・evalの自動実行）＝急所1・2に直接効く。
3. 続けて: LLM-judge自動化 → build/sync/verifyのHook化 → M-08確定。
