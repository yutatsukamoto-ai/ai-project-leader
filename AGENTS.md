# AGENTS.md

このプロジェクトは、生成AI/DX案件をPMBOK第8版ベースで進めるための
Claude Cowork / Claude Code / Codex 向けSkill群とナレッジ基盤。

> **Codex で作業する場合は [`CODEX.md`](CODEX.md) を最初に読むこと。**

## 最初に読む順

最初から全ファイルを読むのではなく、入口文書で現在地をつかんでから、作業内容に応じて深掘りする。Codexで作業する場合は、冒頭の注記どおり先に `CODEX.md` を読む。

1. `README.md`
2. `00_プロジェクト管理/構造レビュー/決定の索引.md`
3. 作業内容に応じて下表の追加文書を読む

| 作業内容 | 追加で読むもの |
|---|---|
| Codexでリポジトリ編集・検証・配布生成をする | `CODEX.md` の制約に従う |
| Skill構造・成果物対応を触る | `20_Skills/成果物マップ.md`、`20_Skills/横断GL対応表.md`、対象Skill、関連する横断ガイドライン |
| Skill追加・改名・横断GL新設など構造変更をする | `プロジェクト骨子.md` の設計原則、`00_プロジェクト管理/構造レビュー/規約・枠を新設するときの心得.md` |
| 案件成果物を作る・更新する | 該当案件の `project-context.md` **のみ**（現在地・次Skill・読むべきGL・文書ポインタが全て載っている）。案件特定が要る場合だけ `30_Flow/実案件/案件インデックス.md` を先に引く |
| 未決課題・構造上の懸念を調べる | `00_プロジェクト管理/構造レビュー/懸念マスター.md` |
| 宿題棚卸し・優先順位の確認 | `00_プロジェクト管理/宿題一覧_棚卸し報告書_2026-06-22.md`、`00_プロジェクト管理/構造レビュー/懸念マスター.md` |
| Skill改名・追加・横断GL新設の波及チェック | `00_プロジェクト管理/構造レビュー/規約・枠を新設するときの心得.md`（波及チェックリスト節）、`bash _tools/build.sh --verify` |

`プロジェクト骨子.md` は設計思想やロードマップを確認するときに読む。懸念マスターは重要だが、初動で必ず読む文書ではなく、未決・懸念・解決ログを追うための台帳として扱う。

## 現在の前提

- 現在の主戦場は Claude Cowork。まず中身・骨組み・配布用Skillを締める。
- Claude Code対応は段階導入済み。Hooks強制・auto-eval・auto-build・CI smoke・subagent定義は実装/実環境検証済み。
- 残る後続フェーズは、3案件目実証、W-02監視push化、M-08のCopilot等へのポータビリティ検証、M-09学習還流。
- Codex版は現時点では、リポジトリ編集・検証・配布生成・スライド生成補助の入口として使う。
- Codexで新規スライドを作る場合、特に指定がなければ必ず `image2-brand-slides` の流れで進める。`slide-craft` は編集可能PPTXが明示された場合や既存PPTXレビュー時の補助に限る。
- Codex配布版は `bash _tools/package-dist.sh --target codex` で作る。配布版には `AGENTS.md` と `CODEX.md` を含め、Claude Code専用の `CLAUDE.md` / `.claude/` は含めない。

## 作業時の約束

- 多段階タスクは勝手に進めない。停止ポイント・承認ゲートを尊重する。
- 不可逆な操作、客先提出物の確定、ベースライン変更、配布物の確定はユーザー確認を取る。
- 既存の未コミット変更はユーザー作業として扱い、戻さない。
- 実案件データは `30_Flow/` 内に置き、配布・Stock化するときは脱識別する。
- ユーザーが実案件を「○○の件」「○○案件」などフラットに呼んだら、まず `30_Flow/実案件/案件インデックス.md` で案件を特定し、該当案件の `project-context.md` を読んでから返答する。読み取りだけなら `30_Flow/現在の作業セッション.md` のアクティブ案件は変更しない。
- 案件成果物を作成・更新する前、または `project-context.md` へ書き戻す前は、`30_Flow/現在の作業セッション.md` を確認する。保存先がアクティブ案件ルート配下でない場合は書き込まず、ユーザー確認を取る。
- 案件を切り替えるときは「現在: {旧案件名}/{旧フェーズ} → 次: {新案件名}/{新フェーズ}、以後の書き込み先: {新案件ルート}」を宣言し、`30_Flow/現在の作業セッション.md` を更新してから作業する。
- 客先提出版に内部用の検査レシートや管理メタをそのまま混ぜない。
- Skill改名・追加・横断GL新設など構造に影響する変更を行ったら、`00_プロジェクト管理/構造レビュー/規約・枠を新設するときの心得.md` の波及チェックリストを確認する。
- Skill・配布・運用ルールを変えたら、完了前に `README.md` / `AGENTS.md` / `CODEX.md` / `プロジェクト骨子.md` など入口文書の同期要否を確認する。影響がなければ更新しない。
- Codexでスライド作成依頼を受けたら、明示指定がない限り `image2-brand-slides` を使う。PPTXレンダー流用やDOM手組みHTMLで代替せず、`message_design.md` → `deck_structure.md` → `design_system.md` → `prompts/` → Image2生成 → HTMLデッキ化の順で進める。

## 初回案件の進め方

1. ユーザーの案件説明・議事録・商談メモを受け取る。
2. `zendan-chain` を入口にする。
3. `00_前段/01_案件理解サマリー` で必ず理解確認を取る。
4. 確認後、`02_課題仮説シート`、必要なら `03_リサーチメモ`、`04_計画提案書` へ進む。
5. `04_計画提案書` 承認後に `00_前段/引き継ぎ書_前段→立ち上げ.md` を起案する。
6. 次は `tachiage-chain`。入力は引き継ぎ書と `project-context.md`。

## パス規約

- 新規案件は `30_Flow/{実案件|模擬案件|テスト}/{案件種別}_案件名/` に作る。
- 案件ルート直下に `project-context.md` を置く。前段だけで停止中の古い案件に無い場合は正常。
- 正典成果物はフェーズフォルダに置く: `00_前段/`, `01_立ち上げ/`, `02_計画/`, `03_実行/`, `04_監視コントロール/`, `05_終結/`。
- `outputs/` はサブエージェント出力、検証レポート、旧版ドラフトなどの中間生成物に限る。
- 次のSkillの入力になる成果物は `outputs/` ではなくフェーズフォルダを優先する。
- Markdown正典の成果物には冒頭メタを置く。提出版では内部管理色が強い項目を外すか文書管理欄へ整える。

## やらないこと

- 承認なしに次フェーズへ進まない。
- `outputs/` を正典成果物の置き場にしない。
- `.skill` を手作業でzipしない。
- 共通referencesのコピーを直接編集しない。
- 実案件データを配布物や `40_Stock/` にそのまま入れない。
- 客先提出版に内部用の検査レシートや管理メタをそのまま混ぜない。

## 正典と同期

- 共通ガイドラインの正典は `40_Stock/横断ガイドライン/`。
- 成果物とSkillの対応表は `20_Skills/成果物マップ.md`。
- 共通referencesを直すときは正典を編集し、`bash _tools/build.sh --sync` でコピーへ反映する。
- `.skill` の手作業zipは禁止。パッケージ化は `bash _tools/build.sh <skillディレクトリ>` または `--all` を使う。

## 検証コマンド

- 構造健全性: `bash _tools/build.sh --verify`
- 共通項ドリフトだけ確認: `bash _tools/build.sh --check`
- 成果物の回帰eval: `bash _tools/eval.sh`
- Hook smoke: `bash _tools/test-hooks.sh`
- Skill改訂時、モデル交代時、配布前は `build.sh --verify` と必要範囲の `eval.sh` を流す。
- Codex配布版: `bash _tools/package-dist.sh --target codex` → `bash _tools/test-dist-codex.sh`
- Claude Code配布版: `bash _tools/build.sh --sync-cc` → `bash _tools/package-dist.sh --target claude-code` → `bash _tools/test-dist-cc.sh`
